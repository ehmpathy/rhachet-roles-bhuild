# vision: --wish @stdin|words

## the outcome world

### day-in-the-life

vlad has an idea while reviewing code. he opens a terminal:

```sh
rhx init.behavior --name add-caching --wish "add redis caching to the user lookup to reduce db load during peak traffic"
```

the behavior directory is created, the wish file already contains his thought, and the route is bound. he's immediately in flow — no editor dance, no copy-paste, no context switch.

### before/after contrast

| before | after |
|--------|-------|
| `rhx init.behavior --name foo --open nvim` | `rhx init.behavior --name foo --wish "..."` |
| wait for editor to open | immediate |
| type wish into editor | wish already typed in command |
| save and close editor | no editor needed |
| 4 steps, ~15 seconds | 1 step, ~3 seconds |

### the "aha" moment

> "wait, i can just write my wish inline? i don't have to open an editor at all?"

the value clicks when the user realizes they can capture their thought at the speed of thought — no friction, no context switch, no waiting for an editor to load.

## user experience

### usecases

| usecase | contract |
|---------|----------|
| quick capture | `--wish "inline words"` |
| stdin pipe | `echo "my wish" \| rhx init.behavior --name foo --wish @stdin` |
| heredoc for multiline | `rhx init.behavior --name foo --wish @stdin <<'EOF'\n...\nEOF` |
| branch-derived name | `--name @branch --wish "..."` |

### contract inputs

```sh
# inline words (most common)
rhx init.behavior --name add-caching --wish "add redis caching for user lookups"

# from stdin (for piping, heredocs, multiline)
echo "my detailed wish" | rhx init.behavior --name add-caching --wish @stdin

# multiline via heredoc
rhx init.behavior --name add-caching --wish @stdin <<'EOF'
want to add redis caching

- cache user lookups
- 5 minute ttl
- invalidate on user update
EOF

# combined with extant options
rhx init.behavior --name @branch --open nvim --wish "quick note to expand on"
```

### contract outputs

- wish file (`0.wish.md`) contains `wish = \n\n<content from --wish>`
- behavior directory created as before
- route bound as before
- footer shows wish file path (if `--open` used, editor still opens for expansion)

### timeline

1. user types command with `--wish "..."` (0s)
2. behavior dir created, wish populated (0.5s)
3. output shows tree + wish path (0.5s)
4. user is done — or continues to `--open` for expansion (optional)

## evaluation

### how well does it solve the goal?

the wish asks for:
- `--wish @stdin|words` — **fully solved** by supporting both inline strings and @stdin
- populate wish file before editor opens — **fully solved**
- write words before hitting execute — **fully solved** by inline `--wish "..."`

### pros

- zero friction capture — thought to file in one command
- composable — works with pipes, heredocs, scripts
- backwards compatible — `--open` still works for expansion
- familiar pattern — mirrors `git commit -m`

### cons

- shell escaping — quotes in wish content require escape or @stdin
- discoverability — users need to know `--wish` exists
- multiline inline — awkward without heredoc

### pit-of-success edgecases

| edgecase | handling |
|----------|----------|
| `--wish ""` (empty) | error: wish content required |
| `--wish` (no arg) | error: missing value |
| `--wish "..."` + `--open nvim` | wish populated, then editor opens (user can expand) |
| quotes in wish | use @stdin or escape |
| very long wish | use @stdin + heredoc |

## open questions & assumptions

### assumptions

- users will prefer inline `--wish "..."` over piping for short wishes
- @stdin pattern is already familiar from other rhx skills (sedreplace, git.commit.set)
- wish file format remains `wish = \n\n<content>`

### resolved questions

1. should `--wish` and `--open` be mutually exclusive?
   - **[answered]**: both work together — wish is populated, then editor opens
   - **evidence**: wisher's example shows both flags combined

2. what happens if wish file already has content?
   - **[answered]**: error if wish file has non-template content
   - **rationale**: pit-of-success — don't silently overwrite user's prior work

3. should we support `--wish @file`?
   - **[answered]**: no — @stdin covers the usecase via `cat file | rhx ... --wish @stdin`
   - **rationale**: kiss — avoid complexity for marginal benefit

4. what about empty `--wish ""`?
   - **[answered]**: error — require non-empty content
   - **rationale**: if user wants empty structure, omit `--wish` flag entirely

### external research needed

- none — this is internal tool with clear patterns from extant skills

## what is awkward?

### shell escaping

```sh
# awkward: quotes need escaping
rhx init.behavior --name foo --wish "add \"quoted text\" handling"

# better: use @stdin
echo 'add "quoted text" handling' | rhx init.behavior --name foo --wish @stdin
```

### multiline inline

```sh
# awkward: can't do this
rhx init.behavior --name foo --wish "line 1
line 2"

# better: heredoc
rhx init.behavior --name foo --wish @stdin <<'EOF'
line 1
line 2
EOF
```

### tradeoff: simplicity vs power

inline `--wish "..."` is simple but limited. @stdin is powerful but requires pipe/heredoc. this tradeoff feels right — simple for simple cases, powerful for complex cases.

the design follows the extant rhx pattern (git.commit.set -m @stdin) so the "awkwardness" is already normalized for users of this ecosystem.
