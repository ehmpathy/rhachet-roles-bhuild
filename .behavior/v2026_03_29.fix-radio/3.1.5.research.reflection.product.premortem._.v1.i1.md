# research: premortem reflection

how could this fail?

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **keyrack api mismatch**: keyrack unlock api changed since we researched it. skill calls fail with "unknown option --key" or similar. developers hit cryptic errors.

2. **github app permissions insufficient**: beaver app provisioned with issues:write but skill needs more (e.g., pull_requests:write for cross-repo scenarios). skill fails at runtime with 403 errors.

3. **test env creds not filled**: tests pass locally (developer has creds) but fail in ci/cd (different machine, no keyrack fill). "tests pass on my machine" syndrome.

---

## what obvious signs did we ignore?

look back from the imagined failure — what signs were present that we dismissed?

- **keyrack is new**: keyrack was just released. api may not be stable. we assumed stable api without version pin.

- **no keyrack integration tests in bhuild**: we added keyrack to production code without first validated the api works in this repo's test harness.

- **wish says "public app"**: we assumed public app is needed but never validated why. if only ehmpathy repos need the app, private app would suffice.

- **role keyrack pattern unclear**: we assumed Role.build() supports keyrack property but didn't verify rhachet's Role type definition.

| beaver app only needs issues:write | must re-provision app with more permissions | medium |

## what is the nightmare scenario?

### worst plausible outcome

**beaver app private key leaked**: someone commits the app credentials to git, or keyrack vault is compromised. attacker gains access to all repos where beaver app is installed.

impact:
- attacker can create/close/modify issues on all installed repos
- attacker can impersonate the beaver robot
- hard to detect since app tokens look like legitimate automation

**recovery**:
- revoke app credentials immediately
- rotate app private key
- audit all issues created since compromise
- notify all affected repo owners

### second worst

**keyrack dependency causes circular import**: keyrack depends on rhachet-roles-bhuild, bhuild depends on keyrack. tests fail with "circular dependency" errors. must untangle before ship.

## what mitigations to consider?

| risk | mitigation | cost |
|------|------------|------|
| keyrack api mismatch | add integration test that validates keyrack unlock api | 1 hour |
| beaver app permissions | document all permissions in app readme, test each endpoint | 2 hours |
| test env creds not filled | add clear error message in jest env if keyrack.source fails | 30 min |
| private key leak | store in 1password, never commit, use keyrack passphrase | 0 (already planned) |
| Role.build keyrack support | verify rhachet type definitions before implementation | 30 min |
| circular dependency | check rhachet/keyrack dependencies before import | 15 min |
