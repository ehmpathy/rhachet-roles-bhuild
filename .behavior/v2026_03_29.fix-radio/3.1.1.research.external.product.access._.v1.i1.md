## 1. github rest api (issues endpoint)

### contract

the create issue endpoint is `POST https://api.github.com/repos/OWNER/REPO/issues` [1].

required headers:
- `Accept: application/vnd.github+json`
- `Authorization: Bearer <token>`

required parameters:
- `title` (string): issue title

optional parameters:
- `body` (string): issue description
- `assignees` (array): usernames to assign
- `labels` (array): label names
- `milestone` (number): milestone id

### authentication

"to authenticate your request, you will need to provide an authentication token with the required scopes or permissions. if you try to use a REST API endpoint without a token or with a token that has insufficient permissions, you will receive a 404 Not Found or 403 Forbidden response." [2]

### permissions required

for github apps, the `issues:write` permission is required. "the success of an API request with an installation access token only depends on the app's permissions" — not the user's own permissions [3].

## 2. github apps authentication

### token lifecycle

"the installation access token will expire after 1 hour" [4]. this fixed expiration is a security feature.

"instead of generating an installation access token, you can use GitHub's Octokit SDKs to authenticate as an app. the SDK will take care of generating an installation access token for you and will regenerate the token once it expires" [5].

### jwt requirements

"GitHub enforces that the JWT token used can only be valid for a maximum of 10 minutes - a new token will be required after this time" [6]. the jwt is used to request installation tokens, not for api calls directly.

### credentials format

installation credentials consist of three fields [7]:
- `appId`: the app ID from app settings
- `privateKey`: content of the *.pem file
- `installationId`: obtained from webhook payloads or REST API

"if your private key contains escaped newlines (`\\n`), they will be automatically replaced with actual newlines" [7].

## 3. github apps vs personal access tokens

### security comparison

"GitHub Apps use short-lived tokens, and if the token is leaked, the token will be valid for a shorter amount of time, which reduces the damage that can be done. by default, the tokens generated by GitHub Apps have an expiration time of 8 hours [for user tokens] / 1 hour [for installation tokens]. in contrast, PATs typically have an expiration time of seven, 30, 60, or 90 days, or no expiration at all" [8].

"GitHub Apps use fine-grained permissions, give the user more control over which repositories the app can access. if your token gets leaked with a PAT, someone could have full access to your resources" [8].

### lifecycle management

"PATs are associated with a specific-user, so if these tokens are associated with a workflow, the original user who created the token must manage the lifecycle of these tokens. in contrast, the GitHub App is managed under your organization settings, so its lifecycle management is not limited to a single GitHub user" [8].

### convergence signal

sources [4], [5], [8], [9] all agree: github apps with installation tokens are more secure than PATs for automation. 4+ sources converge on this recommendation.

## 4. credential management best practices

### ephemeral credentials principle

"short-lived (ephemeral) credentials are dynamic secrets that automatically expire after a brief, predefined period — typically minutes or hours. unlike static secrets that are manually rotated, dynamic secrets are generated on-demand and revoked automatically upon expiration" [10].

"static, long-lived credentials or 'secrets' (e.g. passwords, API keys, SSH keys) remain one of the most significant security vulnerabilities in modern infrastructures" [10].

### automation requirement

"never rotate secrets manually. use the built-in managed rotation features" [11].

"systems should automatically request credentials with a defined time for expiration rather than relying on static keys" [11].

### vault pattern

"adopting a centralized secrets manager is the first step to removing the biggest risk in your IT environment. when you're forced to manage static credentials, use a secrets management platform to contain and control them" [10].

## 5. github cli integration

### environment variable precedence

"the GH_TOKEN and GITHUB_TOKEN environment variables can be used as authentication tokens for the GitHub CLI, and setting these avoids being prompted to authenticate and takes precedence over previously stored credentials" [12].

"GH_TOKEN takes precedence over GITHUB_TOKEN when both are set" [12].

### ci/cd compatibility

"you can set the GH_TOKEN environment variable to authenticate the gh CLI without running `gh auth login`, and you can set this environment variable at the workflow level, job level, or step level" [13].

## 6. github actions GITHUB_TOKEN

### automatic provisioning

"the GITHUB_TOKEN secret is an automatically generated API token provided by GitHub for authentication within GitHub Actions workflows. before each job begins, GitHub fetches an installation access token for the job" [14].

"the token's permissions are limited to the repository that contains your workflow. the token expires when the job is finished" [14].

### permission scoping

"as a good security practice, you should always make sure that actions only have the minimum access they require by limiting the permissions granted to the GITHUB_TOKEN" [14].

"available permissions include: actions, attestations, checks, contents, deployments, id-token, issues, models, discussions, packages, pages, pull-requests, security-events, and statuses" [14].

## 7. oauth 2.0 security principles

### short-lived token rationale

"one of the big benefits of short-lived tokens is the reduction of the attack window (i.e., the length of time that an attacker could exploit endpoints). in a system using long-lived tokens, an attacker who successfully intercepts an access token could reuse it repeatedly until it expires" [15].

### nist guidance

"NIST SP 800-63C emphasizes the importance of using short-lived, narrowly-scoped tokens for federated identity systems to minimize risks" [15].

### rfc 9700 recommendations

"keep access tokens short-lived and pair them with rotating refresh tokens rather than issuing long-lived access tokens; this is a core recommendation in the OAuth 2.0 security BCP and modern guidance" [16].

## 8. ci/cd vs local development

### separation principle

"start by maintaining a strict separation between development, prep, and production secrets. a developer testing locally with dev credentials shouldn't be able to impact production systems" [17].

### limitations of platform-native secrets

"platform-native secrets are isolated to the CI/CD environment, creating duplication and potential sync issues between your CI/CD secrets and application secrets" [17].

### secretless authentication trend

"forward-looking teams are adopting secretless authentication: replacing stored secrets with ephemeral credentials tied to real-time workload identity. rather than storing secrets in GitHub, the workflow authenticates itself when needed, and gets a tightly-scoped credential only for the current job" [17].

---

## 9. private key security

### storage recommendations

"you must store private keys, tokens, and client secrets securely. the private key for your GitHub App grants access to every account that the app is installed on and must be stored securely and never shared broadly. consider storing your GitHub App's private key in a key vault like Azure Key Vault and making it sign-only" [18].

### permission minimization

"when you register a GitHub App, select the minimum permissions that your GitHub App needs. if any keys or tokens for your app become compromised, this will limit the amount of damage that can occur" [18].

## 10. keyrack integration pattern

### unlock flow

based on ehmpathy/keyrack patterns, the unlock command is:
```bash
rhx keyrack unlock --env "$ENV" --passphrase "$PASSPHRASE"
```

this returns json with unlocked vault contents [19].

### ci/cd integration

keyrack integrates with github actions via `actions/create-github-app-token@v2` [19]:
```yaml
- name: get github auth
  uses: actions/create-github-app-token@v2
  with:
    owner: ${{ inputs.creds-github-app-owner }}
    app-id: ${{ inputs.creds-github-app-id }}
    private-key: ${{ secrets.creds-github-app-private-key }}
```

## anti-patterns

### long-lived tokens

sources [8], [10], [15] warn against long-lived tokens:
- increased attack window if compromised
- harder to rotate across integrations
- temptation to share rather than scope

### manual credential management

sources [10], [11] warn against manual rotation:
- prone to human error
- creates inconsistent state
- delays response to compromise

### overly broad permissions

sources [14], [18] warn against excessive permissions:
- violates least privilege principle
- increases blast radius of compromise
- makes audit trail harder to interpret

---

## conflict detection

no significant conflicts detected across sources. all sources converge on:
- short-lived tokens preferred over long-lived
- github apps preferred over PATs for automation
- centralized credential management preferred
- minimum permissions principle
