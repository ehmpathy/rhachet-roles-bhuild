# research: audience reflection

who is this for and why do they care?

---

## who is the audience?

### primary: developers who use radio skill

developers who use `rhx radio.task.push` to broadcast tasks to github issues.

characteristics:
- work on ehmpathy projects or projects that use rhachet-roles-bhuild
- need to dispatch tasks to distributed queues
- currently face auth friction (must set up GITHUB_TOKEN or pass --auth)
- may not understand github app vs PAT tradeoffs

### secondary: security-conscious teams

teams that care about credential hygiene.

characteristics:
- prefer short-lived tokens over long-lived PATs
- want automated credential rotation
- need audit trails for credential access
- follow least-privilege principle

### tertiary: ci/cd pipelines

automated systems that run radio skills.

characteristics:
- use GITHUB_TOKEN from github actions
- cannot run keyrack daemon
- need env var fallback to work seamlessly
- expect zero manual auth configuration

---

## why do they care?

### primary pain points

| pain | impact |
|------|--------|
| manual auth setup | wastes 5-15 minutes per repo |
| --auth flag complexity | cognitive load on every invocation |
| long-lived PAT risks | security exposure if leaked |
| inconsistent auth across repos | confusion about which method to use |

### what goal does this enable?

> "i want to run `rhx radio.task.push` and have it just work, without config."

the goal is **zero-friction task dispatch** with **secure-by-default credentials**.

### what frustration does this eliminate?

- "why do i need to set up auth for every repo?"
- "which env var do i use again?"
- "is my PAT still valid or did it expire?"
- "how do i know my token has the right permissions?"

---

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| primary (developers) | high | medium | productivity loss on every invocation |
| secondary (security teams) | medium | low | compliance risk from long-lived tokens |
| tertiary (ci/cd) | low | low | already works via env var fallback |

### priority rationale

- **primary high**: auth friction is encountered on every skill invocation
- **secondary medium**: security is important but does not block daily work
- **tertiary low**: ci/cd already works, keyrack is an improvement not a requirement

## what do they care about?

### success criteria

| audience | success looks like |
|----------|-------------------|
| developers | skill works first try, no manual setup |
| security teams | tokens auto-rotate, minimal blast radius |
| ci/cd | no changes needed, env var fallback works |

### tradeoffs they would accept

| tradeoff | accept | reject |
|----------|--------|--------|
| 500ms latency for keyrack | yes | no |
| one-time keyrack setup | yes | no |
| per-repo init command | yes | no |
| backwards-incompat change to --auth flag | no | yes |
| require keyrack daemon in ci/cd | no | yes |

### delight vs disappointment

**delighted**: "wait, it just worked? no token setup?"

**disappointed**: "i have to run two setup commands before the skill works?"

---

## how will they experience it?

### developer journey

1. **discovery**: developer runs `rhx radio.task.push`, gets auth error
2. **setup prompt**: error message guides to `rhx keyrack fill --owner ehmpath --env prep`
3. **machine setup**: run fill command once per machine (stores creds)
4. **repo setup**: run `npx rhachet init --keys --roles dispatcher` once per repo
5. **success**: all subsequent invocations "just work"
6. **continuous**: token refresh is transparent, no maintenance needed

### friction points

| point | mitigation |
|-------|------------|
| error message unclear | clear actionable error: "run keyrack fill..." |
| two setup commands | documented in error message order |
| keyrack daemon not in ci/cd | env var fallback works automatically |
| expired token | keyrack auto-refresh with 55-min buffer |

### smooth experience

```bash
# first time setup (once per machine)
rhx keyrack fill --owner ehmpath --env prep

# per-repo setup (once)
npx rhachet init --keys --roles dispatcher

# every subsequent use (zero friction)
rhx radio.task.push --via gh.issues --title "..." --description "..."
```
