# research: external product claims

facts, assumptions, questions, and opinions relevant to radio skill keyrack integration.

---

## claim categories

- [FACT] = indisputable, immutable truth
- [SUMP] = assumption (explicit or implicit)
- [KHUE] = open question to consider
- [OPIN] = subjective opinion to consider

---

## 1. token security claims

### [FACT] installation tokens expire after 1 hour

"installation access tokens expire after one hour, and using an expired token produces a status code of 401 - Unauthorized" [1].

this is a fixed constraint of the github platform — cannot be extended.

### [FACT] pats can be long-lived or non-expiring

"PATs typically have an expiration time of seven, 30, 60, or 90 days, or no expiration at all" [2].

### [OPIN] short-lived tokens are more secure than long-lived

"by eliminating always-on credentials, ephemeral accounts reduce the blast radius of credential theft and make lateral movement significantly harder for attackers" [3].

sources [2], [3], [4], [5] converge on this opinion. strong confidence.

### [FACT] credential abuse is a top attack vector

"credential abuse (use of stolen or compromised credentials) is cited as the initial attack vector in 22% of breaches, making it one of the top two entry points for attackers" [6].

### [FACT] secrets sprawl is measurable

"in 2024 alone, nearly 24 million new hardcoded secrets were detected in public GitHub commits" [6].

## 2. github apps claims

### [FACT] github apps have higher rate limits

"GitHub Apps have a higher API rate limiting threshold than user accounts - installed in an Enterprise, GitHub Apps can have up to 15,000 requests per hour whereas user-created personal access tokens have a limit of 5,000 requests per hour" [7].

### [FACT] public apps can be installed by any github user

"if you set your GitHub App registration to public, any user on GitHub can install it and authorize it" [8].

### [FACT] private apps limited to owner account

"if you set your GitHub App registration to private, it can only be installed on the account that owns the app" [8].

### [SUMP] public app is needed for cross-repo usage

the wish specifies "public app" — this implies the app needs to be installable on multiple orgs/accounts.

### [KHUE] what repos should the beaver app access?

the wish doesn't specify which repos need the beaver app installed. this affects installation scope.

## 3. developer experience claims

### [FACT] 15-minute time-to-value rule

"developers expect quick results. if your product doesn't provide value within 15 minutes, they'll likely move on" [9].

### [OPIN] friction kills developer adoption

"if a product demands extensive configuration before its core functionality, developers are likely to move on to alternatives. their low tolerance for friction means that speed and simplicity are non-negotiable" [9].

### [FACT] structured onboard improves retention

"companies with structured onboard compress productivity timelines to 2-3 weeks while retention improves by 40%" [10].

### [SUMP] keyrack fills once per machine is acceptable

the vision assumes machine-wide fill is low friction. this may not hold if developers work on multiple machines or use ephemeral environments.

---

## 4. credential management claims

### [OPIN] centralized secrets management is preferred

"a centralized secrets manager is the first step to remove the biggest risk in your IT environment" [4].

sources [4], [5], [11] converge on this opinion. strong confidence.

### [FACT] secrets sprawl causes security and velocity problems

"when secrets sprawl takes hold, teams face problems: security risks from hardcoded secrets and untracked credentials, compliance gaps caused by lack of auditability and inconsistent access controls, deployment failures due to outdated or absent secrets across environments, onboard delays as new developers struggle to find what they need" [6].

### [OPIN] secretless authentication is the future

"forward-look teams adopt secretless authentication: replace stored secrets with ephemeral credentials tied to real-time workload identity" [12].

### [SUMP] env var fallback for ci/cd is acceptable

the vision assumes fall back to GITHUB_TOKEN in ci/cd is acceptable. this maintains two credential paths.

---

## 5. enterprise compliance claims

### [OPIN] github apps preferred for enterprise compliance

"enterprise compliance guidance favors GitHub App installation tokens for 24x7 org-wide automation" [7].

### [FACT] some enterprises block pat creation

"in enterprises PAT creation is often blocked" [7].

### [OPIN] fine-grained permissions improve auditability

"fine-grained PATs allow GitHub organizations to limit, review and audit this type of token" [7].

## 6. token renewal claims

### [FACT] octokit sdks handle token renewal automatically

"GitHub's Octokit SDKs will take care of token generation and will regenerate the token once it expires" [13].

### [OPIN] cache tokens with early renewal buffer

"if a token doesn't exist or expires within 5 minutes, it should be refreshed" — early renewal provides buffer to avoid expiration in active use [13].

### [SUMP] keyrack handles renewal transparently

the vision assumes keyrack auto-refreshes with a 55-min buffer. need to verify this mechanism exists.

---

## 7. architecture claims

### [OPIN] centralized management with distributed access preferred

"centralized infrastructure for monitor, rate-limit, security scan, and API management can reduce duplication while teams retain ownership" [14].

### [OPIN] single source of truth for credentials

"all API credentials are kept in one secure location — a single source of truth that simplifies management and minimizes the risk of keys scattered across multiple systems" [14].

### [SUMP] keyrack.yml is single source of truth

the vision assumes keyrack.yml declares all credential requirements. this works for static requirements.

---

## 8. zero trust claims

### [OPIN] ephemeral credentials align with zero trust

"ephemeral credentials' dynamic nature aligns well with the principles of Zero Trust security models" [3].

### [OPIN] minimal implicit trust preferred

"ephemeral accounts align naturally with the zero trust model: minimize implicit trust and persistent access, grant access only when needed, validate continuously, remove it immediately after use" [3].

sources [3], [4], [5] converge on zero trust alignment. strong confidence.

## conflict detection

no significant conflicts detected. all sources converge on:
- short-lived tokens > long-lived tokens
- centralized secrets management > scattered credentials
- github apps > PATs for automation
- ephemeral > static credentials

---

## anti-patterns identified

### hardcoded credentials

sources [4], [6] warn against hardcoded secrets:
- "nearly 24 million new hardcoded secrets were detected in public GitHub commits" [6]
- creates sprawl, audit gaps, breach risk

### manual credential management

sources [4], [5] warn against manual rotation:
- prone to human error
- creates inconsistent state
- delays response to compromise

### long-lived tokens for automation

sources [2], [7] warn against PATs for automation:
- increased blast radius if compromised
- no automatic expiration
- tied to user lifecycle (employee leaves = problems)
