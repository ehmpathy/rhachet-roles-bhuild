# vision: fix-radio keyrack integration

paint a picture of what the world looks like when this wish is fulfilled.

---

## the outcome world

### before

```bash
# developer wants to use radio.task.push with gh.issues channel
rhx radio.task.push --via gh.issues --title "do this" --description "..."

# error: no auth detected for gh.issues channel
# tips:
# ├─ --auth as-human
# │     use gh cli logged-in session (interactive)
# ├─ --auth as-robot:env(VAR)
# │     use token from environment variable
# └─ --auth as-robot:shx(command)
#       execute shell command to retrieve token

# developer scrambles to set up GITHUB_TOKEN
export GITHUB_TOKEN=$(op item get github-pat --fields token)  # long-lived PAT

# or they manually wire up --auth every time
rhx radio.task.push --via gh.issues --auth as-robot:shx(op item get ...) --title "..." --description "..."
```

the developer faces friction:
- must understand auth modes
- must configure env vars or pass --auth
- often falls back to long-lived PATs (security concern)
- each repo reinvents auth setup

### after

```bash
# developer runs radio skill — it just works
rhx radio.task.push --via gh.issues --title "do this" --description "..."

# 🎙️ created: do this
#    ├─ exid: 42
#    ├─ status: QUEUED
#    ├─ repo: ehmpathy/rhachet-roles-bhuild
#    └─ via: gh.issues
```

under the hood:
1. skill calls `keyrack unlock --owner ehmpath --env prep --key EHMPATH_BEAVER_GITHUB_TOKEN`
2. keyrack retrieves just that key's creds (github app json)
3. mechanism `EPHEMERAL_VIA_GITHUB_APP` translates json → short-lived `ghs_*` token
4. skill uses the ephemeral token for gh api calls

### the "aha" moment

> "wait, it just worked? no token setup? and it's using short-lived tokens automatically?"

the value clicks when developers realize:
- zero manual credential configuration
- ephemeral tokens by default (1-hour lifetime)
- keyrack handles the translation transparently
- same path for tests, local dev, and ci/cd

## user experience

### prerequisite: github app provisioned

before any of this works, the ehmpath-beaver github app must be provisioned:

```
@repo/provision/ehmpath-beaver/
├── readme.md           # app permissions, setup instructions
├── manifest.json       # github app manifest
└── terraform/          # infrastructure as code
```

this is a one-time setup that produces:
- appId
- privateKey
- installationId

these credentials are stored in the ehmpath keyrack vault.

### setup

```bash
# once per machine: fill your vault with ehmpath credentials
rhx keyrack fill --owner ehmpath --env prep

# once per repo: grant this repo access to the keys
npx rhachet init --keys --roles dispatcher
```

that's it. radio skills will now "just work" with ephemeral github tokens.

the credentials live machine-wide in keyrack's vault. each repo that runs `init --keys` gains access to those credentials — no need to fill again.

### usecases

| usecase | goal | contract |
|---------|------|----------|
| provision github app | one-time infra setup | `@repo/provision/ehmpath-beaver` with terraform |
| setup auth | one-time keyrack config | `npx rhachet init --keys --roles dispatcher && rhx keyrack fill --owner ehmpath --env prep` |
| push task to github issues | broadcast work to distributed queue | `rhx radio.task.push --via gh.issues --title "..." --description "..."` |
| pull task from github issues | claim work from distributed queue | `rhx radio.task.pull --via gh.issues --exid 42 --status CLAIMED` |
| run integration tests | verify radio behavior | `npm run test:integration` (keyrack unlocks test creds) |
| ci/cd workflows | automated test runs | `npm run prepare:rhachet` before test steps |

### contract inputs & outputs

**radio.task.push**

```bash
# input: task details
rhx radio.task.push \
  --via gh.issues \
  --title "fix the flaky test" \
  --description "the test in src/foo.test.ts fails intermittently"

# output: task created
# 🎙️ created: fix the flaky test
#    ├─ exid: 42
#    ├─ status: QUEUED
#    └─ via: gh.issues
```

**keyrack unlock (internal)**

```bash
# the skill internally calls keyrack for just the key it needs
rhx keyrack unlock --owner ehmpath --env prep --key EHMPATH_BEAVER_GITHUB_TOKEN

# keyrack returns: EHMPATH_BEAVER_GITHUB_TOKEN=ghs_xxx (translated from app json)
```

### timelines

1. **developer runs skill** (t=0)
2. **skill calls keyrack unlock for specific key** (t=50ms)
3. **keyrack reads encrypted app json from vault** (t=100ms)
4. **keyrack translates json → ephemeral token** (t=500ms)
5. **skill uses token for gh api call** (t=600ms)
6. **gh api responds** (t=1200ms)
7. **skill outputs result** (t=1250ms)

total latency: ~1.2s (mostly gh api round trip)

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| radio skill "just works" | 10/10 | no manual auth config needed |
| short-lived tokens by default | 10/10 | `EPHEMERAL_VIA_GITHUB_APP` mechanism |
| same path for tests/dev/ci | 9/10 | ci falls back to env var (acceptable) |
| secure credential storage | 10/10 | keyrack + encrypted vault |

### pros

- zero-friction developer experience
- security by default (ephemeral tokens)
- single source of truth for credentials (keyrack.yml)
- tests use production-like credential path
- ci/cd compatible (env var fallback)

### cons

- initial setup complexity (provision github app, store creds)
- ~500ms latency for token translation (acceptable)
- requires keyrack daemon running locally

### pit of success: edgecases

| edgecase | how we handle it |
|----------|------------------|
| keyrack not unlocked | clear error: "keyrack not unlocked for env=prep" |
| expired token | keyrack auto-refreshes (55-min buffer before expiry) |
| absent app creds | clear error: "EHMPATH_BEAVER_GITHUB_TOKEN not found in vault" |
| ci/cd environment | `npm run prepare:rhachet` sources keyrack before tests |
| tests without creds | test env keyrack.yml specifies required creds |

## what is awkward?

### minor friction

1. **initial provisioning complexity**: developer must provision github app, store creds, configure keyrack.yml — but this is one-time setup

2. **two credential paths**: keyrack (local/test) vs env var (ci/cd) — but keyrack handles this transparently

3. **keyrack daemon dependency**: requires daemon running locally — but mechanic role already handles this

### acceptable tradeoffs

| tradeoff | why acceptable |
|----------|----------------|
| provisioning effort | one-time setup, pays off in perpetuity |
| keyrack latency | 500ms is negligible for cli usage |
| daemon dependency | mechanic role already manages daemon |

### no awkwardness found

the design aligns with:
- extant keyrack patterns (xai, bhrain)
- extant declastruct-github patterns
- extant radio skill architecture

the credential flow is natural: app creds → keyrack → ephemeral token → skill → gh api.
