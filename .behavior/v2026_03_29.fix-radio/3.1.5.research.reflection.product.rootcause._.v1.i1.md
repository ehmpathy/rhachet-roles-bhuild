# research: rootcause reflection

why is this needed?

---

## what is the symptom?

### observable problem

the radio skill requires manual auth configuration before it works.

### how it manifests

```bash
rhx radio.task.push --via gh.issues --title "..."
# error: no auth detected for gh.issues channel
```

developer must either:
- set GITHUB_TOKEN env var manually
- pass --auth flag on every invocation
- use --auth as-human (interactive, not automation-friendly)

### when it occurs

every time a developer:
- sets up a new repo with radio skill
- runs radio skill on a new machine
- switches between repos with different auth configs

---

## 5 whys tree

### branch 1: radio skill lacks default auth

1. **why does radio skill require manual auth?**
   → skill has no default auth mechanism. it expects explicit --auth or GITHUB_TOKEN.

2. **why does skill have no default auth?**
   → skill was designed before keyrack existed. only fallback was env var.

3. **why was env var the only fallback?**
   → no centralized credential management system existed in rhachet.

4. **why was there no centralized credential management?**
   → keyrack was not yet released. credential management was out of scope.

5. **why was keyrack delayed?**
   → **[root cause]** keyrack is complex infrastructure. it required separate design and release.

### branch 2: long-lived tokens are a security concern

1. **why do developers use long-lived PATs?**
   → PATs are easy to create and don't expire quickly.

2. **why are PATs easier than github apps?**
   → github apps require provision, install, and credential management.

3. **why is github app credential management hard?**
   → app credentials are json blobs with multiple fields (appId, privateKey, installationId).

4. **why are json creds harder than single tokens?**
   → no standard toolchain to translate json → ephemeral token.

5. **why was there no toolchain?**
   → **[root cause]** keyrack's EPHEMERAL_VIA_GITHUB_APP mechanism fills this gap.

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| symptom | radio skill requires manual auth | no |
| immediate | skill has no default auth mechanism | no |
| intermediate | no centralized credential management existed | no |
| deep | keyrack not yet integrated into bhuild | **yes** |
| foundational | keyrack mechanism for ephemeral tokens was absent | **yes** (addressed by keyrack release) |

---

## recommendation

### address root cause

integrate keyrack into radio skill as the default auth mechanism:

1. skill tries keyrack unlock first
2. skill falls back to GITHUB_TOKEN env var
3. skill fails with clear error if neither available

this addresses the root cause by:
- provide default auth that "just works"
- leverage keyrack's ephemeral token mechanism
- eliminate manual PAT management for developers

### why not address symptom?

to address the symptom (e.g., add more --auth modes, better error messages) would:
- increase complexity without solve the core problem
- perpetuate long-lived token usage
- require manual setup for each repo

the root cause fix is strictly better: simpler code, better security, better DX.
