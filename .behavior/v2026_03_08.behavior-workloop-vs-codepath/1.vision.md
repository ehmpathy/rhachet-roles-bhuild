
## the outcome world

### what does a day-in-the-life look like with this in place?

**before:**
a mechanic receives a behavior to implement. they dive straight into the solution (the product), but midway realize they don't have test patterns, api keys aren't configured, the test framework doesn't support their usecase, and they spend hours on infra debug instead of value delivery. the blueprint mixes "how do i test this" with "what am i to build" into one blob.

**after:**
the behavior route separates concerns cleanly:

1. **3.3.0.blueprint.factory** declares _what tools the mechanic needs_ (the machine)
   - test frameworks to use (jest, vitest, playwright, etc)
   - dev loops (build, run, debug commands)
   - access credentials and keys required
   - oss libraries to leverage

2. **3.3.1.blueprint.product** declares _what code the mechanic will write_ (the deliverable)
   - domain objects and operations (code.prod)
   - test coverage plan (code.test)
   - file diffs and code structure
   - contract shapes and behaviors

the mechanic now sees two distinct artifacts: "here's what tools you'll use" vs "here's what code you'll write".

### what's the before/after contrast?

| aspect | before | after |
|--------|--------|-------|
| blueprint | single blob that mixes tools with code | two blueprints: factory (tools) + product (code to write) |
| playloops | vague concept that tried to do too much | renamed to research.frames.testloops, focused on test frameworks |
| research | flat list of 3.1.research.* | organized into 3.1.1 (external knowledge), 3.1.2 (internal patterns), 3.1.3 (code patterns) |
| repro scenarios | implicit or absent | explicit distill.repros.experience artifact |
| mechanic experience | "wait, do i have the tools to test this?" | "factory blueprint tells me exactly what i need" |

### what's the "aha" moment where the value clicks?

> "oh, factory is the TOOLS - test frameworks, deployment infra, creds. product is the DELIVERABLE - what code we actually write, the spec (tests) and the impl (prod)."

the mechanic realizes they can focus purely on code design (prod + test) in the product blueprint, because factory already declared what tools they'll use to build and verify.

## user experience

### what usecases do folks fulfill? what goals?

1. **mechanic starts a behavior**: reads factory blueprint first to understand what tools they'll use, then reads product blueprint to understand what code to write

2. **reviewer validates a blueprint**: checks factory for adequate test infra and dev feedback loops, separately checks product for code design quality and test coverage

3. **wisher observing progress**: sees explicit repros.experience distillation showing how their user journey will be replicated in tests

### what contract inputs & outputs do they leverage?

**inputs to factory blueprint:**
- 3.1.1.research.knowledge.* (external: access, claims, domain)
- 3.1.2.research.patterns.* (templates, test frames, oss resources, deployment patterns)

**outputs from factory blueprint:**
- test frameworks to use
- deployment infra to use
- dev loop configurations (build, test, deploy commands)
- access/creds requirements
- oss libraries to leverage

**inputs to product blueprint:**
- factory blueprint (declares the tools)
- 3.1.3.research.code.* (code.prod, code.test patterns)
- 3.2.distill.* (domain, repros.experience)
- criteria

**outputs from product blueprint:**
- filediff tree
- product tree (code.prod + code.test)
- domain decomposition
- test coverage plan

## mental model

### how would users describe this to a friend?

> "factory is the workshop setup before you build furniture. product is the furniture itself. you wouldn't mix 'i need a table saw' with 'the table has four legs' - those are separate concerns."

### what analogies or metaphors fit?

| analogy | factory (tools) | product (deliverable) |
|---------|-----------------|----------------------|
| manufacturing | assembly line, tools, machines | the product design + build steps |
| construction | scaffolding, cranes, permits | the blueprint + construction steps |
| cooking | kitchen, utensils, appliances | the recipe + prepared dish |
| music | studio, instruments, mixer | the song + performance |

the key insight: **factory = tools, product = deliverable (spec + impl)**

### what terms would they use vs what terms would we use?

| user term | our term | meaning |
|-----------|----------|---------|
| "dev environment" | factory | tools we use |
| "test framework" | factory | tools for verification |
| "deployment setup" | factory | infra for delivery |
| "the actual code" | product | what we write (spec + impl) |
| "test scenarios" | repros.experience | user journey reproductions |

## evaluation

### how well does it solve the goals?

| goal | how solved |
|------|------------|
| separate concerns | two explicit blueprints with distinct purposes |
| deprecate playloops | replaced by factory + research.frames.testloops |
| organize research | 3.1.1 external, 3.1.2 internal patterns, 3.1.3 code patterns |
| user experience repros | explicit distill.repros.experience artifact |

### what are the pros?

- **clarity**: mechanic knows exactly where to look for what
- **completeness**: can't forget the tooling (factory comes first)
- **reviewability**: reviewer can assess tools separately from code design
- **reusability**: factory patterns can be shared across behaviors

### what are the cons?

- **more artifacts**: two blueprints instead of one (but clearer)
- **migration**: prior behaviors use single blueprint pattern
- **learning curve**: new concepts to internalize

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | mitigation |
|----------|------------|
| simple behavior doesn't need factory | factory can be minimal/empty, but still forces thought about verification |
| product depends on factory | order ensures factory is done first (3.3.0 before 3.3.1) |
| mechanic skips factory | route guards ensure factory stone passes before product |

## open questions & assumptions

### what assumptions have we made?

1. **order matters**: factory must be thought through before product
2. **all behaviors need verification plan**: even simple ones benefit from explicit factory
3. **research sections are meaningful**: the 3.1.1/3.1.2/3.1.3 split adds value

### what questions remain unanswered?

1. should factory have its own guard with self-reviews?
2. should product explicitly reference factory as a dependency?
3. what's the migration path for prior behaviors that use single blueprint?
4. do we need both stone and guard files for each new blueprint?

### what must we validate with the wisher before we proceed?

1. confirm the 3.1.1/3.1.2/3.1.3 research split makes sense
   - 3.1.1 = external knowledge (access, claims, domain)
   - 3.1.2 = internal patterns (templates, testframes, oss)
   - 3.1.3 = code patterns (prod, test)
2. confirm distill.repros.experience is the right name for user journey reproductions
3. confirm factory comes before product (3.3.0 vs 3.3.1)

---

## what is awkward?

### what feels off or forced?

- **numbering**: 3.3.0 and 3.3.1 suggests they're both blueprints at the same level, but factory is really foundational to product. maybe 3.2.5 for factory and 3.3 for product?
- **research section numbers**: 3.1.1, 3.1.2, 3.1.3 may feel overly hierarchical

### where does the design fight the user's mental model?

- users might expect "blueprint" to mean the code design, not the verification tooling
- "factory" is intuitive but may feel too industrial for some

### what tradeoffs feel uncomfortable?

- added complexity to the route (more stones, more files) vs clarity of separation
- "factory/product" metaphor is strong but may not resonate with all users
