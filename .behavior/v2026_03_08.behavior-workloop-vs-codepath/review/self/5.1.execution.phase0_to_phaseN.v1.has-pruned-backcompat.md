# self-review: has-pruned-backcompat

## the question

was backwards compatibility added that was not explicitly requested?

## review

### backwards compatibility found

| component | what | was it requested? |
|-----------|------|-------------------|
| product blueprint regex | matches both `3.3.1.blueprint.product.*` and `3.3.blueprint.*` (legacy) | not explicitly requested |

### analysis

the regex in `getLatestBlueprintByBehavior.ts`:
```ts
input.which === 'factory'
  ? /\.blueprint\.factory\.v(\d+)\.i(\d+)\.md$/
  : /\.blueprint(?:\.product)?\.v(\d+)\.i(\d+)\.md$/;
```

the `(?:\.product)?` makes `.product` optional, which allows legacy `3.3.blueprint.v1.i1.md` files to match.

### decision

**open question for wisher**: should we maintain backwards compat for legacy `3.3.blueprint.*` files?

options:
1. keep backcompat (current) — legacy behaviors continue to work
2. remove backcompat — require explicit migration to `3.3.1.blueprint.product.*`

**recommendation**: keep for now since prior behaviors exist with legacy names. this is a reasonable default until wisher decides otherwise.

## conclusion

one backwards compat concern found. flagged for wisher decision. kept for now as reasonable default.
