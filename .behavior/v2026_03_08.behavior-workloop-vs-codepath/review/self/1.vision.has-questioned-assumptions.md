# self-review: has-questioned-assumptions

## hidden assumptions surfaced

### assumption 1: workloops MUST come before codepaths

**what do we assume here without evidence?**
that verification must be planned before code is designed.

**what evidence supports this assumption?**
- the "machine that builds the machine" metaphor implies workloops is foundational
- you can't verify what you don't know how to test
- test-driven development (TDD) philosophy supports this

**what if the opposite were true?**
if codepaths came first, mechanics might design code without consideration for how to test it. verification would be an afterthought.

**did the wisher actually say this, or did we infer it?**
the wisher said "3.3.0.blueprint.workloops" and "3.3.1.blueprint.codepaths" - the numbers imply order, but the wisher didn't explicitly say "workloops must be done first".

**verdict**: ⚠️ inferred, should confirm with wisher

---

### assumption 2: all behaviors need both blueprints

**what do we assume here without evidence?**
that every behavior benefits from explicit workloops.

**what evidence supports this assumption?**
none explicit. we assumed it because we want consistency.

**what if the opposite were true?**
simple behaviors (e.g., rename a variable) don't need workloops. forced blueprints add overhead.

**what exceptions exist?**
- trivial behaviors with no test changes
- documentation-only behaviors
- behaviors that reuse established patterns

**verdict**: ⚠️ may need optional workloops for simple behaviors
