# self-review: has-questioned-requirements

## requirements questioned

### requirement 1: two separate blueprints (workloops vs codepaths)

**who said this was needed?**
the wisher, in 0.wish.md, explicitly asked for this separation.

**what evidence supports this requirement?**
- the current single blueprint mixes concerns
- mechanics report confusion about what's tools vs deliverable
- the "machine that builds the machine" analogy is strong

**what if we didn't do this?**
mechanics would continue to mix test setup with code design in a single artifact, which results in blueprints that are harder to review and validate independently.

**is the scope appropriate?**
yes - this is a structural change to the route, not a feature addition. it's focused.

**could we achieve the goal in a simpler way?**
we could add sections within the single blueprint, but that doesn't enforce the order (workloops first) or enable independent review.

**verdict**: ✅ justified - wisher explicitly requested, clear value proposition

### requirement 2: research split into 3.1.1, 3.1.2, 3.1.3

**who said this was needed?**
the wisher, in the "bonus ask" section of 0.wish.md.

**what evidence supports this requirement?**
the current flat 3.1.research.* list doesn't distinguish:
- external knowledge (access, claims, domain)
- internal patterns (templates, testframes, oss)
- code patterns (prod, test)

**what if we didn't do this?**
research would remain a flat list, harder to know which research is about external resources vs internal patterns.

**is the scope appropriate?**
potentially over-scoped. the wisher listed 3.1.1, 3.1.2, 3.1.3 but the distinction between 3.1.2 (internal patterns) and 3.1.3 (code patterns) is unclear. code patterns ARE internal patterns.

**could we achieve the goal in a simpler way?**
perhaps just two categories:
- 3.1.1 = external knowledge (access, claims, domain)
- 3.1.2 = internal patterns (templates, testframes, code patterns)

**verdict**: ⚠️ needs clarification - the 3.1.2 vs 3.1.3 split may be artificial
