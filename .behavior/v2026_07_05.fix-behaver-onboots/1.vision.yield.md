# 1. vision — fix behaver onboot to find `.yield.md` paths

## the wish, restated

when a session starts on a branch bound to a behavior, the `boot-behavior`
sessionstart hook injects the behavior's context (wish, vision, criteria,
blueprint) so the agent starts fully aware. after the route migration that
renamed stage outputs to `*.yield.md`, this hook went partially blind — it
still looks for the *old* filenames, so vision / criteria / blueprint context
silently stops loading. we want it to find the new paths, and we want thorough
integration tests so this gap can never silently reopen.

---

## the outcome world

### day-in-the-life

a driver binds a behavior to their branch, drives the route, and produces
`1.vision.yield.md`, `2.1.criteria.blackbox.yield.md`,
`3.3.1.blueprint.product.yield.md`, etc. the next time a session starts (or
context compacts), the boot hook re-injects **all** of that accumulated
thinking — not just the wish. the agent picks up exactly where the route left
off, with vision and blueprint in view.

### before / after

| moment | before (broken) | after (fixed) |
|--------|-----------------|---------------|
| session start on bound branch | wish loads; vision/criteria/blueprint silently absent | wish + vision + criteria + blueprint all load |
| after compaction | agent re-primed with wish only | agent re-primed with full behavior context |
| driver has produced `1.vision.yield.md` | hook looks for `1.vision.md`, finds no match | hook finds `1.vision.yield.md` |
| blueprint produced as `3.3.1.blueprint.product.yield.md` | regex expects `v{N}.i{N}.md`, finds no match | hook finds `.yield.md` blueprint |

### the "aha"

the agent, mid-route, references its own vision and blueprint without the human
re-pasting them. the context that the route worked so hard to produce actually
reaches the agent that needs it. the boot hook becomes a faithful mirror of the
route's yielded artifacts.

---

## user experience

### who & what

- **primary user**: the agent/clone driving a bound behavior route. it does not
  invoke anything directly — the `SessionStart` hook fires `boot.behavior.ts`
  automatically.
- **beneficiary human**: the driver who no longer has to manually re-feed vision
  or blueprint after a fresh session or compaction.

### contract (inputs & outputs)

- **input**: the current git branch → resolves to a bound `behaviorDir` via
  `getBranchBehaviorBind`.
- **output**: stdout blocks, one per found artifact:
  `<behavior-wish path="...">…</behavior-wish>`,
  `<behavior-vision path="...">…</behavior-vision>`, etc.
- **behavior**: each artifact is optional except the wish; absent files are
  silently skipped (except the required wish, which warns to stderr).

### the file-name contract (post-migration)

| stage | artifact the hook must find |
|-------|------------------------------|
| wish | `0.wish.md` (unchanged — the input, not a yield) |
| vision | `1.vision.yield.md` |
| criteria (blackbox) | `2.1.criteria.blackbox.yield.md` |
| criteria (blueprint) | `2.3.criteria.blueprint.yield.md` |
| blueprint (factory) | `3.3.0.blueprint.factory.yield.md` |
| blueprint (product) | `3.3.1.blueprint.product.yield.md` |

### timeline

1. driver binds behavior, drives route, yields artifacts.
2. session ends / compacts / restarts.
3. `SessionStart` hook fires → `bootBehavior()` runs.
4. hook resolves bound behaviorDir, emits each found artifact block.
5. agent reads full context in its opening turn.

## evaluation

### how well it solves the goal

- directly closes the gap: hook finds `.yield.md` artifacts.
- integration tests lock the contract: a future rename that desyncs the hook
  fails a test instead of silent context loss.

### scope boundary (names change, set does not)

this fix swaps the *names* of the artifacts the hook already surfaces (wish,
vision, criteria blackbox + blueprint, blueprint factory + product). it does
**not** expand the surfaced set — research (`3.1.*`), distill (`3.2.*`), roadmap
(`4.1`), execution (`5.1.*`), and verification (`5.3`) yields are intentionally
absent from the boot hook. to surface those work-notes would bloat every
session-start / compaction with low-signal context. an expansion of the set is a
separate wish, not this fix.

### pros

- small, contained change (one contract file + one helper + tests).
- restores the core value of the hook (full context re-injection).
- tests turn an invisible failure mode into a loud one.

### cons / tradeoffs

- **backcompat question**: legacy behaviors on disk still use `1.vision.md`,
  `2.criteria.md`, `3.3.blueprint.v1.i1.md`. do we keep finding those too, or
  only the new names? (see open questions.)
- `getLatestBlueprintByBehavior` currently version-parses `v{N}.i{N}` — the new
  yield naming has no version suffix, so its whole reason-to-exist may collapse
  to "find the single `.yield.md` blueprint".

### edge cases & pit-of-success

- both old and new blueprint files present → must pick deterministically.
- multiple binds on a branch → already handled (warn, exit 0).
- absent optional artifact → silently skipped (unchanged, correct).
- hook must never block session start → `exit 0` always (unchanged).

## open questions & assumptions

### assumptions made

- the `.yield.md` convention from commit `bcecabc` (#173, 2026-04-08) is the
  current, stable naming — confirmed by reading the live templates in
  `src/domain.operations/behavior/init/templates/`.
- the wish's "onBoot hook" = `src/contract/cli/boot.behavior.ts` (the TS impl
  the shell hook delegates to) — confirmed via grep + the shell delegator.

### open questions (triaged)

> **[superseded 2026-07-07 — wisher steer]** the backcompat questions below were
> answered "support both" on 2026-07-06, but the wisher later narrowed scope:
> **support only the current behavior template**. `0.wish.md` stays the fixed
> input name; every other artifact is a `${stoneName}.yield.md`. legacy `.md`
> behaviors are no longer found. this dissolves questions 1–4 (no precedence, no
> version-sort, no both-present tiebreak) and removes `asArtifactByPriority` +
> `getOneLatestBlueprintByBehavior` entirely. the notes below are kept as the
> historical decision trail.

1. **[answered] backcompat**: should the hook still find *legacy* names
   (`1.vision.md`, `2.criteria.md`, versioned blueprints) for old behaviors, or
   move to yield-only? resolved by the reuse decision below (per wisher,
   2026-07-06): **support both**. bhrain's `asArtifactByPriority` — which we adopt
   — explicitly ranks legacy tiers (`.v1.i1.md`, `.i1.md`, `.md` fallback) below
   the yield tiers, so legacy behaviors keep loading and yield wins when both
   exist. this is now canon, not a judgment call.
2. **[answered] blueprint helper**: since yield names carry no `v{N}.i{N}`,
   `getLatestBlueprintByBehavior` can be simplified to find the single
   `3.3.x.blueprint.*.yield.md` (legacy version-sort kept only as fallback).
   resolved via code: grep confirms `boot.behavior.ts` is the **sole production
   caller** (other matches are the helper's own test + the bind-module index), so
   it can be modified/simplified freely with no other-caller breakage.
3. **[answered] both-present precedence**: if a behavior mid-migration holds BOTH
   `1.vision.md` and `1.vision.yield.md` (or old + new blueprint), which wins?
   resolved via logic: prefer the **`.yield.md`** (current convention) when both
   exist — it reflects the driver's latest output; the legacy file is a stale
   remnant. the blueprint stage encodes this as "check yield first, fall back to
   legacy" per artifact.
4. **[wisher] multi-version legacy blueprint sort**: the blueprint finder still
   picks the highest legacy `v{N}.i{N}` (e.g. v2.i1 over v1.i1). this is finer
   than "support both" and is beyond bhrain's canonical precedence — bhrain's
   `asArtifactByPriority` only ranks `.v1.i1.md` / `.i1.md`, with no
   multi-version sort. so when radio #262 swaps to the bhrain import, this sort
   WILL be lost (old multi-version behaviors would then pick v1.i1). kept for now
   only to avoid a regression of the extant `getLatestBlueprintByBehavior.test.ts`
   (which asserts v2.i1 wins). keep it, or drop it to match canon? flagged for the
   wisher; behavior preserved until decided.

### decision: reuse bhrain artifact precedence (mirror now, import later)

bhrain already owns the canonical artifact precedence logic — this collapses Q1
(backcompat) and Q3 (both-present precedence) into a single source of truth:

- `getAllStoneArtifacts({ stone, route })` globs `${stone}.yield*` + `${stone}*.md`
- `asArtifactByPriority({ artifacts })` picks by priority:
  1. `.yield.md` → 2. `.yield.*` → 3. `.yield` → 4. `.v1.i1.md` → 5. `.i1.md` →
  fallback: first `.md`

the catch: these ops are NOT in bhrain's public `exports` (only `.`,
`./sdk/route`, `./cli/*`, `./registry`; `./sdk/route` re-exports bind/step/
`getAllStones`/`setStoneAs*` but not the artifact ops). node blocks deep imports
since there is no wildcard export.

**decided (per wisher, 2026-07-06):**
- **mirror** `asArtifactByPriority` locally in bhuild for now, with a `// TODO`
  that points at the two radio tasks below.
- **dispatch radio task → `ehmpathy/rhachet-roles-bhrain`**: export
  `asArtifactByPriority` (+ ideally `getAllStoneArtifacts`) via `./sdk/route`.
- **dispatch radio task → `@this` (bhuild)**: once bhrain publishes the export,
  swap the local mirror for the bhrain import and drop the TODO.

this keeps precedence canonical long-term while it unblocks the fix now.

## groundwork

### external research

none — no external dependencies. this is a pure internal-file-naming fix.

### internal research

- **contract to conform to**: `bootBehavior()` in
  `src/contract/cli/boot.behavior.ts:41` — emits `<behavior-{tag} path="…">`
  blocks via `outputBehaviorFile`. currently hardcodes `1.vision.md` (line 94),
  `2.1.criteria.blackbox.md` / `2.3.criteria.blueprint.md` / `2.criteria.md`
  (lines 97–111), and delegates blueprint discovery to
  `getLatestBlueprintByBehavior`.
- **blueprint helper**: `getLatestBlueprintByBehavior.ts:16` — regex
  `/\.blueprint(?:\.product)?\.v(\d+)\.i(\d+)\.md$/` (product) and
  `/\.blueprint\.factory\.v(\d+)\.i(\d+)\.md$/` (factory). neither matches
  `*.yield.md`.
- **new naming source of truth**: init templates emit `1.vision.yield.md`,
  `2.1.criteria.blackbox.yield.md`, `2.3.criteria.blueprint.yield.md`,
  `3.3.0.blueprint.factory.yield.md`, `3.3.1.blueprint.product.yield.md`
  (verified in `src/domain.operations/behavior/init/templates/*.stone`).
- **test patterns**: BDD via `test-fns` (`given/when/then/useBeforeAll`);
  domain-op tests use asset dirs under `blackbox/role=behaver/.test/assets/`
  (see `getLatestBlueprintByBehavior.test.ts`). no test currently exercises
  `bootBehavior`'s file discovery — this is the untested gap.
- **stdout format to match**: `<behavior-{tag} path="{relpath}">\n{content}\n</behavior-{tag}>` (`boot.behavior.ts:28-30`).

### the migration that caused the gap

commit `bcecabc` (#173) renamed template artifacts `.v1.i1.md → .yield.md` and
updated tests/snapshots — but `boot.behavior.ts` and
`getLatestBlueprintByBehavior.ts` were not in that changeset, so they still hunt
for the old names.

## what is awkward?

- **`getLatestBlueprintByBehavior` is now a misnomer**: "latest" implies picking
  among versions, but the new convention has exactly one `.yield.md` blueprint
  per type. keeping version-sort logic for a single-file target feels forced.
  cleanest is to find the `.yield.md` directly, with legacy version-sort only as
  a fallback branch.
- **dual-naming risk**: if we support both old and new, the hook grows a
  find-yield-then-fallback branch per artifact. slightly more code, but it keeps
  old behaviors alive. the awkwardness is real but the safety is worth it —
  pending wisher confirmation on backcompat.
- **testing a hook that writes to stdout**: the natural test target is
  `bootBehavior()` (or a testable file-discovery helper it calls), asserting
  which artifact blocks appear for a given on-disk behavior fixture. we may want
  to extract the "which files exist" decision into a pure, unit-testable
  function so tests don't have to capture stdout — worth deciding at blueprint.
