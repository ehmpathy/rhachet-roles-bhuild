# vision: structured peer reviews with levels and budgets

## the outcome world

### before

peer reviews run as flat commands without identity or resource controls:

```yaml
peer:
  - $rhx review --repo bhrain ... --output '$route/.reviews/$stone.peer-review.mech-failhides.md'
  - $rhx review --repo bhrain ... --output '$route/.reviews/$stone.peer-review.arch-opport-decomposition.md'
```

problems:
- no limit on review rounds — expensive reviews can run indefinitely
- no execution ordering — cheap and expensive reviews run in the same batch
- no per-reviewer tracking — can't see which reviewer exhausted its budget

### after

peer reviews are structured with identity, budget, and level:

```yaml
peer:
  # level 1: cheap reviewers run first (all in parallel)
  - slug: mech-failhides
    run: $rhx review --repo bhrain ... --output '$route/.reviews/$stone.peer-review.mech-failhides.md'
    budget: 5
    level: 1

  - slug: arch-opport-decomposition
    run: $rhx review --repo bhrain ... --output '$route/.reviews/$stone.peer-review.arch-opport-decomposition.md'
    budget: 5
    level: 1

  # level 3: expensive reviewers run after level 1/2 are terminal (pass or exhausted)
  - slug: behavior-intent-vs-blueprint
    run: rhx enroll claude --roles behaver,architect,mechanic -p "review the current blueprint for omissions or divergences from the wished and envisioned behavioral intent..."
    budget: 3
    level: 3
```

### the "aha" moment

when a cheap reviewer (mech-failhides) exhausts its 5-round budget without passing, the route halts for human review — but expensive reviewers (level 3) don't run unless level 1/2 are terminal. this prevents wasting expensive reviews on work that still has cheap blockers.

## user experience

### usecases

1. **guard author**: defines peer reviewers with appropriate budgets and levels
2. **brain navigating route**: sees which reviewers are active, exhausted, or awaiting
3. **human reviewing route progress**: monitors round counts and approval needs

### contract inputs & outputs

**guard file structure (input)**:
```yaml
reviews:
  peer:
    - slug: <identifier>
      run: <command>
      budget: <max-rounds>
      level: <1|2|3|...>
```

**route.drive output (stdout)**:
```
🗿 route.drive
   ├─ reviewers (level 1)
   │  ├─ mech-failhides: 2/5 rounds, active
   │  ├─ arch-opport-decomposition: 1/5 rounds, active
   │  └─ arch-smell-scopeleaks: 5/5 rounds, exhausted
   │
   ├─ reviewers (level 3)
   │  └─ behavior-intent-vs-blueprint: awaiting (level 1 not terminal)
```

### timeline

1. brain creates artifact, runs `--as passed`
2. level 1 reviewers run (all in parallel)
3. brain addresses blockers, re-runs `--as passed`
4. level 1 reviewers run again (round 2/5)
5. ... iterate until level 1 reviewers pass or exhaust
6. once level 1 is terminal, level 2 unlocks (if any)
7. once level 2 is terminal, level 3 unlocks (if any)
8. once all levels terminal, judges evaluate

## mental model

### how users describe it

"cheaper reviewers run first — once they're done or exhausted, more expensive ones kick in. each reviewer has a budget cap so we don't spin forever."

### analogies

- **tiered customer support**: level 1 support handles common issues; only escalates to level 2 (specialists) when stuck
- **assembly line QA**: basic checks first, detailed inspection after basic passes
- **code review workflow**: linters run before humans review; no human time wasted on lint failures

### terms

| user term | system term |
|-----------|-------------|
| "budget" | `budget` field — max rounds |
| "level" | `level` field — execution tier (1 runs before 2) |
| "exhausted" | reviewer used all budget rounds without passing |
| "terminal" | all reviewers at a level are either passed or exhausted |
| "unlocked" | higher level becomes active once prior level is terminal |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| limit expensive review rounds | yes — budget caps each reviewer |
| prioritize cheap reviews | yes — level 1 runs before level 2/3 |
| track per-reviewer state | yes — slug identifies each reviewer |
| human oversight on exhaustion | yes — exhaustion requires approval to proceed |

### pros

- prevents runaway review loops (budget cap)
- saves money by running cheap reviewers first
- clear visibility into review state
- backwards compatible (reviewers without budget/level use defaults)

### cons

- guard files become more verbose (structured vs simple command)
- authors must understand level/budget semantics

### edge cases

| edge case | pit of success behavior |
|-----------|------------------------|
| reviewer without budget/level fields | defaults to budget: infinity, level: 1 (backwards compat) |
| all reviewers at level exhausted | route halts, requires human approval |
| level 2 reviewer but no level 1 | level 2 runs immediately (no prior level to await) |
| reviewer passes before budget exhausted | terminates as "passed", releases next level |

## what is awkward?

### verbosity increase

converting from:
```yaml
peer:
  - $rhx review ... --output '$route/.reviews/$stone.peer-review.mech-failhides.md'
```

to:
```yaml
peer:
  - slug: mech-failhides
    run: $rhx review ... --output '$route/.reviews/$stone.peer-review.mech-failhides.md'
    budget: 5
    level: 1
```

this is 4x more lines per reviewer. however:
- clear semantics outweigh brevity
- structured format enables bhrain features (budget tracking, level gating)
- one-time migration cost

### level 3 gap

if all current reviews go to level 1, and expensive reviews go to level 3, there's no level 2. this is fine — levels can have gaps — but might feel odd to some authors.

### slug naming convention

current pattern uses `# slug = mech-failhides` as comment. new structured format uses `slug: mech-failhides` field. the comment convention is now redundant but harmless — could be kept for human readability or removed for DRYness.
