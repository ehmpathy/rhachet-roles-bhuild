## assumption 1: 5 whys is universally applicable

**what do we assume here?**
that both bugs and features benefit from "why?" drill.

**what evidence supports this?**
- toyota production system uses 5 whys for defects
- many orgs use "why?" to validate feature requests
- wisher explicitly mentioned both: "if there's an issue - or built the solution at the root cause, if there's a new feature"

**what if the opposite were true?**
- some bugs have obvious root causes (typo, copy-paste error)
- some features are purely additive (no "why is this broken" to answer)
- to force 5 whys might feel bureaucratic

**did the wisher actually say this?**
yes — wisher mentioned both issues and features in the wish.

**what exceptions exist?**
- trivial bugs: typo in error message, obvious copy-paste mistake
- greenfield features: no prior state to analyze

**verdict**: VALID assumption, but overhead concern is real — documented in "what is awkward" section

## assumption 2: root cause is identifiable

**what do we assume here?**
that we can reach a sensible root cause within reasonable depth.

**what evidence supports this?**
- 5 whys typically converges after 3-7 iterations
- most technical issues have identifiable architectural causes

**what if the opposite were true?**
- some issues are emergent (no single root cause)
- some root causes are organizational or historical (outside scope)
- analysis paralysis: forever ask "why?" without answer

**did the wisher actually say this?**
no — wisher assumed root cause exists by ask to "solve at root cause".

**what exceptions exist?**
- distributed system failures with multiple factors
- race conditions where "root cause" is the architecture itself
- legacy code where original reason is lost

**verdict**: VALID assumption with limits — vision documents "5 whys goes too deep" as edgecase

## assumption 3: self-review can validate alignment

**what do we assume here?**
that a self-review question can sensibly assess whether blueprint addresses rootcause.

**what evidence supports this?**
- extant self-reviews in blueprint guard work well (coverage, adherance)
- review questions can prompt structured comparison

**what if the opposite were true?**
- reviewer might rubber-stamp without deep analysis
- false positives: blueprint appears aligned but isn't
- false negatives: valid symptom treatment flagged as wrong

**did the wisher actually say this?**
yes — wisher asked for "a self review to the blueprint that verifies".

**what exceptions exist?**
- complex multi-layer root causes where alignment is subjective
- time-boxed behaviors where full root cause fix is out of scope

**verdict**: VALID assumption — documented "justified symptom treatment" escape hatch

## assumption 4: research stone before distillation

**what do we assume here?**
that rootcause research should complete BEFORE domain distillation.

**what evidence supports this?**
- root cause informs what domain concepts matter
- distill without rootcause might focus on wrong abstractions

**what if the opposite were true?**
- domain distillation might reveal root cause naturally
- could do rootcause analysis after code research, not before distill

**did the wisher actually say this?**
no — wisher didn't specify order, just asked for "dedicated stone".

**what exceptions exist?**
- iterative discovery: rootcause emerges in distillation
- parallel work: do both, cross-reference

**verdict**: INFERRED assumption — should validate with wisher

**action**: added to open questions: "where does 3.1.5 rootcause fit in the stone order?"

---

## assumption 5: justified symptom treatment is acceptable

**what do we assume here?**
that blueprints can explicitly choose to treat symptoms if root cause is out of scope, and the self-review will accept this.

**what evidence supports this?**
- pragmatic: sometimes root cause is too expensive to fix
- documented escape: "root cause is X, but we treat symptom Y because [reason]"

**what if the opposite were true?**
- stricter: never allow symptom treatment, always fix root cause
- this would block behaviors where root cause is in another repo/team

**did the wisher actually say this?**
no — wisher said "solve at root cause" without escape clause.

**what exceptions exist?**
- root cause in third-party library
- root cause requires org-level change
- time-sensitive fix where symptom treatment is pragmatic

**verdict**: INFERRED assumption — should validate with wisher

**action**: added to open questions: "confirm 'justified symptom treatment' escape hatch is appropriate"

---

## assumption 6: branches in 5 whys are always valuable

**what do we assume here?**
that to explore multiple branches is worth the effort.

**what evidence supports this?**
- wisher explicitly asked for "full branches of options"
- multiple hypotheses prevent tunnel vision

**what if the opposite were true?**
- most issues have one obvious root cause path
- branches add complexity without proportional value
- analysis paralysis: too many branches to explore

**did the wisher actually say this?**
yes — "drill down 5 whys, w/ full branches of options".

**what exceptions exist?**
- obvious single-path issues
- time-boxed analysis where branches are luxury

**verdict**: VALID assumption — wisher explicitly requested branches
