- 3.1.2 = external.factory (testloops, oss.levers, templates)
- 3.1.3 = internal.product.code (prod, test)

### question 3: what if root cause requires multi-repo changes?

**reason**:
- common scenario: root cause is in a shared library, upstream service, or infrastructure
- behavior scope is typically single-repo
- mechanic cannot fix external repos in same behavior

**answer**: document the root cause, then scope the behavior to what CAN be changed.

pattern:
```
root cause identified: absent validation in shared-auth-lib
behavior scope: this repo cannot change shared-auth-lib
approach: add validation layer in this repo that compensates
note: opened issue #123 in shared-auth-lib for proper fix
```

the self-review accepts this as "justified symptom treatment" if:
1. root cause is clearly documented
2. scope limitation is explained
3. follow-up is tracked (issue, task, etc.)

**action**: this is already covered by the "justified symptom treatment" edgecase in vision

### question 4: blueprint guard only or execution guard too?

**reason**:
- blueprint guard: validates the PLAN addresses root cause
- execution guard: validates the WORK addresses root cause

pros of blueprint only:
- catches issues before work begins
- avoids duplicate review effort

pros of also execution:
- ensures execution didn't drift from blueprint
- catches cases where blueprint was vague

**answer**: blueprint guard is sufficient.

rationale:
- if blueprint explicitly addresses root cause, execution follows
- if execution drifts, extant guards (coverage, adherance) catch it
- double root-cause check is redundant

**action**: vision already states blueprint guard — no change needed
