# factory opportunities research

## the constraint

**bottleneck: manual test verification of reflection stone prompts**

the template files are static text — the only way to verify prompt quality is to use them in a real behavior and see if they surface good reflections. this is inherently manual and slow.

---

## flow and queues

- **queue**: wisher approval queue for behavior routes
- **wait**: stillness gate on self-reviews (~90 seconds)
- **wip limit**: one behavior at a time on a branch

no changes proposed — these are intentional friction for deliberate work.

---

## waste

- **handoffs**: none identified
- **setup**: temp directory creation in tests is minimal
- **partial work**: none identified
- **manual errors**: template variable `$BEHAVIOR_DIR_REL` must be spelled correctly — typos fail silently

**minor improvement**: lint step to verify template variables are valid.

---

## small batches

- **incremental verification**: yes — each template can be tested independently
- **smallest unit**: single template file
- **early feedback**: tests verify file creation and content; prompt quality requires usage

---

## autonomy

- **independence**: full — template changes are self-contained
- **consultations**: wisher approval for vision and playtest stones
- **external blocks**: none

---

## density

- **execution vs setup**: ~80% execution, ~20% setup
- **time waste**: none significant — tests are fast

---

## systems view

**improvement scope**: this behavior benefits all future behaviors

reflection stones become part of the standard template set. every behavior route that uses them will benefit from better prompts.

---

## recommended improvement

**none required for this behavior.**

the factory is well-suited for template additions. no changes needed.

**optional future improvement**: prompt lint to verify reflection stone outputs include expected sections (audience tiers, 5 whys tree structure, premortem scenarios).
