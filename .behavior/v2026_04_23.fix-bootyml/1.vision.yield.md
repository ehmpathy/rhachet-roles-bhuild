# vision: boot.yml for bhuild roles

## the outcome world

### before

every time a behaver boots, all 21 briefs load as full `say` content:
- behavior.blueprint (4 briefs)
- behavior.criteria (11 briefs)
- behavior.roadmap (2 briefs)
- behavior.verification (3 briefs)
- behavior.wish (2 briefs)

this burns ~13,000 tokens just on behaver briefs alone — every single session.

decomposer has 7 briefs, same story.

the briefs load whether or not they're relevant to the current task.

### after

boot.yml files exist for each role with briefs:
- behaver (21 briefs → categorized into say vs ref)
- decomposer (7 briefs → categorized into say vs ref)

the most important briefs load as `say` (full content).
reference/supplementary briefs load as `ref` (path only, content on demand).

### the "aha" moment

the developer runs `rhx route.drive` and sees the boot summary:
```
briefs = 21
├── say = 5
└── ref = 16
tokens ≈ 4,200 (down from 13,000)
```

~70% token reduction on behaver role alone.

## user experience

### usecases

1. **clone boots behaver role** — essential briefs load (bdd format, clear desires), reference briefs available but not in context
2. **clone needs verification philosophy** — reads the ref, philosophy loads on demand
3. **developer adds new brief to behaver** — adds to boot.yml under appropriate subject/say/ref category

### contract inputs & outputs

**input**: boot.yml in role directory
```yaml
always:
  briefs:
    say:
      - briefs/practices/behavior.criteria/rule.require.bdd-format.md
      - briefs/practices/behavior.wish/rule.prefer.clear-desires.md
    ref:
      - briefs/practices/behavior.verification/philosophy.verification-strictness.md
      - briefs/practices/behavior.blueprint/rule.require.criteria-satisfied.md
```

**output**: selective brief load
- `say` briefs: full content in context
- `ref` briefs: path reference, content on demand

### timeline

1. boot.yml created for behaver
2. boot.yml created for decomposer
3. token usage drops immediately on next session

## mental model

### how users describe this

> "boot.yml is like a table of contents — it tells the clone what's essential to read now and what's available for reference later."

### analogies

- **book index vs full text**: say = chapters you read, ref = index entries you look up
- **imports vs lazy load**: say = eager load, ref = lazy load
- **active vs archived**: say = active knowledge, ref = archived knowledge

### terms

| user says | we say |
|-----------|--------|
| "always load this" | always.say |
| "reference only" | always.ref |
| "load when on X" | subject.X.say |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| reduce token waste | yes — ref briefs don't consume context |
| preserve brief availability | yes — refs are available on demand |
| explicit categorization | yes — boot.yml declares what's essential |

### pros

- immediate token savings (~70% for behaver)
- explicit declaration of brief importance
- follows established pattern (mechanic, driver already have boot.yml)
- no code changes needed — just yaml config

### cons

- requires manual categorization of briefs
- must remember to update boot.yml when briefs are added

### edge cases

| edge case | pit of success |
|-----------|----------------|
| brief not in boot.yml | rhachet warns about uncategorized briefs |
| typo in brief path | rhachet fails fast with clear error |
| new brief added | brief.ref path shows in boot summary as "also" |

## what is awkward?

### categorization decisions

to decide which briefs are `say` vs `ref` requires judgment:
- **say candidates**: rules that apply to every behavior (bdd-format, clear-desires)
- **ref candidates**: definitions, philosophy, supplementary examples

the mechanic boot.yml provides guidance — it categorizes:
- core rules as `say`
- demos, seeds, supplements as `ref`

### no subject categories for behaver?

the mechanic has `subject.code.prod`, `subject.code.test`, etc.

behaver works on behaviors, not code — all briefs are always relevant. no subject-based load seems needed. but this could be revisited if behaver grows more specialized tasks.
