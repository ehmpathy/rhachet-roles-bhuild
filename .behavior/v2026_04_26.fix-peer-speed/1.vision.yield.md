# vision: fix peer review speed

## the outcome world

### before

- mechanic finishes execution stone work
- runs `rhx route.stone.set --as passed`
- guard triggers 5 sequential `rhachet enroll claude` peer reviews
- **15 minutes pass** with no progress feedback
- mechanic (and human) sit idle, unsure if stuck or working
- sometimes hangs indefinitely with no timeout

### after

- mechanic finishes execution stone work
- runs `rhx route.stone.set --as passed`
- guard completes in < 30 seconds
- feedback loop stays tight, momentum preserved

### the "aha" moment

"i no longer dread running `--as passed` because it doesn't eat 15 minutes of my life"

## user experience

### usecases

| goal | before | after |
|------|--------|-------|
| pass execution stone | wait 15min | wait <30s |
| debug hung guard | unclear if stuck | timeout surfaces issue |
| iterate on code | momentum killed | momentum preserved |

  - ./node_modules/.bin/rhachet enroll claude --roles ... -p "review..."

### pros/cons

**option A: comment out**
- ✅ immediate fix
- ✅ minimal risk
- ✅ comments document why (comeback later)
- ❌ loses peer review value
- ❌ technical debt

**option B: add timeout**
- ✅ immediate fix
- ✅ still attempts reviews
- ⚠️ may kill valid long reviews
- ⚠️ doesn't fix root slowness

**option C: redesign** (future)
- ✅ proper solution
- ❌ significant effort
- ❌ not what wisher asked for right now

### recommendation

**option A: comment out** — per wisher's request:
> "if we cant find a way to make them fast, lets just comment them out in the guard files (comment, not remove, so we remember to come back to them)"

the root cause (sequential claude spawns) can't be fixed quickly. commenting out with clear documentation preserves the intent while unblocking work.

## what is awkward?

### losing peer review value

commenting out removes a quality gate. the tradeoff is:
- peer reviews that take 15min aren't actually used (people skip/cancel)
- a fast-but-absent review is no worse than a slow-and-skipped review

### no clear comeback mechanism

the comment says "come back later" but there's no:
- tracking issue
- follow-up wish
- timeline

**mitigation**: document the intent clearly in comments and optionally create a follow-up wish for "fast peer reviews"

### the real fix is harder

the proper solution would be:
- reuse extant claude session for reviews
- or use lighter-weight review mechanism (grep, ast, etc.)
- or parallelize reviews

but that's a bigger architectural change, not what's asked for now.
