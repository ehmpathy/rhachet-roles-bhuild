# research: leverage measurement in software engineering

this document captures research on how leverage is defined and measured by researchers, builders, and industry practitioners.


---


## 1. definitions of leverage


### 1.1. technical leverage (academic definition)

from the foundational ICSE 2021 paper by massacci and pashchenko:

> "in finance, leverage is the ratio between assets borrowed from others and one's own assets. a matching situation is present in software: by using free open-source software (FOSS) libraries a developer leverages on other people's code to multiply the offered functionalities with a much smaller own codebase."

**formula:**
```
technical_leverage = dependencies_code / own_code
```

**key findings:**
- small python packages have median leverage of 6.9x their own code
- large packages have median leverage of 0.1x (rely less on dependencies)
- high leverage correlates with 60% higher vulnerability risk (maven), 4-7x higher risk (npm)
- leverage pays off: 4% delay in release cycle provides 4x more code

**citation:** [massacci & pashchenko, "technical leverage in a software ecosystem: development opportunities and security risks", ICSE 2021](https://dl.acm.org/doi/abs/10.1109/ICSE43902.2021.00125)


### 1.2. force multiplier (industry definition)

from software engineering practice:

> "a force multiplier is something that makes something else more effective. in software engineering, it refers to tools and practices that enhance productivity and efficiency, allowing engineers to achieve greater output with the same resources."

**key aspects:**
- automation is a force multiplier: makes processes reproducible, then scales to parallel executions
- engineers can be force multipliers: mentoring, creating tools, encoding knowledge in libraries
- CI/CD pipelines amplify developer productivity, reduce errors, increase release frequency

**citations:**
- [matthew mayer, "software development force multipliers"](https://matthewkmayer.github.io/blag/public/post/software-dev-force-multipliers/)
- [steve smith (ardalis), "becoming a developer team force multiplier"](https://ardalis.com/becoming-a-developer-team-force-multiplier/)

## 2. industry frameworks for measuring leverage


### 2.1. DORA metrics

the devops research and assessment (DORA) framework measures software delivery performance via four key metrics:

| metric | measures | elite benchmark |
|--------|----------|-----------------|
| deployment frequency | throughput | on-demand (multiple/day) |
| lead time for changes | throughput | < 1 hour |
| change failure rate | stability | < 5% |
| failed deployment recovery time | stability | < 1 hour |

**leverage connection:**
- higher throughput with maintained stability = leveraged development
- documentation quality doubles likelihood of meeting targets
- platform engineering boosts productivity when done right

**2024 findings:**
- elite cluster stable, but high-performance cluster shrunk from 31% to 22%
- AI boosts individual productivity but worsens team delivery performance
- psychological safety is strongest predictor of delivery performance

**citations:**
- [DORA 2024 accelerate state of devops report](https://dora.dev/research/2024/dora-report/)
- [google cloud blog, "announcing the 2024 DORA report"](https://cloud.google.com/blog/products/devops-sre/announcing-the-2024-dora-report)


### 2.2. SPACE framework

developed by microsoft research (storey, forsgren, maddila, zimmermann), SPACE captures five dimensions:

| dimension | what it measures |
|-----------|------------------|
| **S**atisfaction & well-being | developer fulfillment, health, work impact |
| **P**erformance | outcome quality, product impact |
| **A**ctivity | coding, testing, debugging, collaboration levels |
| **C**ommunication & collaboration | code review quality, integration velocity, onboarding time |
| **E**fficiency & flow | development smoothness from ideation to deployment |

**leverage connection:**
- efficiency and flow dimension directly measures leverage: minimizing waste, maximizing value creation
- productivity cannot be reduced to a single dimension — requires constellation of metrics
- SPACE's 20 metrics span entire developer experience vs DORA's 4 pipeline metrics

**citation:** [storey et al., "the SPACE of developer productivity", ACM queue 2021](https://queue.acm.org/detail.cfm?id=3454124)


### 2.3. DX core 4 framework

a unified framework that encapsulates DORA, SPACE, and devex:

| dimension | what it measures |
|-----------|------------------|
| **speed** | time to deliver production-ready code |
| **effectiveness** | navigation of development processes/workflows |
| **quality** | stability and reliability in production |
| **impact** | contribution to business value beyond code |

**leverage connection:**
- developer experience index (DXI): every 1-point increase saves ~10 minutes/week/engineer
- vercel reduced cycle times by ~50% using DX core 4 signals
- measures at team level, not individual (avoids goodhart's law)

**citation:** [getdx, "measuring developer productivity with the DX core 4"](https://getdx.com/research/measuring-developer-productivity-with-the-dx-core-4/)


### 2.4. mckinsey developer velocity index (DVI)

46 drivers across 13 capability areas measuring developer velocity:

**key findings:**
- top-quartile DVI companies outperform bottom-quartile by 5x on revenue growth
- best-in-class tools are top contributor to business success
- top-quartile companies score 36% higher on open-source adoption

**leverage connection:**
- distinguishes inner loop (writing, testing, building, deploying) from outer loop (admin tasks)
- leverage = maximizing inner loop time, minimizing outer loop overhead
- tools enable greater productivity, visibility, and coordination

**citation:** [mckinsey, "developer velocity: how software excellence fuels business performance"](https://www.mckinsey.com/industries/technology-media-and-telecommunications/our-insights/developer-velocity-how-software-excellence-fuels-business-performance)

## 3. platform engineering and leverage


### 3.1. internal developer platform (IDP) metrics

platform engineering creates a "source multiplier effect":

**productivity metrics tracked:**
- developer productivity (64% of orgs)
- fewer security incidents (62%)
- software release frequency (59%)
- developer satisfaction (58%)
- cost reduction (50%)

**leverage measurements:**
- provisioning time: from hours/days → seconds/minutes
- developer autonomy: inverse of support requests per year
- time to tenth PR: normalized onboarding efficiency

**impact statistics:**
- companies using IDPs deliver updates ~40% faster
- operational overhead cut nearly in half
- ~40% higher developer NPS from smoother workflows

**citations:**
- [mia-platform, "the metrics of a successful internal developer platform"](https://mia-platform.eu/blog/metrics-internal-developer-platform/)
- [cortex, "the 2024 state of developer productivity"](https://www.cortex.io/report/the-2024-state-of-developer-productivity)


### 3.2. platform engineering adoption

- 83% of orgs have adopted or planning platform engineering
- 20% fully implemented, 33% in progress
- small teams can support thousands of developers

**leverage insight:**
> "developers spend as little as 12.5% to 30% of their time per week writing code. platform engineering enables developers to focus on business logic versus infrastructure mastery."

**citation:** [cloudbees, "platform engineering is a key driver of developer productivity"](https://www.cloudbees.com/newsroom/survey-platform-engineering-key-driver-developer-productivity)

## 4. ROI and efficiency measurement


### 4.1. engineering ROI metrics

**formula:**
```
software_roi = (gains - costs) / costs
```

**industry benchmarks:**
- decent IT project ROI: 5-10% (isixsigma)
- good ROI: ~20% (clearpoint strategy)
- advanced efficiency metrics users: 45% better resource utilization (gartner 2025)
- structured technical debt tracking: 47% higher maintenance efficiency

**time-based leverage:**
- return on time saved quantifies value of automation
- measures workforce productivity and cost reduction

**citation:** [fullscale, "the complete guide to engineering ROI"](https://fullscale.io/blog/measuring-engineering-roi-guide/)


### 4.2. developer experience index (DXI)

14 dimensions combined into single score:
- predictive against velocity, quality, efficiency outcomes
- actionable for management decisions

**leverage insight:**
> "companies are feeling increased pressure to measure and improve developer productivity as markets emphasize efficient growth and ROI."

**citation:** [getdx, "developer experience index: the one number you need to increase ROI per engineer"](https://getdx.com/research/the-one-number-you-need-to-increase-roi-per-engineer/)

## 5. software reuse metrics


### 5.1. reuse metrics categories

six types of reuse measurement (frakes & terry):

| category | what it measures |
|----------|------------------|
| reuse cost-benefit models | economic analysis, quality/productivity payoff |
| maturity assessment models | how advanced reuse programs are |
| amount of reuse metrics | percentages of reuse for lifecycle objects |
| failure modes analysis | impediments to reuse |
| reusability metrics | likelihood artifact is reusable |
| reuse library metrics | library effectiveness |

**leverage connection:**
- technical leverage is a form of reuse metric
- measures dependencies as "borrowed assets"

**citation:** [frakes & terry, "software reuse: metrics and models", ACM computing surveys](https://dl.acm.org/doi/pdf/10.1145/234528.234531)

## 6. synthesis: leverage measurement approaches


### 6.1. dimension categories

based on research, leverage can be measured across:

| dimension | approach | examples |
|-----------|----------|----------|
| **code leverage** | ratio of dependencies to own code | technical leverage formula |
| **time leverage** | time saved per unit effort | ROI on time saved |
| **velocity leverage** | throughput improvement | deployment frequency increase |
| **maintenance leverage** | ongoing cost reduction | incident reduction, MTTR improvement |
| **knowledge leverage** | skill/knowledge multiplication | onboarding time, documentation quality |
| **automation leverage** | manual effort elimination | provisioning time reduction |


### 6.2. measurement levels

| level | what to measure | source |
|-------|-----------------|--------|
| **individual** | code contributions, review turnaround | SPACE activity |
| **team** | deployment frequency, cycle time | DORA metrics |
| **platform** | developer autonomy, provisioning time | IDP metrics |
| **organization** | revenue growth correlation, DVI score | mckinsey DVI |


### 6.3. formulas observed in industry

```
technical_leverage = dependencies_loc / own_loc

velocity_leverage = output_with_tool / output_without_tool

time_leverage = time_saved / time_invested

roi_leverage = (gains - costs) / costs

platform_leverage = developers_supported / platform_team_size
```

### 8.3. key insights from research

1. **leverage is a ratio, not absolute value**: always measured relative to baseline
2. **leverage has dual nature**: opportunity (productivity gain) and risk (dependency exposure)
3. **leverage compounds transitively**: unblocking one behavior may unblock many others
4. **leverage requires multi-dimensional measurement**: no single metric captures it
5. **leverage measurement should avoid individual targeting**: team/system level to prevent gaming
