# vision: fix radio task skill auth

## the outcome world

### day-in-the-life

a beaver (robot) runs `rhx radio.task.push --via gh.issues --into @this --title "..."`.

behind the scenes, the skill needs a github token from keyrack. today, if the keyrack
vault is locked, the whole thing face-plants with a raw `MalfunctionError` dumping keyrack
stdout — even though the fix is a trivial one-command unlock. the robot doesn't know that,
so it either gives up or spams the human.

with this wish fulfilled:

- **locked vault → just works (with a breadcrumb).** the skill auto-unlocks keyrack for the
  beaver token before it even tries to read the grant, and emits a one-line "🔓 unlocked keyrack"
  note so the log shows what happened. the robot never sees a lock error for the happy path.
- **absent key → a helpful nudge with two ways forward.** if the beaver token simply was never
  configured on this host, the skill names both fixes: *"key is not set — either set the beaver
  key (`rhx keyrack set ...`) to stay as-robot, or re-run with `--auth as-human`."* it points to
  the proper as-robot setup **and** the as-human fallback, instead of a stack trace. keyrack's
  own `absent` attempt already supplies the exact `rhx keyrack set` command to reuse.
- **wrong/broken auth → a loud malfunction.** if the key *is* configured but the vault
  can't grant it (blocked, corrupt, unlock genuinely failed), the skill fails loud with a
  real diagnosis. no silent guessing, no misleading "try as-human" when the key is present.

### before / after contrast

| scenario | before | after |
|----------|--------|-------|
| vault locked | `MalfunctionError: keyrack: ...` raw dump | auto-unlock, retry, succeed |
| key absent (never set) | `MalfunctionError` (same as all failures) | `ConstraintError`: set the beaver key (`rhx keyrack set ...`) **or** re-run `--auth as-human` |
| key set but grant blocked | `MalfunctionError` | `MalfunctionError` with real cause (unchanged, correct) |

the "aha" moment: **the error you get tells you exactly whose problem it is.** absent = "you
haven't set it up, here's the shortcut." blocked/locked-after-unlock = "something is actually
wrong, here's the diagnosis." the robot stops treating every auth hiccup as fatal.

## user experience

### usecases & goals

- **beaver pushes/pulls radio tasks** without babysitting keyrack unlock state.
- **human on a fresh host** gets guided to **both** fixes — set the beaver key to stay as-robot,
  or fall back to `--auth as-human` — instead of hunting docs, when the token isn't configured.
- **operator diagnosing a real auth failure** gets a truthful malfunction, not a misleading
  "just auth as human" that would mask a genuine config problem.

### contract inputs & outputs

the contract surface is the extant `--auth` flag on `radio.task.push` / `radio.task.pull`.
no new required inputs. behavior changes live inside the default `as-robot:via-keyrack(owner)`
path and its error handler.

**scope note:** the fix lands at the shared auth layer (`getOneRadioContextFromCliArgs` →
`getGithubTokenByAuthArg` → `getAuthFromKeyrack`), which **both** `radio.task.push` and
`radio.task.pull` route through. so a single change covers both skills — the push-only examples
below are illustrative, not a scope limit.

inputs (unchanged shape):
- `--auth as-robot:via-keyrack(owner)` — default (`ehmpath`); **now auto-unlocks first**
- `--auth as-robot:env(VAR)` — explicit token from env
- `--auth as-robot:shx(cmd)` — explicit token from shell
- `--auth as-human` — gh cli session (null token)

outputs (new distinctions on the keyrack path):
- **granted** → `{ token, role: 'as-robot' }` (unchanged); if an auto-unlock ran first, emit a
  one-line "🔓 unlocked keyrack" note
- **absent** → `ConstraintError` naming both fixes: set the beaver key (`rhx keyrack set ...`,
  reusing keyrack's own `fix` hint) to stay as-robot, or re-run with `--auth as-human`
- **locked** → auto-unlock, retry; still locked → `MalfunctionError` that forwards the raw
  keyrack stdout (with its `fix` hint)
- **blocked** → `MalfunctionError` that forwards the raw keyrack stdout (real diagnosis)

### what leverage looks like

```sh
# happy path — vault was locked, now transparently unlocked
rhx radio.task.push --via gh.issues --into @this --title "feat: X"
# → (auto-unlocks keyrack, grants beaver token, pushes issue)

# fresh host, beaver token never configured
rhx radio.task.push --via gh.issues --into @this --title "feat: X"
# → ✋ key is not set. either:
#      • set the beaver key:  rhx keyrack set --key EHMPATH_BEAVER_GITHUB_TOKEN --env prep
#      • or re-run as human:  --auth as-human

# path A — human sets the beaver key, then re-runs as robot
rhx keyrack set --key EHMPATH_BEAVER_GITHUB_TOKEN --env prep
rhx radio.task.push --via gh.issues --into @this --title "feat: X"
# → (auto-unlocks, grants beaver token, pushes issue)

# path B — human uses their own gh cli session instead
rhx radio.task.push --via gh.issues --into @this --title "feat: X" --auth as-human
# → (uses gh cli session)
```

### timeline

1. skill derives auth via `getGithubTokenByAuthArg`
2. default → keyrack path → **auto-unlock** → `keyrack.get`
3. branch on grant status: granted / absent / locked / blocked
4. either return token, guide to as-human, or malfunction
5. token flows to `daoRadioTaskViaGhIssues.runGhCommand` as `GH_TOKEN`

## mental model

- to a friend: *"the radio skill now unlocks its own keyring, and if the key was never cut it
  tells you to just use your own login instead of crashing."*
- analogy: a smart lock that **tries its own keycard first** (auto-unlock), and if there's no
  card on file **buzzes you to badge in yourself** (as-human) — but if the card is present and
  the reader rejects it, it raises a real alarm (malfunction).

| our terms | user terms |
|-----------|------------|
| keyrack grant status `absent` | "the key isn't set / never configured" |
| keyrack grant status `locked` | "the vault is locked" |
| auto-unlock | "unlock the keyring for me" |
| `as-human` | "use my own github login" |
| `MalfunctionError` | "something is actually broken" |
| `ConstraintError` | "you need to do/choose something" |

## evaluation

### how well it solves the goals

- ✅ auto-unlock before robot auth flow → covers task #251
- ✅ absent → guide to as-human → covers task #256
- ✅ "only if explicitly absent, otherwise malfunction if set but wrong" → maps cleanly onto
  the keyrack `absent` vs `locked`/`blocked` status distinction

### pros

- reuses the keyrack grant-status discriminated union that is already modeled — the
  distinction the wish asks for is *already present*, we just weren't branching on it
- error class correctly signals ownership: `ConstraintError` (caller must choose) vs
  `MalfunctionError` (system broken)
- happy-path friction (locked vault) disappears

### cons / tradeoffs

- auto-unlock introduces a side-effect (unlock) into what was a pure read; must stay
  idempotent and safe to re-run
- headless/CI unlock — **[wisher decision 2026-07-05]** headless is fine, no guard needed:
  the unlock targets the **exact key** (`--key EHMPATH_BEAVER_GITHUB_TOKEN`), not the whole
  env, and the beaver key is well-scoped so it grants non-interactively even headless. the
  earlier "guard non-TTY → malfunction" requirement is **retracted** by the wisher.
- the friendly "want to try --auth as-human?" must **not** fire in test env (as-human is
  already forbidden there) — absent in tests should still fail loud

### edgecases & pit-of-success

| edgecase | keep-in-pit behavior |
|----------|---------------------|
| absent in test env | still fail loud — do not suggest as-human (tests forbid it) |
| unlock succeeds but grant still absent | absent guidance (key truly not set) |
| headless unlock | fine — key-scoped unlock grants non-interactively; no guard needed [wisher 2026-07-05] |
| unlock succeeds but grant still locked | malfunction that forwards keyrack stdout |
| explicit `--auth as-robot:env/shx` | no keyrack, no auto-unlock — untouched |
| non-keyrack owners | same status-branch logic applies for any owner |

## open questions & assumptions

### assumptions

1. **`absent` is the right signal for "key is not set."** the keyrack type docs say
   `absent = 'key not configured on host'` — this is exactly the wish's "explicitly absent."
2. **auto-unlock is available programmatically or via `rhx keyrack unlock`.** the permission
   list pre-approves `rhx keyrack unlock --owner ehmpath --env ...`, and rhachet ships
   `unlockKeyrackKeys` internally.
3. the friendly guidance belongs in `getAuthFromKeyrack` (or a thin wrapper) since that's
   where the grant status is inspected today.

### triaged questions — [answered] / [research] / [wisher]

**[decided by wisher] — settled, locked into the design:**
- **Q1 → EMIT.** auto-unlock emits a one-line note on success (e.g. "🔓 unlocked keyrack") for
  observability — not silent. the robot's log shows that an unlock happened.
- **Q2 → two-path nudge (confirmed).** absent offers **both** "set the beaver key
  (`rhx keyrack set ...`)" AND "`--auth as-human`". one micro-choice left for blueprint: exact
  copy / option order / whether to echo the key slug + owner.
- **Q3 → FORWARD keyrack stdout.** for `locked`-after-unlock and `blocked`, the malfunction
  forwards the raw keyrack stdout (which already carries the per-status `fix` hint from
  `getKeyrackKeyGrant.js:79-88`). no cleaned-up rewrite — keep the full diagnostic.

**[research → mostly answered] — one design constraint remains for blueprint:**
- **A7:** read `unlockKeyrackKeys.js`. findings: (a) it is idempotent-friendly — checks
  `isUnlocked` before it unlocks and dedupes by slug; (b) it has **prerequisites** — needs a
  `hostManifest` (`rhx keyrack init`) AND a repo `keyrack.yml` that declares the key, else it
  fails fast; (c) headless: the unlock targets the **exact key**, and per the wisher the beaver
  key is well-scoped so it grants non-interactively even headless. blueprint must (1) confirm the
  beaver token is declared in the radio repo's `keyrack.yml`.
  **prerequisite (1) confirmed:** `src/domain.roles/dispatcher/keyrack.yml:10` declares
  `EHMPATH_BEAVER_GITHUB_TOKEN` under `env.prep` (org `ehmpathy`), which matches the `env:'prep'`
  grant call — so the repo manifest requirement for auto-unlock is already satisfied.
  **[wisher decision 2026-07-05]** the headless/non-TTY guard is **retracted** — not needed
  because the key-scoped unlock works headless. no guard task remains.

**[answered] — settled now from source (see "validated in vision" below):**
- A2 (`keyrack.get` does not auto-unlock), A1 (`absent` false-positive risk is low),
  A5 (set-but-wrong → locked/blocked), A8 (fix covers push + pull), A9 (owner `ehmpath`),
  A10 (`absent` → ConstraintError exit 2), A4 (as-human can itself fail — keep its error clear).

### validated in vision (source-confirmed)

- **`keyrack.get` does NOT auto-unlock** — verified in `getKeyrackKeyGrant.js:14-17`: grant order
  is `os.daemon (unlock) → os.envvar (ci) → locked`, and *"never reads from vault — vault access
  is exclusively via unlock."* so R1 (auto-unlock) genuinely requires an **explicit unlock step**
  before the grant, not merely a config flag. premise validated.
- **the status map is real** — `inferKeyrackKeyStatusWhenNotGranted.js:56-92` confirms:
  `locked` = key set in vault but not unlocked → **auto-unlock**; `absent` = key never set (checks
  env slug + `env=all` fallback against a vault-agnostic inventory) → **guide to as-human**;
  `blocked` (`getKeyrackKeyGrant.js:100-109`) = granted but fails mechanism firewall →
  **malfunction**. the wish's three-way intent maps 1:1 onto these.
- **false-positive risk on `absent` is low** — since absent checks both the env-scoped slug and
  the `env=all` fallback, an env mismatch surfaces as `locked` (key present), not a false
  `absent`. residual risk only for legacy keys without an inventory entry.

## groundwork

### external research

none — no external APIs. all dependencies are internal (rhachet keyrack SDK) and the github
cli (already wired via `daoRadioTaskViaGhIssues`).

### internal research

verified against current code:

- **auth entry:** `src/domain.operations/radio/auth/getGithubTokenByAuthArg.ts:35` — defaults
  to `as-robot:via-keyrack(ehmpath)`; dispatches keyrack / shx / env / as-human modes.
- **keyrack read:** `src/domain.operations/radio/auth/getAuthFromKeyrack.ts:16-30` — calls
  `keyrack.get({ for: { key }, env, owner })`; throws `MalfunctionError` for **any**
  `status !== 'granted'` (absent, locked, blocked all collapsed together). **this is the core
  gap.** no unlock attempt, no status differentiation.
- **keyrack status contract:** `node_modules/rhachet/dist/domain.objects/keyrack/KeyrackGrantStatus.d.ts`
  → `'granted' | 'absent' | 'locked' | 'blocked'`; `KeyrackGrantAttempt.d.ts` shows each
  non-granted variant carries `slug`, `message`, and optional `fix`.
- **keyrack sdk:** `node_modules/rhachet/dist/contract/sdk.keyrack.d.ts` — exposes
  `keyrack.get/set/source`. no public `unlock`; internal `unlockKeyrackKeys` lives at
  `dist/domain.operations/keyrack/session/unlockKeyrackKeys.js`. unlock likely reached via CLI
  `rhx keyrack unlock` (pre-approved in permissions).
- **as-human flow:** `src/access/daos/daoRadioTask/daoRadioTaskViaGhIssues.ts:37-56` — null
  token means gh cli session auth (strips `GH_TOKEN`/`GITHUB_TOKEN`, lets gh cli login stand).
  so guiding to `--auth as-human` lands the user on a working, already-tested path.
- **test-env guard:** `getGithubTokenByAuthArg.ts:76-86` — `as-human` already fails loud in
  test env. our absent-guidance must respect the same guard.
- **vocab to reuse:** `as-robot` / `as-human`, `ConstraintError` / `MalfunctionError`,
  keyrack status names (`absent`/`locked`/`blocked`), `--auth as-...` flag grammar.

## what is awkward

- **auto-unlock is a write hidden in a read.** `getAuthFromKeyrack` reads a grant today; adding
  unlock makes it mutate host state. it may deserve its own operation
  (`unlockKeyrackForRadio` + `getAuthFromKeyrack`) rather than smuggling a side-effect into a
  getter — an architecture call for the blueprint.
- **the absent/locked line is subtle.** "key is not set" (absent) vs "vault is locked" (locked)
  are different, but a human may perceive both as "it didn't work." the copy must be precise so
  the as-human suggestion only appears when it's genuinely the right fallback.
- **unlock reachability is uncertain.** if the only unlock path is the `rhx keyrack unlock` CLI,
  then a domain.operation shelling out to a CLI is a slight layer smell; if `unlockKeyrackKeys`
  is importable, cleaner. must decide before blueprint.
- **CI/headless unlock** — **[wisher decision 2026-07-05]** resolved: the unlock is key-scoped
  (`--key EHMPATH_BEAVER_GITHUB_TOKEN`, not the whole env) and the beaver key grants
  non-interactively even headless, so it does not hang. no guard needed.
