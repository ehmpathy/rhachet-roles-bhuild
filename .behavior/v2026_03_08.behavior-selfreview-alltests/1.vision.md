# vision: self-review for all tests

## the outcome world

### before

a behaver completes execution, marks `--as passed`, and the guard checks code artifacts. but tests? those were run... probably. maybe some failed. maybe some were skipped. the human finds out later — or worse, doesn't — when something breaks in production.

the handoff is implicit: "i think i'm done, good luck."

### after

a behaver cannot pass execution without proof that all tests pass.

**the cardinal rule**: clones never offload work onto humans unless there is truly, fundamentally no other option. the default is to fix it yourself. handoff is the absolute last resort — not a convenience, not an escape hatch, but a genuine impossibility.

when tests fail, the behaver must exhaust every option: debug, research, try alternatives, ask for help from other clones. only when they hit a wall that is physically impossible to climb without human intervention (credentials only the human possesses, external service access only the human can grant, manual approval that requires human authority) — only then may they create `5.3.verification.handoff.v$N.to_human.md`. and even then, they must articulate exactly why no other path exists.

before the behaver marks complete, a playtest document is emitted — a step-by-step manual QA checklist the human can walk through to verify the deliverable feels right.

### the "aha" moment

the human reviews the PR and sees:
1. all tests green ✓ (5.3 verification passed)
2. no skips, no prior failures carried forward
3. a playtest they can follow to verify the feature themselves (5.5 playtest)

they run the playtest, approve, and think: "this isn't just code review — this is a complete handoff."

## user experience

### usecases

| who | goal | contract |
|-----|------|----------|
| behaver | prove tests pass before done | `rhx route.stone.set --stone 5.1.execution... --as passed` triggers test verification |
| behaver | hand off insurmountable blockers | emits `5.3.verification.handoff.v$N.to_human.md` only after all options exhausted |
| human | verify no gaps slipped through | reads playtest, walks through manual QA |
| human | unblock tests that need credentials/access | follows `5.3.verification.handoff.v$N.to_human.md`, runs tests, confirms green |

### timeline

```
behaver: completes implementation (5.1.execution)
behaver: runs `npm run test`
  ├─ all pass → proceed to 5.3
  └─ some fail
       ├─ fixable by behaver → fix and retry
       └─ absolutely cannot fix → question self, then emit 5.3.verification.handoff.v$N.to_human.md
behaver: rhx route.stone.set --stone 5.3.verification --as passed
  └─ guard verifies: no skips, all tests passed
behaver: emits 5.5.playtest.md with manual QA steps
behaver: rhx route.stone.set --stone 5.5.playtest --as passed
  └─ guard blocks until human approves
human: runs playtest, verifies deliverable, then approves
  └─ rhx route.stone.set --stone 5.5.playtest --as approved
```

### what the contract looks like

the guard checks:

```yaml
# 5.3.verification.guard
reviews:
  - slug: has-behavior-coverage
    say: |
      verify all behaviors from wish and vision are covered by explicit acceptance tests.

      walk through:
      - 0.wish.md — what did the wisher ask for?
      - 1.vision.md — what outcomes did we promise?

      for each behavior:
      - is there an explicit acceptance test that reproduces it?
      - can you point to the test file and describe what it verifies?
      - are there any behaviors left without test coverage?

      if any behavior cannot be tested due to human-only blockers:
      - create 5.3.verification.handoff.v$N.to_human.md with what the human needs to unblock
      - explain why you cannot write or run the test without them
      - end the handoff with: "after unblock, run: rhx route.stone.set --stone 5.3.verification --as rewound"

      every behavior in wish/vision must have an acceptance test.

  - slug: has-zero-test-skips
    say: |
      verify no tests are skipped:
      - no .skip() or .only()
      - no "skip if no credentials" patterns that silently pass
      - zero prior failures carried forward

      failures are better than skips. skips hide problems. failures expose them.

      all tests must run.

  - slug: has-all-tests-passed
    say: |
      verify all test suites pass via `npm run test`.

      if any fail, fix them. that is the default. not handoff.

      only if you hit a wall that is physically impossible to climb:
      - what did you try? list every approach you attempted.
      - why did each fail? be specific.
      - what makes this fundamentally impossible without human intervention?
      - is this truly a "human possesses the key" situation (credentials, access, authority)?
      - or did you quit too early?

      if and only if you can answer all of the above honestly, then:
      - create 5.3.verification.handoff.v$N.to_human.md with the answers above
      - the human should see you exhausted every option
      - end the handoff with: "after unblock, run: rhx route.stone.set --stone 5.3.verification --as rewound"
```

```yaml
# 5.5.playtest.stone
emit a playtest for the human to manually verify the deliverable:
- step-by-step verification instructions
- covers edge cases and happy paths
- human can follow without prior context

artifact: 5.5.playtest.md

# 5.5.playtest.guard
reviews:
  - slug: has-clear-instructions
    say: |
      is each step clear and unambiguous?
      - can a human follow without prior context?
      - are commands copy-pasteable?
      - are expected outcomes explicit?

  - slug: has-vision-coverage
    say: |
      does the playtest verify all requirements from the vision?
      - walk through 1.vision.md
      - for each goal, is there a step that verifies it?
      - are any requirements left untested?

  - slug: has-edgecase-coverage
    say: |
      does the playtest consider edge cases?
      - what could go wrong?
      - what inputs are unusual but valid?
      - what happens at boundaries?

judges:
  - mechanism: approved?
    # after self-reviews pass, human runs the playtest and approves
    # this is the final gate — manual QA by the human
```

## mental model

### how users describe it

"the behaver can't mark done until tests prove it works. they fix everything themselves — handoff is only for things that are literally impossible without me, like giving them my credentials. and there's always a checklist i can run through to verify it myself."

### analogies

| concept | analogy |
|---------|---------|
| test verification | pre-flight checklist before takeoff |
| 5.3.verification.handoff.v$N.to_human.md | emergency flare — only when truly stranded, not when tired |
| playtest | test drive before buying the car |
| zero skips | no crossed fingers allowed |

### terms

| user says | we say |
|-----------|--------|
| "did tests pass?" | "has-all-tests-passed" |
| "any work left for me?" | "5.3.verification.handoff.v$N.to_human.md" |
| "how do i verify this works?" | "5.5.playtest.md" |
| "no shortcuts" | "zero-test-skips" |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| tests must pass | yes | 5.3 guard blocks until green or handoff exists |
| no silent skips | yes | 5.3 guard checks for skip patterns, prior failures |
| human knows next steps | yes | 5.3.verification.handoff.v$N.to_human.md with actionable items |
| manual QA possible | yes | 5.5.playtest.md emitted with step-by-step |

### pros

- **no ambiguity**: tests either pass or there's a clear handoff
- **no silent failures**: skips and "no credentials" patterns are caught
- **manual verification**: playtest gives human confidence
- **separation of concerns**: 5.1 execution → 5.3 verification → 5.5 playtest

### cons

- **longer cycle**: verification adds time before done
- **credential friction**: tests that need secrets require human action
- **playtest maintenance**: playtests can drift from implementation

### pit of success

| edgecase | guard behavior |
|----------|----------------|
| test fails, behaver ignores | 5.3 guard blocks — test results checked |
| test skipped silently | 5.3 guard scans for skip patterns |
| credentials absent | behaver must prove they exhausted options, then handoff |
| no playtest | 5.5 stone blocks — playtest.md required |
| playtest exists but empty | 5.5 guard checks for meaningful content |

## open questions & assumptions

### assumptions

1. **tests are runnable**: `npm run test` exists and runs all test suites
2. **credential blockers are identifiable**: behaver can distinguish "needs human" from "bug in code"
3. **playtest is useful**: manual QA adds value beyond automated tests
4. **5.3 is the right placement**: verification as separate stone makes sense

### decisions

1. **prior failure tolerance**: never. a known-failed test is never acceptable. fix it or handoff.
2. **zero flakes**: flaky tests must be fixed, not tolerated. no exceptions.
3. **credential detection**: trust the behaver's handoff — they have context on why tests fail.
4. **playtest scope**: cover all behaviors mentioned in wish and vision. acceptance/integration/unit tests handle the rest.

### out of scope

1. **integration with CI**: separate concern, not part of this behavior.

## what is awkward?

### feels off

- **handoff path**: `5.3.verification.handoff.v$N.to_human.md` is clear but verbose — is there a better convention?

### fights mental model

- **blocked by external dependencies**: feels harsh when the blocker is "await api key"

note: the behaver declares the route — multiple roles (architects, mechanics, ergonomists, etc.) run through it.

### why this works

| principle | rationale |
|-----------|-----------|
| thoroughness over speed | slower but higher trust = more delegation = more parallel |
| handoffs over hacks | human eyes when needed, not worked around |
| zero tolerance over flexibility | high nines reliability — worth the friction |

---

*when this vision is fulfilled, no deliverable ships without proven tests, clear handoffs, and a path for manual verification. the human knows exactly what they're getting and what they need to do.*
