# vision: peer-review failhides

## the outcome world

### day-in-the-life

a behaver completes their blueprint or execution phase and runs `rhx route.stone.set --stone 3.3.blueprint.v1 --as passed`. before the guard approves, bhrain's reviewer automatically scans the work for failhides — try/catch patterns that silently swallow errors.

the review output lands in `.behavior/$route/.reviews/` alongside other review artifacts. if failhides are found, the behaver sees a clear report:

```
⚠️ failhide detected at src/logic/processOrder.ts:47

  try {
    await saveOrder(order);
  } catch (error) {
    console.log('failed');  // ← error swallowed, not rethrown
  }
```

### before/after contrast

| before | after |
|--------|-------|
| self-reviews rely on behaver's vigilance | automated peer review catches patterns behaver might miss |
| failhides slip through to production | failhides blocked at design-time (blueprint) and build-time (execution) |
| hours of debug chasing hidden errors | fail-fast always, errors surface immediately |

### the "aha" moment

the first time a behaver sees bhrain catch a failhide they didn't notice — that's when the value clicks. the robot reviewer doesn't get tired, doesn't rush, and applies the rule consistently every time.

## user experience

### usecases

1. **blueprint phase**: before approving architecture decisions, review for try/catch patterns that would hide errors in the proposed design
2. **execution phase**: before approving code changes, review actual code for failhide patterns

### contract inputs & outputs

**blueprint invocation** (inside 3.3.blueprint.v1.guard):
```yaml
reviews:
  peer:
    - slug: has-no-failhides
      via: npx rhachet run --repo bhrain --skill review \
        --rules ".agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.*.md" \
        --diffs since-main \
        --paths-with ".behavior/$route/3.3.blueprint.*.md" \
        --join intersect \
        --output ".behavior/$route/.reviews/$stone.peer-review.failhides.md" \
        --mode hard \
        --brain anthropic/claude/haiku
```

**execution invocation** (inside 5.1.execution.*.guard):
```yaml
reviews:
  peer:
    - slug: has-no-failhides
      via: npx rhachet run --repo bhrain --skill review \
        --rules ".agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.*.md" \
        --diffs since-main \
        --paths-with "src/**/*.ts" \
        --join intersect \
        --output ".behavior/$route/.reviews/$stone.peer-review.failhides.md" \
        --mode hard \
        --brain anthropic/claude/haiku
```

**output**: a markdown review report at `.behavior/$route/.reviews/{stone}.peer-review.failhides.md`

### timeline

1. behaver works on blueprint/execution
2. behaver calls `rhx route.stone.set --as passed`
3. guard triggers peer review via bhrain's review skill
4. review skill scans intersection of diffs + scoped paths against failhide rules
5. review output written to reviews directory
6. if violations found → guard blocks until fixed
7. if clean → guard passes

## evaluation

### how well does it solve the goals?

✅ catches failhides at blueprint phase (before code extant)
✅ catches failhides at execution phase (in actual code)
✅ automated — no human vigilance required
✅ consistent — same rules applied every time
✅ composable — uses extant bhrain review skill + mechanic rules

### pros

- **early detection**: catches design-level failhide tendencies in blueprints
- **automated**: no extra effort from behaver
- **leverages extant tools**: bhrain reviewer + mechanic rules already present
- **visible output**: review artifact shows exactly what was caught

### cons

- **false positives possible**: legitimate try/catch patterns might be flagged
- **rule quality dependent**: only as good as the failhide rules themselves
- **diff scope**: reviewing only changed files might miss context

### pit of success

- `--diffs since-main --paths-with <scope> --join intersect` reviews only changed files within phase scope
- `--mode hard` ensures thorough review with full file content
- output to `.reviews/` keeps artifacts organized and reviewable
- blocking guard prevents proceeding with violations

## open questions & assumptions

### assumptions

1. **diffs + paths with intersect**: `--diffs since-main --paths-with <scope> --join intersect` ensures only changed files within scope are reviewed
2. **hard mode is appropriate**: we assume full content review (vs soft path-only) is worth the cost
3. **peer section**: we assume this belongs in `reviews.peer:` (automated external) vs `reviews.self:` (behaver introspection)
4. **all pitofsuccess.errors rules**: we include all rules in the errors directory (error-hiding, helpful-error-wrap, fail-fast) via `rule.*.md` glob

## what is awkward?

### blueprint review scope

~~blueprints are markdown documents describing architecture — they don't have try/catch patterns in the traditional sense. reviewing a `.md` file for failhides feels forced.~~

**resolved**: blueprint review checks for error handling patterns described in prose (e.g., designs that would lead to failhides). execution review checks actual code. scopes are explicit via `--paths`.

### rule glob pattern

~~the rules are split across `pt1.md` and `pt2.md` files, plus `.min` variants. the glob needs to include the right ones while excluding condensed `.min` files.~~

**resolved**: using `rule.*.md` glob captures all rules in pitofsuccess.errors/ (error-forbid, error-wrap, fail-fast) while excluding `.min` condensed variants. clean and simple.

### peer vs self section

~~self-reviews are prompts for the behaver to introspect. peer reviews are external automated checks. adding bhrain review to `reviews.peer:` section makes semantic sense, but i'm not 100% sure that section is present in the guard schema yet.~~

**resolved**: verified guard schema supports `reviews.peer:` as a list of shell commands. no awkwardness here.
