# gap analysis: questioned-assumptions

## assumptions identified

### 1. bhrain review skill accepts our arguments

**assumption**: `--paths-with`, `--join intersect`, `--brain` are valid arguments

**evidence**: verified via `rhx review --help` — all arguments documented and supported

**verdict**: valid assumption, confirmed via docs

### 2. guard schema supports `reviews.peer:` section with `slug` and `via` fields

**assumption**: guards can have a `reviews.peer:` section with `slug` and `via` fields

**evidence verified**: checked `node_modules/rhachet-roles-bhrain/dist/domain.objects/Driver/RouteStoneGuard.d.ts`:
```typescript
export type RouteStoneGuardReviewPeer = string;

export interface RouteStoneGuardReviewsStructured {
    self: RouteStoneGuardReviewSelf[];
    peer: RouteStoneGuardReviewPeer[];  // array of strings!
}
```

**verdict**: INVALID ASSUMPTION — bhrain expects `reviews.peer:` to be an array of plain shell command strings, NOT objects with `slug` and `via` fields.

**correct format**:
```yaml
reviews:
  peer:
    - "npx rhachet run --repo bhrain --skill review ..."
```

**action required**: fix the blueprint to use correct schema format

### 3. `$route` and `$stone` variables are expanded by guard system

**assumption**: the guard system will expand these variables in peer review commands

**evidence verified**: checked `node_modules/rhachet-roles-bhrain/dist/domain.operations/route/guard/runStoneGuardReviews.js` lines 164-170:
```javascript
const substituteVars = (cmd, vars) => {
    return cmd
        .replace(/\$stone/g, vars.stone)
        .replace(/\$route/g, vars.route)
        .replace(/\$hash/g, vars.hash)
        .replace(/\$output/g, vars.output);
};
```

**verdict**: VALID — `$stone`, `$route`, `$hash`, and `$output` are all substituted in peer review commands

### 4. haiku brain has sufficient context for review

**assumption**: claude/haiku can handle the review within its context limits

**evidence**: the `--join intersect` flag scopes review to only files that are both in the diff AND match the paths-with glob. for typical PRs, this should be a small subset.

**verdict**: likely valid — haiku has ~200k context. even a large PR with 50 changed ts files should fit. verify in execution phase if reviews fail due to context limits

### 5. failhide rules apply to prose (blueprints)

**assumption**: review of blueprint.md for failhide patterns makes sense

**question**: blueprints are prose — do the rules apply to prose or just code?

**verdict**: rules describe patterns that can appear in either prose or code. prose might describe "catch errors and log them" which is a failhide design. valid assumption.
