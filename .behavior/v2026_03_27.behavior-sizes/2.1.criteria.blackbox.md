# blackbox criteria: behavior sizes

---

# usecase.1 = init with size

given a developer wants to start a behavior
  when they run `init.behavior --name fix-typo --size nano`
    then the behavior directory is created with nano-level stones
      sothat they can start work without need to delete unneeded files
    then the stones include: wish, vision, blueprint, roadmap, execution
      sothat they have the minimal ceremony for small changes
    then guards accompany each stone
      sothat review gates still apply

  when they run `init.behavior --name add-validation --size mini`
    then the behavior directory is created with mini-level stones
      sothat they get criteria and codepath research
    then the stones include nano stones plus criteria and code research
      sothat they can define "done" and understand the codepaths

  when they run `init.behavior --name new-endpoint --size medi`
    then the behavior directory is created with medi-level stones
      sothat they get reflection and playtest stones
    then the stones include mini stones plus reflection and playtest
      sothat they can premortem and validate with users

  when they run `init.behavior --name new-auth-system --size mega`
    then the behavior directory is created with mega-level stones
      sothat they get full domain research and distillation
    then the stones include medi stones plus domain research, factory research, and distillation
      sothat they can chart novel territory with proper ceremony

  when they run `init.behavior --name platform-migration --size giga`
    then the behavior directory is created with giga-level stones
      sothat they signal decomposition is expected
    then the stones match mega stones
      sothat the current feature set is preserved for future expansion

---

# usecase.2 = default size

given a developer wants to start a behavior
  when they run `init.behavior --name some-feature` without --size
    then medi size is used
      sothat most behaviors get appropriate ceremony by default
    then medi-level stones are created
      sothat they have reflection and playtest by default

---

# usecase.3 = size composes with guard

given a developer wants a small but critical fix
  when they run `init.behavior --name critical-fix --size mini --guard heavy`
    then mini-level stones are created
      sothat they get minimal ceremony
    then heavy guards are applied to those stones
      sothat critical fixes get thorough review
    then the two flags operate independently
      sothat users control both dimensions

---

# usecase.4 = help shows sizes

given a developer wants to understand size options
  when they run `init.behavior --help`
    then the output shows all five sizes with descriptions
      sothat users can pick the right size
    then each size description explains when to use it
      sothat users understand the tradeoffs
    then examples show realistic usage
      sothat users can copy-paste for their scenario

---

# usecase.5 = wrong size recovery

given a developer picked the wrong size
  when they started with nano but need mega-level work
    then they can manually add stones from templates
      sothat scope expansion does not require restart
  when they started with mega but work is simpler
    then they can manually remove unneeded stones
      sothat they reduce ceremony without restart

---

# usecase.6 = feedback template

given any behavior is created
  when any size is used (nano through giga)
    then the feedback template is included
      sothat humans can always provide input on behaviors
    then the template path is refs/template.[feedback].v1.[given].by_human.md
      sothat the name convention is consistent
