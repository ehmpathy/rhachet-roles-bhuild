# vision: behavior sizes

> `--size nano|mini|medi|mega|giga` for init.behavior

---

## the outcome world

### before

```bash
$ npx rhachet run --skill init.behavior --name fix-typo
# creates 40+ template files
# developer immediately deletes 35 of them
# "i just wanted to fix a typo, why do i need factory research?"
```

every behavior starts the same: full ceremony. a one-liner fix gets the same thoughtroute as a multi-month architecture overhaul. developers spend the first 10 minutes of every behavior deleting stones they don't need.

### after

```bash
$ npx rhachet run --skill init.behavior --name fix-typo --size nano
# creates 5 files: wish, vision, blueprint, execution
# ready to go

$ npx rhachet run --skill init.behavior --name new-auth-system --size mega
# creates full suite: research, factory, reflection, playtest
# appropriate ceremony for the scope
```

the thoughtroute matches the work. small changes get small structure. big changes get big structure.

### the "aha" moment

> "oh, the size flag is like choosing the right vehicle for the trip. nano is a bicycle, mega is a cargo ship. you don't take a cargo ship to the grocery store."

## user experience

### use cases

| size | when to use | examples |
|------|-------------|----------|
| nano | small isolated changes, no cascades | fix typo, bump version, add simple test, tweak config |
| mini | small focused changes, single-file features | add validation rule, new utility function, simple refactor |
| medi | work within known domains | novel endpoint, integration with extant service, reflection |
| mega | novel domain work | novel subsystem, novel domain research + distillation |
| giga | novel domain work, decomposition expected | new product area, platform migration, major version release |

**key distinction:** medi = operate within known territory. mega = chart novel territory (research + distill + guard).

### contract

```bash
# input
npx rhachet run --skill init.behavior --name <name> [--size <size>] [--guard <level>]
```

## evaluation

### how well does it solve the goals?

**goal: stop deleting stones**
- ✅ users pick appropriate size upfront
- ✅ no more deleting 30 files after init

**goal: right ceremony for scope**
- ✅ nano: minimal friction for tiny changes
- ✅ mega: full ceremony for significant work
- ✅ medi default matches most work

### pros

1. **reduced friction** - small changes don't feel over-engineered
2. **clear expectations** - size signals scope to reviewers
3. **faster onboarding** - new users start with nano, grow into mega
4. **better defaults** - medi is sensible middle ground

### cons

1. **new decision** - users must choose size (mitigation: sensible default)
2. **size mismatch risk** - user picks nano for mega-scope work (mitigation: can always add stones later)
3. **documentation needed** - users need to learn what each size includes

### pit-of-success patterns

| edgecase | pit-of-success |
|----------|----------------|
| user picks wrong size | can always add missing stones manually via templates |
| user unsure what size | medi is sensible default |
| mega work disguised as nano | guards still enforce review; scope reveals itself |
| giga scope creep | user can always delete stones manually |

## open questions & assumptions

### assumptions made

1. **medi is correct default** - assumes most behaviors are medium-scope
2. **stone groupings are correct** - assumes current categorization matches developer mental model
3. **five sizes is right** - not four, not six

### questions for wisher

1. **[answered] is medi the right default?** yes - medi is the correct default.
2. **[answered] should giga warn or block?** neither - giga is just "all stones", reserved for future expansion
3. **[answered] should --guard compose with --size?** yes - orthogonal axes. size = which stones, guard = gate weight. e.g., `--size mini --guard heavy` for a small but critical fix.
4. **[wisher] should nano include criteria?** wisher said "vision, blueprint, execution" but criteria defines "done" - is it needed?
5. **[answered] should mega get heavy guards throughout?** n/a - guards are orthogonal to size. --size controls which stones, --guard controls gate weight.

## what is awkward?

### naming: nano vs micro

"nano" sounds technical. alternatives:
- tiny / small / medium / large / huge
- xs / s / m / l / xl
- quick / light / standard / full / epic

current choice (nano/mini/medi/mega/giga) uses consistent latin-rooted prefixes, which is elegant but not immediately intuitive.

### stone inheritance is implicit

the relationship between sizes isn't explicitly "mini = nano + more". users might expect `--size mega --exclude research.factory` but that's not supported.

mitigation: keep it simple. pick a size. if you need customization, add/remove stones manually.

### medi is not a real word

"medi" is unusual. alternatives:
- "medium" (verbose)
- "mid" (too casual)
- "norm" (unclear)
- "base" (misleading)

current choice: "medi" is short, consistent with pattern, and memorable once learned.

---

## summary

`--size` brings right-sized ceremony to behavior-driven development. nano for sticky-note work, mega for design-doc work. medi as sensible default. reduces stone deletion, matches mental models, preserves flexibility.
