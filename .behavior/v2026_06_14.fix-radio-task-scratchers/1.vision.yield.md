# vision: fix radio task scratchers

## the outcome world

### before

```sh
# run radio.task.push --help
$ rhx radio.task.push --help
✋ ConstraintError: --via must be one of: "gh.issues", "os.fileops"

# dispatch task without --into
$ rhx radio.task.push --via gh.issues --title "fix bug"
🎙️ task created: fix bug
   ├─ repo: ehmpathy/rhachet-roles-bhuild   # silently used current repo
   └─ via: gh.issues

# radio.uses shows turtle mascot (wrong identity)
$ rhx radio.uses get
🐢 cowabunga!

🐚 radio.uses
   └─ local: allowed
```

problems:
- `--help` fails with validation error instead of showing help
- `--into` defaults silently to current repo — pit of failure
- dispatcher uses turtle mascot 🐢 (ehmpathy's mascot) instead of beaver 🦫 (bhuild's mascot)

### after

```sh
# run radio.task.push --help
$ rhx radio.task.push --help
🦫 lets check the meter...

🎙️ radio.task.push --help
   ├─ usage
   │  ├─ radio.task.push --via gh.issues --into owner/repo --title "..." --description "..."
   │  ├─ radio.task.push --via gh.issues --into @this --title "..."
   │  └─ echo "details" | radio.task.push --via gh.issues --into @this --title "..." --description @stdin
   │
   └─ options
      ├─ --via         channel: gh.issues or os.fileops (required)
      ├─ --into        target: @this (current repo) or owner/repo (required)
      ├─ --title       task title (required for new tasks)
      ├─ --description task description (supports @stdin)
      ├─ --exid        external id for updates
      ├─ --status      status: QUEUED, CLAIMED, DELIVERED
      └─ --idem        idempotency: findsert or upsert

# dispatch task requires explicit --into
$ rhx radio.task.push --via gh.issues --title "fix bug"
✋ ConstraintError: --into is required

# explicit --into @this for current repo
$ rhx radio.task.push --via gh.issues --into @this --title "fix bug"
🦫 back in the river!

🎙️ radio.task.push
   ├─ title: fix bug
   ├─ repo: ehmpathy/rhachet-roles-bhuild
   └─ via: gh.issues
   # note: github issue created with title "🎙️ fix bug"

# or explicit owner/repo
$ rhx radio.task.push --via gh.issues --into ehmpathy/svc-quotes --title "fix bug"
🦫 back in the river!

🎙️ radio.task.push
   ├─ title: fix bug
   ├─ repo: ehmpathy/svc-quotes
   └─ via: gh.issues
   # note: github issue created with title "🎙️ fix bug"

# radio.uses shows beaver mascot (correct identity)
$ rhx radio.uses get
🦫 lets check the meter...

🎙️ radio.uses
   └─ local: allowed
```

### the "aha" moment

when you accidentally dispatch a task to the wrong repo, you've lost work or created confusion. requiring explicit `--into` makes the target visible and intentional — the moment you see `--into @this` or `--into owner/repo`, you know exactly where the task is going.

## user experience

### usecases

| goal | contract | example |
|------|----------|---------|
| dispatch task to current repo | `--into @this` | `rhx radio.task.push --via gh.issues --into @this --title "..."` |
| dispatch task to another repo | `--into owner/repo` | `rhx radio.task.push --via gh.issues --into ehmpathy/svc-quotes --title "..."` |
| see usage help | `--help` | `rhx radio.task.push --help` |
| check permissions | (unchanged) | `rhx radio.uses get` |

### timelines

1. **first use**: user runs `--help` to see usage
2. **dispatch**: user explicitly targets repo via `--into`
3. **verify**: user sees beaver mascot confirming dispatcher identity

## mental model

### how users would describe this

> "radio.task.push is like posting to a work queue — you pick the queue (`--via`) and explicitly say where it goes (`--into`). no silent defaults."

### analogies

- `--into @this` is like `--repo .` in git commands — explicit current directory
- `--into owner/repo` is like `--remote origin/main` — explicit target

### terms

| user term | our term |
|-----------|----------|
| "send to" | `--into` |
| "current repo" | `@this` |
| "channel" | `--via` |

## evaluation

### pros

- **pit of success**: explicit `--into` prevents accidental dispatch to wrong repo
- **discoverability**: `--help` shows usage instead of error
- **identity**: beaver mascot reinforces dispatcher role

### cons

- **breaking change**: extant scripts without `--into` will fail
- **verbosity**: `--into @this` is longer than implicit default

### edgecases

| edgecase | response |
|----------|----------|
| not in git repo, no `--into` | `✋ ConstraintError: --into is required` |
| in git repo, no `--into` | `✋ ConstraintError: --into is required` (explicit > implicit) |
| `--into @this` outside git repo | `✋ ConstraintError: @this requires a git repository` |
| `--into invalid` | `✋ ConstraintError: --into must be @this or owner/repo format` |

## open questions & assumptions

### assumptions

- `@this` sentinel value is intuitive for "current repo"
- beaver mascot 🦫 is the correct identity for dispatcher role
- incompatible change to require `--into` is acceptable

### scope note

`radio.task.pull` also uses the same output functions (`radio.uses.output.sh`). if we change the mascot for push, pull should also get the beaver mascot for consistency. this expands scope slightly but ensures uniform dispatcher identity.

### open questions

1. **[answered]** is `@this` the right sentinel, or should it be `@cwd`, `.`, or else?
   - `@this` follows the `@stdin` sentinel pattern already used in the codebase
   - `.` could be confused with filesystem paths
   - `@cwd` is more verbose and less intuitive
   - **answer**: `@this` is correct

2. **[answered]** should we provide a migration period with deprecation warn, or hard break immediately?
   - **answer**: hard break — require `--into` immediately, no deprecation period

3. **[answered]** what beaver phrases should replace the turtle phrases in radio.uses?
   - **answer**:
     - allow success: "back in the river!"
     - block success: "dammed up tight"
     - get/check: "lets check the meter..." (keep generic)
     - error: "hit a snag..."

4. **[answered]** should we define a shared `rule.im_an.bhuild_beaver.md` persona for all bhuild roles, or should dispatcher reuse behaver's phrases?
   - **answer**: dispatcher should reuse behaver's beaver phrases (no new persona needed)

## what is awkward?

- `--into @this` feels slightly verbose compared to implicit default, but explicit is safer
- `🦫` (beaver) + `🎙️` (microphone) combined might be visually busy — consider just `🦫` or just `🎙️`
- **scope expansion**: beaver persona (`rule.im_an.bhuild_beaver.md`) does not exist yet — this fix requires that persona to be defined first, or we reuse behaver's phrases
- behaver already uses some beaver phrases (`oh, behave!`, `wassup?`) — should dispatcher share these or have its own?
