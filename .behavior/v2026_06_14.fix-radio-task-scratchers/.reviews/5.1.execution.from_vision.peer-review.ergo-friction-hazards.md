# blocker.1: Unhandled errors produce unclear error messages

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/contract/cli/radioTaskPull.ts
- src/contract/cli/radioTaskPush.ts

The main CLI entrypoints perform critical user operations (context assembly, task pull/push) without any try/catch or error handling. Failures in these paths (auth errors, network problems, domain malfunctions, etc.) will cause uncaught promise rejections or exceptions. This results in raw stack traces or default node errors instead of clear, actionable messages telling the user what went wrong AND how to fix it (via outputRadioResult with hints). The rule requires every error path to be addressed for smooth UX; unhandled paths are friction hazards.

**snippet**:
```ts
  // assemble context for channel (may involve auth)
  const context = await getOneRadioContextFromCliArgs(
    {
      via: named.via,
      repo,
      auth: named.auth ?? null,
    },
    { env: process.env, shx },
  );

  // handle list mode
  if (mode.kind === 'list') {
    const result = await radioTaskPullAll(
      {
        via: named.via,
        repo,
        ...(mode.filter !== null && { filter: mode.filter }),
        ...(mode.limit !== null && { limit: mode.limit }),
      },
      context,
    );

    outputRadioResult({
      message: asTaskListOutput({ tasks: result.tasks, via: named.via }),
    });
    return;
  }

  // handle single mode
  const result = await radioTaskPullOne(
    {
      via: named.via,
      repo,
      ref: mode.ref,
    },
    context,
  );

  outputRadioResult({
    message: asTaskDetailOutput({
      task: result.task,
      via: named.via,
      outcome: null,
      cached: result.cached,
    }),
  });
```

# blocker.2: Inconsistent output mechanism for help vs results

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/contract/cli/radioTaskPull.ts
- src/contract/cli/radioTaskPush.ts

Help is output directly with console.log(HELP_TEXT) while success and error results use outputRadioResult (which supports isError and hints). This violates output consistency: stdout/stderr and formatting may differ across invocations (help always stdout, errors potentially stderr or formatted differently). The rule requires consistent stdout/stderr and stable response formats across all invocations to avoid breaking downstream callers or causing user confusion.

**snippet**:
```ts
  if (named.help || named.h) {
    console.log(HELP_TEXT);
    process.exit(0);
  }
```

# blocker.3: Empty ask in error hint produces unclear error messages

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/contract/cli/radioTaskPull.ts
- src/contract/cli/radioTaskPush.ts

When reporting required arg errors for --from/--into, the error hint is constructed with ask: '' (empty). This leads to incomplete or awkward error presentation for the user (unlike the --via error which populates ask). Errors must be clear and consistent; partial hints degrade UX and trust as per the friction hazards rule.

**snippet**:
```ts
    outputRadioResult({
      message: repoResult.error.message,
      isError: true,
      hint: { ask: '', command: repoResult.error.hint },
    });
    process.exit(2);
```
