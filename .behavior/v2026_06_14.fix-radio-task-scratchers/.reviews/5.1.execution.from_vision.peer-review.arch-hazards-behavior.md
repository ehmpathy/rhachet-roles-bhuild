# blocker.1: Implicit external state in @this repo derivation causes non-determinism and hidden side effects

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/radio/cli/getOneRadioTaskRepoFromCliArg.ts:37
- src/contract/cli/radioTaskPull.ts:48
- src/contract/cli/radioTaskPush.ts:58

getOneRadioTaskRepoFromCliArg depends on implicit external state (current cwd and git repo) when input.arg === '@this'. It calls getRepoFromGitContext() with no parameters, so the function's output for the same input ('@this') can differ based on process state (non-determinism). This is order-dependent (assumes correct cwd/git context was 'set' by caller environment) and a hidden side effect (reads global state not in signature or inputs). The docstring claims to prefer explicit over implicit but the implementation does the opposite. Per the rule, non-determinism must be explicit and documented; undocumented hazards like this are blockers as they lead to unpredictable behavior under different runtime conditions (e.g. different cwd, concurrent processes, or git state changes).

**snippet**:
