# vision: fix peer review brain size

## the outcome world

### before

- peer reviews use `rhachet enroll claude` without brain flag
- defaults to opus (larger, slower model)
- each review takes ~3 minutes
- 5 sequential reviews = ~15 minutes total
- see `.behavior/v2026_04_26.fix-peer-speed` for context

### after

- peer reviews explicitly use `--model sonnet`
- sonnet is faster while still capable for code review
- each review takes ~30-60 seconds
- 5 sequential reviews = ~2.5-5 minutes total
- guard passage is no longer painfully slow

### the "aha" moment

"sonnet is fast enough for peer review but doesn't compromise quality"

## user experience

### usecases

| goal | before | after |
|------|--------|-------|
| pass execution stone | ~15min for peer reviews | ~2-5min for peer reviews |
| iterate on code | momentum killed by wait | momentum preserved |
| run CI | peer reviews may timeout | peer reviews complete reliably |

### contract: guard peer review commands

**before:**
```yaml
peer:
  - ./node_modules/.bin/rhachet enroll claude --roles behaver,architect,mechanic -p "review..."
```

**after:**
```yaml
peer:
  - ./node_modules/.bin/rhachet enroll claude --model sonnet --roles behaver,architect,mechanic -p "review..."
```

### timeline

1. mechanic runs `--as passed` on execution stone
2. guard triggers peer reviews with sonnet brain
3. each review completes in ~30-60s instead of ~3min
4. total guard passage time drops from ~15min to ~2-5min

### pros

### cons

- ⚠️ sonnet may miss subtle issues that opus would catch

none significant — the `--model` flag is a native Claude CLI option (rhachet forwards all flags)

## what is awkward?

### sonnet vs opus tradeoff

sonnet is faster but potentially less thorough. however:
- peer reviews are a secondary quality gate (self-reviews are primary)
- speed enables more frequent runs, which may catch more issues overall
- the tradeoff is acceptable for this use case
