# 1.vision — fulfill radio task #237 (`radio.task.push --help`)

## tl;dr

radio task #237 reports that `radio.task.push --help` "fails with a validation error
instead of showing help." **that bug no longer reproduces** — it was already fixed by
commit `9136073` (#242, "fix help flag"). i verified both `--help` and `-h` render the
usage + options tree correctly today.

what #237 is *missing* is a **guard**: `radio.task.pull` has a contract-boundary
integration test that snapshots its `--help` output (`radioTaskPull.integration.test.ts`
case6), but `radio.task.push` has **no equivalent test**. the help behavior works but is
unprotected — a future refactor could silently re-break it.

**the vision:** fulfill #237 by adding a `radioTaskPush.integration.test.ts` that locks in
the `--help` / `-h` contract, mirroring pull's case6. this brings push to parity with pull
and institutionalizes the #242 fix so it cannot regress.

> confirmed direction with the wisher: "add regression test (recommended)."

## user experience

### who is the user?

the primary user of this deliverable is the **maintainer / ci pipeline**, not an end-user
of the CLI. the test is the product. the secondary beneficiary is the **human who runs
`radio.task.push --help`** — their experience stays good, permanently.

### usecases fulfilled

- **maintainer**: "prove `push --help` output stays correct across refactors" → the test
  asserts + snapshots the contract.
- **reviewer**: "see the exact help text a human gets, in the diff" → the snapshot.
- **#237 filer**: "the help works and won't break again" → the guard.

### contract inputs & outputs (the test's own contract)

the test invokes the built CLI entry `cliRadioTaskPush` directly at the contract boundary,
injecting `process.argv`, mirroring pull's `runHelp()` harness:

```ts
// input: argv with the help flag
process.argv = ['node', '--help'];   // and a sibling case for ['node', '-h']

// output: captured stdout (console.log), asserted + snapshotted
expect(stdout).toContain('radio.task.push --help');
expect(stdout).toContain('--via');
expect(stdout).toContain('--auth');
expect(stdout).toMatchSnapshot();
```

`--help` calls `process.exit(0)`, so the harness stubs `process.exit` to throw a sentinel
and stubs `console.log` to capture lines — identical to pull's proven pattern
(`radioTaskPull.integration.test.ts:65-87`).

### timeline

2. run `rhx git.repo.test --what integration --scope radioTaskPush` → snapshot writes on
   first run
3. reviewer vibechecks the snapshot
4. merge → #237 closes

## evaluation

### how well it solves the goal

- #237's literal ask ("push --help should work") = **already true**; we verified it.
- #237's *durable* fulfillment ("...and stay working") = **the test delivers it.**

### pros

- tiny, low-risk: adds a test only — no production code change.
- mirrors an established, proven pattern (pull case6) → high confidence, low novelty.
- restores pull/push symmetry the codebase clearly intends.
- snapshot gives reviewers a caller-faced vibecheck.

### cons / tradeoffs

- lacks a user-felt "fix" today (the fix already shipped) — the value is purely preventive.
  accepted, because the wisher chose this path.
- snapshot couples the test to the exact `HELP_TEXT` string; intentional edits to help text
  require a `--resnap`. this is the desired behavior (changes surface in review), matching
  pull.

### edgecases & pit-of-success

| edgecase | handling |
|----------|----------|
| `-h` short flag | cover as its own case alongside `--help` (both route to the same `HELP_TEXT`) |
| help must not touch network/gh | `--help` returns before any auth/i/o — no credentials, deterministic, ci-safe |
| `process.exit(0)` ends the process | stub `exit` to throw a sentinel, caught via `getError` — proven in pull's `runHelp` |
| snapshot nondeterminism | help text is a static constant — fully deterministic, unlike a live gh pull |

## what is awkward?

- **"fixing" a bug that is not broken.** the honest framing is that #237 is stale on its
  literal wording; we reinterpret it as "guard the fix." this is a slight stretch of the
  task's words, but it's the highest-value reading and the wisher endorsed it.
- **the test is the deliverable, with no prod code change.** a reviewer skimming for a
  "fix" will find only a new test file. the vision + PR description must make the "already
  fixed by #242, now guarded" story explicit so it does not read as a no-op.
- **snapshot vs assertions overlap.** we both `toContain(...)` and `toMatchSnapshot()`;
  that's deliberate (assertions = functional guard, snapshot = visual vibecheck) and
  matches pull, but it can feel redundant at first glance.
