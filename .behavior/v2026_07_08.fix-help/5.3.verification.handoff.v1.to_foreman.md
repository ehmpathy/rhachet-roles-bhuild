# 5.3.verification — handoff v1 to foreman

## the wall

two verification peer reviewers (r2 `ergo-contract-snapshots`, r4
`ergo-acceptance-journey-coverage`) block the stone with the same rule:
`rule.require.contract-snapshot-exhaustiveness`. they require the new
`radioTaskPush.integration.test.ts` to snapshot **all** contract variants — positive
(success), negative (error), and edge — not only the `--help` / `-h` help text.

this conflicts with a **human-approved** decision: the vision (`1.vision.yield.md`, approved
via `--as approved`) scoped #237 to `--help`/`-h` only and **explicitly deferred** a push
success case. a prior execution-stone peer reviewer independently confirmed push's cli
error/success paths were "correctly out of scope for #237."

so two human-enrolled artifacts are in direct tension: the approved help-only vision vs the
verification gate's exhaustiveness rule.

## what i tried / verified

- **negative paths are cleanly achievable** (no creds, deterministic): absent `--via` →
  `BadRequestError: --via is required`; absent `--into` → `--into is required`. both fire
  before any i/o or the permission gate. i can add these snapshots trivially.
- **the positive path is genuinely obstructed.** unlike pull (no permission gate, so its
  case7 seeds a local task and reads it back cleanly), `cliRadioTaskPush` runs an
  unconditional permission gate (`getOneRadioUsagePermissionDecision`,
  radioTaskPush.ts:183-196) that reads the **real** `process.cwd()/.meter/radio.uses.jsonc`
  and the **real** home dir. verified:
  - the repo has no `.meter/` → the gate blocks by default (`safe default (unset)`).
  - the cli passes `sourceCwd: process.cwd()` and default (real) homeDir — there is **no
    clean injection point** at the cli boundary.
  - to make a push permission-allowed i would have to either (a) write radio.uses config
    into the real repo root (side-effect pollution, cleanup-fragile, and a global block in
    someone's home would still override it → non-deterministic across machines/ci), or
    (b) refactor the production cli to accept an injectable homeDir/cwd for the gate.
  - both violate test hygiene (which the same gate's r8 `mech-test-scope-purity` enforces)
    or materially exceed #237's approved scope.
- push's **success behavior is already deterministically covered** at the domain tier:
  `radioTaskPush.via-os-fileops.integration.test.ts` (create/update/idempotency/bootstrap +
  "title required"/"description required" errors). the gap is only the *cli-boundary
  snapshot*, not the behavior.

## why this is above my authority

- an expansion of a **human-approved** vision's scope is a scope decision i should not make
  unilaterally.
- a deterministic positive-path cli snapshot needs a **production refactor** (inject
  homeDir/cwd into the permission gate) — a design change well beyond a #237 regression
  guard.

## the decision i need

pick one:

- **(A) expand #237** to full exhaustive push cli coverage — accept the positive-path test
  design work, likely a small prod refactor to make the permission gate injectable. larger
  effort than the approved vision.
- **(B) add negative-path snapshots only** (absent `--via`/`--into`, which i can do now),
  and accept that the positive-path cli snapshot is deferred because push's success is
  covered at the domain tier + the cli positive path is permission-gate-entangled. requires
  a relaxation of the two reviewers' threshold or an accepted exception.
- **(C) hold help-only** per the approved vision, rely on domain-tier success coverage, and
  record the exhaustiveness reviewers as an accepted exception for this scoped deliverable.

my recommendation: **(B)** — it adds real, clean negative-path regression protection now,
without a prod refactor or a flaky permission-gate test, and leans on the domain-tier
success coverage that already exists.
