# vision: yield.md artifact naming

## the outcome world

### day-in-the-life

a behaver working on a feature opens their `.behavior/` directory:

```
.behavior/v2026_04_07.my-feature/
├── 0.wish.md
├── 1.vision.stone
├── 1.vision.yield.md              ← immediately clear this is vision's output
├── 3.3.1.blueprint.product.v1.stone
├── 3.3.1.blueprint.product.v1.yield.md
└── .route/
```

they instantly see:
- stone files (instructions)
- yield files (outputs)
- the pairing is obvious from the name prefix

### before/after contrast

| before | after |
|--------|-------|
| `3.3.1.blueprint.product.v1.i1.md` | `3.3.1.blueprint.product.v1.yield.md` |
| "what does i1 mean?" | "ah, the yield of this stone" |
| sorts after `.md` files unpredictably | sorts immediately after its stone |
| version + iteration naming unclear | clear semantic: this is what the stone produced |

### the "aha" moment

"oh — every stone has a yield. the yield is the output artifact. `1.vision.stone` yields `1.vision.yield.md`. simple."

## user experience

### usecases

1. **navigating behavior directories** — user browses files, immediately knows which are inputs (stones) vs outputs (yields)
2. **referencing prior outputs** — user writes a stone that references `$BEHAVIOR_DIR_REL/1.vision.yield.md` — the name is self-documenting
3. **reviewing work** — user sees `*.yield.md` files and knows exactly what to review

### contract inputs & outputs

**current (before):**
```
emit into $BEHAVIOR_DIR_REL/3.3.1.blueprint.product.v1.i1.md

reference
- $BEHAVIOR_DIR_REL/3.2.distill.domain.*.v1.i1.md (if declared)
```

**proposed (after):**
```
emit into $BEHAVIOR_DIR_REL/3.3.1.blueprint.product.v1.yield.md

reference
- $BEHAVIOR_DIR_REL/3.2.distill.domain.*.yield.md (if declared)
```

### what it looks like to leverage them

```
# in a stone file
reference
- $BEHAVIOR_DIR_REL/1.vision.yield.md
- $BEHAVIOR_DIR_REL/3.1.3.research.internal.product.code.prod.*.yield.md

emit into $BEHAVIOR_DIR_REL/3.3.1.blueprint.product.v1.yield.md
```

the naming is:
- self-documenting: "this is the yield"
- consistent: all outputs follow the same pattern
- grep-able: `*.yield.md` finds all outputs

### timelines

1. behaver runs stone
2. stone emits to `{stone-name}.yield.md`
3. behaver (or guard) references `{stone-name}.yield.md`
4. next stone consumes the yield

## mental model

### how users describe to a friend

> "each stone produces a yield. the stone is the instruction, the yield is the output. they share the same name prefix so you always know what goes together."

### analogies

| analogy | stone | yield |
|---------|-------|-------|
| farming | seed packet | harvest |
| recipe | instructions | dish |
| function | definition | return value |

### terms: user vs us

| user might say | we say |
|----------------|--------|
| "output" | "yield" |
| "result" | "yield" |
| "artifact" | "yield" |
| "the thing the stone makes" | "yield" |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| better alpha-ordering | ✅ | `yield.md` sorts after `.stone` |
| clearer naming | ✅ | "yield" is unambiguous |
| self-documenting | ✅ | name explains purpose |
| consistent pattern | ✅ | all outputs follow same convention |

### pros

- **clarity**: "yield" is intuitive — it's what the stone produces
- **sorting**: files alpha-sort into logical groups (stone then yield)
- **grep-ability**: `*.yield.md` finds all outputs
- **semantic**: the name carries meaning vs opaque `v1.i1`

### cons

- **new pattern to learn**: current users need to learn new pattern (minimal)

note: no migration needed — we only update the stone templates to prescribe the new convention. extant behavior directories keep their current names.

### edgecases and pit of success

| edgecase | how we handle |
|----------|---------------|
| multiple iterations | keep `v1`, `v2` prefix on stone; yield follows: `3.3.1.blueprint.product.v2.yield.md` |
| wildcard references | `*.yield.md` still works: `$BEHAVIOR_DIR_REL/3.1.3.research.*.yield.md` |
| no yield (guard-only stones) | no yield file emitted — pattern doesn't force one |

## open questions & assumptions

### assumptions

1. **yield is always `.md`** — we assume yields are markdown files
2. **one yield per stone** — we assume each stone produces at most one primary yield
3. **alpha-sorting matters** — we assume users navigate via file browsers that alpha-sort

### answered [answered]

- is `yield` the right word? → yes, wisher chose it explicitly in the wish
- should we support non-md yields? → not in scope for now; keep it simple with `.md` only
- can a stone produce multiple yields? → not in scope for now; one yield per stone
- do we need migration? → no; only update templates, extant directories keep current names

## what is awkward?

### what feels off

- **`yield` as a suffix vs prefix** — we chose suffix (`1.vision.yield.md`) to keep alpha-sorting; prefix (`yield.1.vision.md`) would group all yields together but separate from their stones

### where design fights mental model

- users might expect `{stone}.output.md` or `{stone}.out.md` — "yield" is slightly unusual but more evocative

### uncomfortable tradeoffs

- **consistency vs flexibility** — forcing `yield.md` as the suffix means we can't easily support `yield.json` without changing the pattern to `{stone}.yield/{filename}` or similar

---

overall: the `yield.md` pattern is clearer, sorts better, and names what the artifact truly is — the yield of the stone.
