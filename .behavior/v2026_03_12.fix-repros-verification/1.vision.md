# vision: fix repros-verification connection

## the outcome world

### what does a day-in-the-life look like with this in place?

a mechanic drives through the behavior route. at the repros stone, they don't just list "what will be tested" — they sketch actual journey tests with full BDD structure: `given('[caseN]')` → `when('[tN]')` → `then()` blocks that step through each phase of the user experience. they consider: "what would the terminal, browser, or renderer show at each step? what would the caller see?"

at the verification stone, the mechanic doesn't start from scratch. they look back at the repros artifact and ask: "did i actually implement each reproduction i promised? do my tests have snapshot assertions that capture the input/output so reviewers can see exactly what the caller experiences?"

### what's the before/after contrast?

**before:**
- repros stone: vague table of "experiences" without concrete test sketches
- verification stone: generic checklist that doesn't reference repros
- result: mechanic finishes the route, pr gets reviewed, foreman discovers:
  - no journey-style acceptance tests
  - no snapshots that show what the caller-visible output looks like
  - tests exist but don't capture the *experience*

**after:**
- repros stone: includes concrete journey test sketches with full BDD `given/when/then` structure
- repros stone: explicitly requires snapshot consideration for outputs
- verification stone: checklist references repros artifact directly
- verification guard: self-review asks "did you implement the journey tests from repros?"
- verification guard: self-review asks "do your snapshots capture the input/output?"
- result: mechanic can't pass verification without proof they delivered on repros promises

### what's the "aha" moment where the value clicks?

the mechanic is at verification, about to answer the self-review: "did you implement each journey test sketched in repros?" they realize they forgot to write the `when([t2]) second action` test. they go back and add it. the snapshot captures the terminal output. now when the foreman reviews, they can literally *see* what the user would see.

## user experience

### what usecases do folks fulfill?

1. **mechanic at repros stone**: sketches journey tests that mirror real user workflows
2. **mechanic at verification stone**: validates that sketched tests were implemented with proper snapshots
3. **foreman in pr review**: reads snapshot files and sees exactly what the caller experiences

### what contract inputs & outputs do they leverage?

**repros stone outputs:**
- journey test sketches with full BDD structure: `given('[caseN]')` → `when('[tN]')` → `then()`
- input/output pairs that show what caller provides and what they receive
- snapshot coverage plan (which outputs need `.snap` files)
- experience reproduction table that links vision → test → snapshot
- critical path identification (happy paths that must be frictionless)
- ergonomics review (does the input/output feel right?)

**repros guard self-reviews:**
- `has-critical-paths-identified`: are the happy paths clearly marked as critical?
- `has-ergonomics-reviewed`: does the input/output feel natural? any friction?
- `has-play-test-convention`: are journey tests named `.play.test.ts` (or `.play.integration.test.ts` / `.play.acceptance.test.ts` if repo requires)?

**verification stone inputs:**
- references `3.2.distill.repros.*.md` artifact
- checklist item: "for each journey in repros, point to test file"
- checklist item: "for each test, verify snapshot assertions exist"
- checklist item: "for each critical path, verify it is frictionless"

**verification guard self-reviews:**
- `has-journey-tests-from-repros`: did you implement each journey sketched?
- `has-snapshot-coverage`: do snapshots capture input/output for caller visibility?
- `has-critical-paths-frictionless`: are the critical paths frictionless in practice?
- `has-ergonomics-validated`: does the actual input/output match what felt right at repros?
- `has-play-test-convention`: are journey test files named `.play.test.ts` (or fallback convention)?

## mental model

### how would users describe this to a friend?

"the repros stone is where i plan my journey tests. the verification stone is where i prove i did what i planned. if i can't point back to repros and say 'here's the test for that journey, and here's what the output looks like in the snapshot,' i can't pass."

### what analogies or metaphors fit?

- **repros = test design document** — sketches the journey before implementation
- **verification = acceptance checklist** — proves each sketch became reality
- **snapshots = screenshots for terminal apps** — reviewers see exactly what users see

### definitions

**`.snap`** = a snapshot file that captures what the audience sees. generic term for:
- **code output**: jest `.snap` file with terminal output, response content, or other text
- **browser output**: screenshot `.png` or visual regression `.snap`
- **renderer output**: any visual capture from a renderer

**`.play.test.ts`** = a journey test file that plays through a user experience step by step. named to distinguish from:
- `.test.ts` = unit tests
- `.integration.test.ts` = integration tests
- `.acceptance.test.ts` = acceptance tests (blackbox)
- `.play.test.ts` = journey tests that play through experiences with snapshots

if the repo does not directly support `.play.test.ts`, use:
- `.play.integration.test.ts` = journey test run as integration test
- `.play.acceptance.test.ts` = journey test run as acceptance test

### what terms would they use vs what terms would we use?

| we say | they might say |
|--------|----------------|
| journey test | end-to-end flow test |
| `.play.test.ts` | scenario test, e2e test |
| `given/when/then` with `[tN]` | step-by-step scenario |
| `.snap` | output capture, screenshot |
| repros artifact | test plan |

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| journey tests absent | repros stone requires full BDD `given/when/then` sketches with input/output |
| snapshots absent | repros stone requires snapshot coverage plan |
| repros not connected to verification | verification checklist references repros directly |
| critical paths not identified | repros stone marks critical paths; verification confirms frictionless |
| ergonomics not reviewed | repros stone reviews input/output feel; verification validates in practice |
| verification guard doesn't check | new self-reviews for journeys, snapshots, critical paths, ergonomics |

### what are the pros?

- **explicit connection** between repros → verification
- **snapshot visibility** gives foreman confidence in pr review
- **journey test structure** ensures step-by-step coverage
- **input/output pairs** reveal usage friction early (at repros, not in prod)
- **critical path focus** ensures happy paths are frictionless
- **ergonomics review** catches awkward interfaces before implementation
- **self-review guards** catch omissions before human review

### what are the cons?

- **more upfront work** at repros stone (sketches take time)
- **snapshot maintenance** — snapshots need update when output changes
- **initial effort** — mechanics must understand journey test structure

### what edgecases exist?

- **no visible output** — some behaviors are purely file-based or side-effect only; snapshots may be file content instead
- **dynamic output** — timestamps, uuids in output need mask in snapshots
- **very long output** — may need truncation strategy

pit of success:
- repros guard self-review asks "did you consider snapshot coverage for all outputs?"
- verification guard self-review asks "are dynamic values masked in snapshots?"

## what is awkward?

### where does the design fight the user's mental model?

- mechanics might think "repros = list of tests" not "repros = journey sketches"
- ask for snapshot coverage at repros (before tests exist) feels premature

### what tradeoffs feel uncomfortable?

- guard on repros stone increases friction at research phase
- require snapshots for all behaviors may feel burdensome for simple behaviors

### mitigations

- repros stone content explains *why* journey sketches help (not just what)
- snapshot coverage is "plan" not "promise" — verification is where it's enforced
- guards are self-review only (no peer review) to reduce friction
