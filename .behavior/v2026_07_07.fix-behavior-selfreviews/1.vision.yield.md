# vision: self-review reflection skill

> a skill that reads claude transcripts on the local machine to deterministically
> diagnose **which** self-reviews in the `init.behavior` route actually earn their
> keep — and which are just noise — so we can tune them with evidence, not vibes.

---

## the outcome world

### day-in-the-life

vlad has run the `init.behavior` route **dozens of times across many projects** on this
machine — every `~/.claude/projects/*/` slug that ever drove a behavior route. each route
fires anywhere from 3 (vision, light) to 13 (execution) self-reviews per stone. that's
*hundreds* of "i promise i reviewed this" across the corpus — but which slugs consistently
earn their keep, and which are dead weight on every route, everywhere?

today he runs, with no arguments:

```sh
rhx reflect.selfreviews
```

the skill sweeps **every transcript in every project on the machine**, finds every run of the
**init.behavior route**, reconstructs every self-review window, and — a minute or two later —
hands back a corpus-wide verdict:

```
🔍 reflect.selfreviews
   ├─ corpus  = 41 init.behavior runs · 214 transcripts · 12 projects
   ├─ reviews = 1,308 promised across all stones
   └─ verdict
      ├─ 🟢 genuine-gain    511  (39%)
      ├─ 🟡 genuine-noop    402  (31%)
      └─ 🔴 feigned-noop    395  (30%)

   per-slug across the whole corpus (worst offenders)
   ├─ 🔴 has-questioned-questions      3/88 edits · 81/88 feigned-noop
   ├─ 🔴 has-consistent-conventions    5/74 edits · 63/74 feigned-noop
   ├─ 🟢 has-questioned-assumptions   61/79 edits · surfaced real gaps
   └─ 🟢 behavior-declaration-coverage 66/71 edits · caught missing behaviors
```

the **aha**: `has-questioned-questions` fired **88 times across 41 routes** and produced a
real edit in only 3 — the cheap brain flags the rest as feigned-noops ("clone restated the
questions and promised without triage"). one route could be a fluke; **88 firings across
every project is a verdict**. meanwhile `has-questioned-assumptions` earns its slot corpus-
wide (61/79 led to real edits).

now vlad has machine-scale evidence to act on: delete or rewrite `has-questioned-questions`,
keep `has-questioned-assumptions`, and dig into why `has-consistent-conventions` almost never
bites — confident the pattern is real, not a one-route accident.

*(want to zoom into a single route? `--route <dir>` narrows the same machinery to one.)*

### before / after

| | before | after |
|---|---|---|
| **evidence** | gut feel — "the reviews feel noisy" | per-slug utility rates across *every* route on the machine |
| **sample size** | at best one route's worth of anecdote | the whole corpus — patterns, not accidents |
| **tuning** | guess which reviews to cut, hope | cut the slugs with proven-low yield corpus-wide |
| **visibility** | promise files say "i promise" — no more | transcript reveals what actually happened |
| **cost of a bad review** | invisible tax on every route forever | measured, named, removable |

### the aha moment

the promise file on disk is **content-free**:

```
# promise: has-questioned-questions
i promise i have completed the self-review for "has-questioned-questions".
```

it tells you the review *ran*. it reveals **none of** whether it *mattered*. the only place
the truth lives is the transcript: did the clone read the artifact? did it edit anything
after? was its articulation a genuine critique or a restatement? this skill is the lens that
turns that buried transcript signal into a verdict.

## user experience

### usecases / goals

1. **diagnose (corpus-wide, primary)** — "across *every* behavior route i've ever run on this
   machine, which init.behavior self-reviews are noise?" the machine-wide sweep is the default.
2. **tune** — feed the corpus verdicts back into the guard templates: cut, merge, or reword
   the slugs that prove low-yield across many routes.
3. **drill down** — narrow to one route (`--route`) or one stone (`--stone`) to inspect a
   specific case with the same machinery.

### contract inputs & outputs

**inputs** — all optional; with no flags, the skill sweeps the whole machine.

| flag | meaning | default |
|---|---|---|
| `--route <dir>` | *filter* to one behavior route dir | (unset — **all routes, all projects**) |
| `--project <slug>` | *filter* to one project's transcripts | (unset — all projects) |
| `--stone <name>` | *filter* to one stone (e.g. `1.vision`) | all stones |
| `--brain <choice>` | cheap brain to judge utility, as a `genContextBrain` choice slug | `fireworksai/deepseek/v4-flash` |
| `--mode plan\|apply` | plan = extract + show windows; apply = also run brain judgment | `plan` |
| `--into <path>` | where to emit the report | a machine-level path (e.g. `~/.rhachet/…/reflection.selfreviews.<date>.md`); with `--route`, defaults into that route dir |

**outputs**

- a **treestruct stdout summary** (turtle/beaver vibes) with the corpus verdict rollup +
  per-slug worst-offenders
- a **markdown report** at `--into`: per-slug aggregate rows (firing count, edit rate,
  feigned-noop rate) *and* a per-review detail table (route, stone, edits, read-signal, brain
  verdict + one-line reason)
- a **machine-readable** `reflection.selfreviews.v1.json` — the corpus aggregate, so trends
  can be re-derived or charted without a re-sweep

### what leveraging it looks like

```sh
# the default — sweep every route across every project, deterministic signal only
rhx reflect.selfreviews

# corpus-wide with cheap-brain verdicts
rhx reflect.selfreviews --mode apply

# drill into one route
rhx reflect.selfreviews --route .behavior/v2026_07_07.fix-behavior-selfreviews --mode apply

# corpus-wide but only the vision stone's reviews
rhx reflect.selfreviews --stone 1.vision --mode apply
```

### timeline

1. skill enumerates the corpus — every `~/.claude/projects/<slug>/*.jsonl` (or the subset a
   `--route` / `--project` filter selects)
2. streams each transcript, finds every behavior route, reconstructs the self-review windows
   (chained: prior-promise → this-promise), tolerant of a corrupt or older-schema file
3. **plan mode stops here** — prints per-slug corpus aggregate (windows, edit counts, read
   signals) with zero brain cost
4. **apply mode** feeds each window to the cheap brain (`fireworksai/deepseek/v4-flash`) →
   verdict + reason
5. emits summary stdout + markdown report + `reflection.selfreviews.v1.json` corpus aggregate

## mental model

### how you'd describe it to a friend

> "you know how every AI code-review step swears it reviewed your work? this reads the
> actual chat log and checks whether the review *did* anything — or just said it did."

### analogies

- **a lie detector for self-reviews** — the promise says "i reviewed it"; this checks the
  body language (did you read it? did you change anything?).
- **git blame for review value** — attributes each file change to the review window that
  produced it, so a review with zero attributable changes is exposed.
- **an expense audit** — every self-review is a line-item cost on every route; this finds
  the recurring charges that buy no value.

### their terms vs. our terms

| they say | we say |
|---|---|
| "the review" | a **self-review window** (trigger → promise span in the transcript) |
| "what happened in the review" | a **self-review experience** (articulation text + per-file diffs + duration, distilled from the window) |
| "did it help?" | **utility verdict**: genuine-gain / genuine-noop / feigned-noop |
| "the chat log" | the **transcript** (`~/.claude/projects/<slug>/*.jsonl`) |
| "it changed stuff" | **attributable edits** (Edit/Write tool_use to ARTIFACT files inside the window) |
| "it actually looked" | **read signal** (Read tool_use on a reviewed artifact, not the articulation, in-window) |
| "what it wrote down" | the **articulation** (the `review/self/…` critique file content) |

## evaluation

### how well it solves the goals

- **diagnose which are noise** → directly: per-slug feigned-noop rate.
- **how often** → directly: firing count + verdict distribution per slug.
- **deterministic** → the *extraction* (windows, edit counts, read signals) is 100%
  deterministic from the transcript. only the *utility label* is probabilistic (the brain),
  and even that is grounded in the deterministic evidence.
- **tune the extant reviews** → the report maps 1:1 to guard-template slugs, so acting on it
  is mechanical.

### the deterministic backbone (the key design bet)

the wish says "deterministically assess". the honesty here: **the strongest signals are
already deterministic and need no brain at all**:

| signal | how | what it reveals |
|---|---|---|
| **attributable edits** | count each Edit/Write tool_use to ARTIFACT files within the review's window (prior-promise → this-promise); the mandatory `review/self/` articulation write is left out | zero artifact edits = suspicious |
| **read-the-artifact?** | was there a Read of a reviewed file (not the articulation) in-window? | promised-without-read = feigned-noop |
| **articulation substance** | assistant text volume in-window (a coarse proxy — the real critique lives in the articulation FILE, see phase 1) | one-liner promise vs. real critique |
| **fire frequency** | count of triggers per slug across routes | how often it even runs |

these scalars are the **plan-mode backbone** — cheap, reproducible, zero brain. crucially,
`editCount` covers artifact edits ONLY: every promised review MUST write its articulation under
`<route>/review/self/`, and a tally that treats that mandatory ritual write as an edit saturates
the edit rate to ~100% by construction. leave it out (and re-reads of it), and the signal returns.

### the two-phase apply model (distill, then grade)

the scalar backbone reveals *behavior* (did an artifact change? did it read?). it cannot reveal
whether the *thinking* was real — because the four numbers throw away the one artifact that proves
it: the **articulation file the review actually wrote** (the `A1…An`-style critique), plus *what* it
changed and *how*. the fix: stop the pre-brain compression of that evidence.

**phase 1 — distill the experience (deterministic).** for each self-review window, reconstruct a
`SelfReviewExperience` purely from the transcript — no new inference, just retain what we used to drop:

| field | source (all in-transcript, deterministic) |
|---|---|
| `articulation` | the `content` of the in-window Write to `<route>/review/self/…` — the real critique text |
| `duration` | `lastEvent.timestamp − firstEvent.timestamp` |
| `files[]` | every artifact file the review touched: `{ path, mode: read\|write, diff }` — `diff` from the Edit/Write tool input (old→new / content), the articulation write left out |

the experience is a **bounded** distillation, not a firehose: per-file diff length and file count
are capped (with truncation markers) so 800+ windows stay cheap. `SelfReviewSignals` does not die —
it stays the plan-mode backbone; the experience is the apply-mode enrichment layered on top.

**phase 2 — grade the experience (probabilistic).** the cheap brain reads the FULL experience —
the articulation text, the per-file diffs, the duration — and labels it. it splits a *genuine-noop*
("read it, reasoned in the articulation, correctly found no gap") from a *feigned-noop* ("promised
with a thin/empty articulation and no artifact change"). that judgment — was the written critique
real? — is the one call determinism cannot draw, so it is exactly, and only, where we spend brain
tokens. a grade of real reasoning beats a pattern-match over four integers.

### pros

- turns an invisible tax into a measurable one
- deterministic core → trustworthy, cheap, reproducible
- brain only where determinism genuinely falls short → low cost
- output maps straight to the guard templates we want to tune
- generalizes later to a bhrain route-reflection skill (wish's stated endgame)

### cons / tradeoffs

- **corpus-wide sweep is heavy** — potentially thousands of transcripts across all projects;
  the parse must stream (not slurp) and cache the aggregate so re-runs are cheap.
- **transcript↔route correlation is fiddly** (see awkward). one route can span multiple
  sessions; one session file can hold multiple routes/branches.
- **brain verdict is probabilistic** — mitigated: it's grounded in deterministic evidence, and
  plan mode needs no brain at all.
- **zero-edit ≠ always-bad** — a legitimately-correct artifact yields no edits; we must not
  punish genuine-noops. this is *why* the brain layer exists.
- **one bad transcript must not sink the sweep** — a corrupt or newer-schema file is skipped
  (and reported), never fatal to the corpus run.

### edgecases & pit-of-success

| edgecase | how the contract keeps you safe |
|---|---|
| a `--project`/`--route` filter matches no transcript | fail loud (ConstraintError) with the slug it looked for, not a silent empty report |
| default sweep finds zero behavior routes anywhere | fail loud with a hint (no behavior routes in `~/.claude/projects`), not an empty success |
| one corrupt / newer-schema transcript in the corpus | skip it, tally it under `skipped`, and report the count — never abort the whole sweep |
| route spans multiple session files | correlate by cwd + gitBranch + timestamp, union the windows; report which sessions were used |
| same route re-run across many sessions/branches | each run's windows counted once; per-slug aggregate sums across runs (that's the trend) |
| a review fired but never promised (abandoned) | count it as `unpromised` — itself a useful signal |
| brain unavailable / no key | plan mode still works fully; apply mode fails loud with the fix hint |
| genuine-noop vs feigned-noop | brain grounded in read-signal + edit-count, not vibes |
| older routes with generic slugs (`all-done-junior`) | still parseable; per-slug aggregate just buckets them separately |

## open questions & assumptions

### assumptions made

1. **the transcript is the source of truth** for review utility, because on-disk promise
   files are content-free (verified — see groundwork).
2. self-review windows form a **chain**, not independent spans. the stone's *first* review
   is triggered by `route.stone.set --as passed`; thereafter each `--as promised --that <slug>`
   both **closes** review N and **triggers** review N+1. so the window for review N is
   `[prior-promise (or passed) → this-promise]`, all visible as Bash tool_use in the
   transcript. (verified first-hand as i drove this very route — the promise of one review
   immediately surfaces the next.)
3. edits inside a window are fairly attributable to that review. *(mostly true; interleaving
   is the risk — see awkward.)*

### open questions — triaged

1. **[answered] which cheap brain, concretely?** — **settled by the wisher.** the brain is
   `genContextBrain` from `rhachet` / `rhachet/brains` (verified at
   `node_modules/rhachet/readme.md:59,450`), and the concrete choice is
   **`fireworksai/deepseek/v4-flash`** — the `--brain` default. the call path is
   `genContextBrain({ choice: 'fireworksai/deepseek/v4-flash' })` then
   `context.brain.atom.ask({ role, prompt, schema })` — atom is a **stateless single
   inference** (cheapest grain) and `schema` returns a structured verdict directly. the one
   leftover detail (a research/setup item, not a fork): confirm the `rhachet-brains-*`
   supplier package that serves the `fireworksai/…` choice is installed, and its credential
   is on the keyrack.
2. **[answered] scope of "changes from self reviews"** — decided via logic: the signal set is
   Edit/Write edits + read-signal + articulation substance, **plus Bash tool_use (test runs)
   for verification-stone reviews** (see assumptions-review A3). the wisher may refine depth,
   but a workable default is set. no longer blocking.
3. **[answered] which routes to reflect on** — **settled by the wisher: ALL of them — every
   *instance* of the init.behavior route type, machine-wide.** not the single bound route, but
   also **not every kind of route.** the breadth is *sample count*, not *route variety*: we
   sweep every project's transcripts for runs of the **init.behavior route type** and
   aggregate their self-reviews. this honors the wish's dual instruction — "specific to the
   init.behavior route" + "start focused on the behavior route" (scope stays one route type) —
   while the "all projects" breadth simply supplies enough samples for the per-slug rates to be
   trustworthy. generic any-route reflection stays deferred to bhrain, exactly as the wish
   states. mechanically: the default corpus is `~/.claude/projects/*/*.jsonl`, but the skill
   **filters to behavior-route windows** within them; `--route` narrows to one instance,
   `--project` to one project. this makes **cross-instance aggregation the core deliverable**,
   not an optional add-on. (see the reshaped outcome-world + edgecases below.)
4. **[answered] is verdict taxonomy right?** — **confirmed by the wisher.** three buckets:
   `genuine-gain` / `genuine-noop` / `feigned-noop`. the genuine/genuine/feigned parallelism is
   the point: `genuine-noop` ("looked, found no gap") vs `feigned-noop` ("did not truly look,
   faked the no-op") is the exact distinction the middle bucket exists to draw.

## what is awkward?

1. **transcript↔route correlation is the hard part.** a behavior route can span several
   claude sessions (multiple `*.jsonl`), and a session file can contain multiple routes and
   branches. matching "this window belongs to this route/stone" relies on parsing the actual
   `route.stone.set` bash args and the cwd/branch — deterministic but fiddly, and the top
   source of potential mis-attribution.
2. **edit attribution can smear.** if a clone interleaves unrelated edits inside a review
   window, they get counted toward that review. the window boundary (trigger→promise) is a
   proxy for "review work", not a guarantee. we accept some noise; the brain reason-string
   can flag when edits look unrelated.
3. **the wish says "deterministic" but also "ask a brain".** these are in gentle tension. i
   settle it: *extraction* stays deterministic, *labels* stay probabilistic — but it's worth
   a callout, because someone who expects a purely deterministic verdict will be surprised
   that the genuine-noop-vs-feigned-noop call is a model's judgment.
4. **machine-wide sweep is a broad blast radius.** the default reads *every* transcript in
   *every* project on the machine — potentially thousands of files, some large. this is a
   read-only diagnostic (no mutation), but the default run still touches unrelated projects'
   logs, so it must stream, cache the aggregate, and survive a single corrupt/older-schema
   transcript rather than abort the whole sweep.
5. **zero-edit reviews are ambiguous by nature.** the whole value hinges on the split between
   "correctly found no gaps" (genuine-noop) and "did not truly look" (feigned-noop) — the one
   judgment we can't fully mechanize. it's the reason the skill exists *and* its softest spot.
