> note on the apply-success variant: at the CLI (in-process) layer it is snapped with a
> deterministic injected judge (`reflect.selfreviews.integration.test.ts` [case1 t2]). at the
> shell (subprocess) layer it is deliberately NOT snapped: the real cheap brain's verdict
> varies run to run, so a shell apply-success snapshot would be flaky and would add a live-brain
> call to every acceptance run for no new code path. real-brain apply correctness is proven
> unsnapped in `imagineSelfReviewVerdict.integration.test.ts`. this scope split is documented
> inline at `skill.reflect.selfreviews.acceptance.test.ts` above [case5].

> note on the invalid-input variant: `getCliArgs` surfaces a bad flag via `console.error` + a
> `process.exit(1)` — it does NOT throw. that hard exit cannot be observed in-process (it would
> kill the jest worker), so the invalid-`--mode` snapshot lives only at the subprocess layer
> ([case6]), where the exit code + stderr block are captured cleanly.

### snapshot change rationalization
all `.snap` changes this cycle were resnapped on purpose after known output changes:
- corpus line gained `· N projects` (feature: projectsSwept)
- corpus line pluralization now grammar-correct (`1 route run` vs `2 route runs`, etc.)
- `--help` gained the `~/.rhachet/reflect` default hint (feature: machine-level default path)
- `--help` option columns now width-aligned so every `[default:]` tag starts at one column
- treestruct stdout no longer bookended by stray blank lines (ergo: opens at header, ends at last line)
- plan-mode report verdict + reason cells both render the em-dash placeholder (symmetric `| — | — |`)
- report fixture reasons now agree with their verdict labels (coherent documented example)
- acceptance snap gained [case6] the invalid-`--mode` validation block (new contract variant)
- no accidental snapshot drift; each change traces to a named feature or fix in the yield.

> note on two error icons: the snapshots hold `⛈️  error: input invalid` for arg-validation
> errors and `✋ ConstraintError: ...` for domain errors. this is a deliberate repo-wide
> convention, not a blemish: `⛈️` is emitted by the shared `src/infra/cli/getCliArgs.ts`
> validator (also seen in `getCliArgs.test.ts.snap` and the three `init.behavior.*` acceptance
> snaps), while `✋` is emitted by `helpful-errors` for domain constraint walls. the two icons
> mark two error classes; a caller reads input-invalid vs domain-wall at a glance.
>
> note on the json aggregate snapshot: it snaps the RAW file bytes via
> `expect(readFileSync(jsonPath, 'utf-8')).toMatchSnapshot()`, so the diff shows exactly what a
> caller opens — the real `JSON.stringify(x, null, 2)` output, strict json with no comma after
> the last member, consistent with the error blocks (also raw strings). an earlier version
> snapped a re-parsed object, whose jest pretty-format added a comma after the last member and
> diverged from the true artifact; the raw-bytes snapshot is both blemish-free and more correct.
