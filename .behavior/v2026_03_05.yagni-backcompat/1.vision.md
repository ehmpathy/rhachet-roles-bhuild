# vision: minimalism & consistency self-reviews

## the outcome world

### before: coverage and adherance only

the current self-review frame checks two dimensions:
- **coverage**: did we implement all that was required?
- **adherance**: did we implement it correctly?

this catches omissions and violations, but misses two other failure modes:

```
developer implements feature X
  + adds "helpful" feature Y (not requested)
  + adds backwards compatibility for deprecated API (not requested)
  + adds new utility function instead of extant one
  + uses new name convention instead of extant patterns

self-reviews pass ✓
  - coverage: all requirements met
  - adherance: implementation matches spec

but feature shipped with:
  - unnecessary code to maintain (minimalism failure)
  - inconsistent patterns to understand (consistency failure)
```

### after: minimalism and consistency dimensions added

the self-review frame adds two new dimensions:

```
                    coverage      adherance      minimalism     consistency
                  ┌─────────────┬─────────────┬─────────────┬─────────────┐
  behavior.decl   │ omitted     │ spec        │ extras      │ divergence  │
                  │ features    │ drift       │ not asked   │ from extant │
                  └─────────────┴─────────────┴─────────────┴─────────────┘
```

four new reviews:

| dimension | review | catches | question |
|-----------|--------|---------|----------|
| minimalism | `has-pruned-yagni` | over-built code, gold-plated additions, premature abstraction | "was this asked for?" |
| minimalism | `has-pruned-backcompat` | backwards compat that wasn't explicitly requested | "did they say to keep the old way?" |
| consistency | `has-consistent-mechanisms` | new mechanisms that duplicate extant functionality | "does this already exist in the codebase?" |
| consistency | `has-consistent-conventions` | divergence from extant names/patterns | "what conventions does the codebase use?" |

### the "aha" moment

the frame clicks when you realize:

> juniors over-build as often as they under-build, and they reinvent as often as they reuse

coverage catches under-building. minimalism catches over-building. consistency catches reinventing.

together they enforce a focused, integrated implementation:
- not less than required (coverage)
- not more than required (minimalism)
- not different from extant (consistency)
- and what extant code is correct (adherance)

## user experience

### usecases

1. **behaver agent** builds blueprint for a feature
2. **guard triggers** minimalism reviews
3. **agent finds** an abstraction it added "for flexibility" → removes it
4. **guard triggers** consistency reviews
5. **agent finds** a new utility that duplicates extant one → uses extant
6. **agent finds** new name convention → aligns with extant conventions
7. **human reviewer** sees: focused, integrated implementation

### contract shape

```yaml
reviews:
  self:
    # minimalism: YAGNI
    - slug: has-pruned-yagni
      say: |
        review for extras that were not prescribed.

        YAGNI = "you ain't gonna need it"

        for each component in the blueprint, ask:
        - was this explicitly requested in the vision or criteria?
        - is this the minimum viable way to satisfy the requirement?
        - did we add abstraction "for future flexibility"?
        - did we add features "while we're here"?
        - did we optimize before we knew it was needed?

        if something was not requested, delete it or flag it as an open question
        for the wisher to decide.

        emit findings to .behavior/$thisBehavior/reviews/{stone}.minimalism-yagni.md

    # minimalism: backwards compatibility
    - slug: has-pruned-backcompat
      say: |
        review for backwards compatibility that was not explicitly requested.

        for each backwards-compat concern in the blueprint, ask:
        - did the wisher explicitly say to maintain this compatibility?
        - is there evidence this backwards compat is needed?
        - or did we assume it "to be safe"?

        if backwards compat was not explicitly requested:
        1. flag it as an open question for the wisher
        2. plan to eliminate it if not confirmed as required
        3. make the open question very clearly reported

        emit findings to .behavior/$thisBehavior/reviews/{stone}.minimalism-backcompat.md

    # consistency: mechanisms
    - slug: has-consistent-mechanisms
      say: |
        review for new mechanisms that duplicate extant functionality.

        unless the ask was to refactor, be consistent with extant mechanisms.

        for each new mechanism in the blueprint, ask:
        - does the codebase already have a mechanism that does this?
        - do we duplicate extant utilities, helpers, or patterns?
        - could we reuse an extant component instead of a new one?

        if a new mechanism duplicates extant functionality:
        1. replace with the extant mechanism
        2. or flag as an open question if unsure

        emit findings to .behavior/$thisBehavior/reviews/{stone}.consistency-mechanisms.md

    # consistency: conventions
    - slug: has-consistent-conventions
      say: |
        review for divergence from extant names and patterns.

        unless the ask was to refactor, be consistent with extant conventions.

        for each name choice in the blueprint, ask:
        - what name conventions does the codebase use?
        - do we use a different namespace, prefix, or suffix pattern?
        - do we introduce new terms when extant terms exist?
        - does our structure match extant patterns?

        if we diverge from extant conventions:
        1. align with the extant convention
        2. or flag as an open question if the extant convention seems wrong

        emit findings to .behavior/$thisBehavior/reviews/{stone}.consistency-conventions.md
```

### timeline

1. behaver builds blueprint
2. guard triggers minimalism reviews
3. behaver removes 2 unrequested abstractions
4. behaver flags 1 backcompat assumption as open question
5. guard triggers consistency reviews
6. behaver replaces new utility with extant one
7. behaver renames to match extant conventions
8. behaver emits reports, requests approval
9. human sees: focused, integrated implementation

## mental model

### how users describe it

> "minimalism catches me when I over-engineer. consistency catches me when I reinvent the wheel or go off-brand."

### analogies

**minimalism - the sculptor's chisel**: a sculpture is finished not when there's no more to add, but when there's no more to remove.

**consistency - the style guide**: a codebase has its own voice. new code should sound like it belongs, not like a foreign import.

**the integration test**: if your code can't find its dependencies because you renamed conventions, you failed the consistency review.

### terminology

| user term | our term |
|-----------|----------|
| "extra stuff" | minimalism violation (yagni) |
| "gold-plating" | minimalism violation (yagni) |
| "just in case" | minimalism violation (backcompat) |
| "reinventing the wheel" | consistency violation (mechanisms) |
| "off-brand names" | consistency violation (conventions) |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| catch over-engineering | yes - minimalism/yagni review |
| catch unasked backcompat | yes - minimalism/backcompat review |
| catch duplicated mechanisms | yes - consistency/mechanisms review |
| catch name divergence | yes - consistency/conventions review |
| work in light + heavy guards | yes - designed for both |

### pros

- **explicit**: named reviews for specific anti-patterns
- **actionable**: clear delete-or-align guidance
- **organized**: two clear dimensions (minimalism + consistency)
- **complements extant frame**: coverage + adherance + minimalism + consistency = complete

### cons

- **more reviews**: 4 additional reviews
- **judgment calls**: some items may be genuinely unclear
- **research required**: consistency reviews need codebase exploration

### pit of success

the frame keeps users safe by:
- default to all 4 reviews on blueprint guards
- mandatory open question flagging for backcompat
- "align with extant" as default path for consistency
- elimination as default path for minimalism

### edgecases

| edgecase | handle |
|----------|--------|
| genuinely needed flexibility | flag as open question, let wisher decide |
| extant convention is bad | flag as open question, suggest refactor scope |
| no extant pattern exists | ok to introduce, but document the precedent |
| refactor task | consistency reviews relax (explicitly changing conventions) |

## open questions & assumptions

### assumptions

1. **over-engineering is common**: juniors (and agents) add extras frequently enough to warrant systematic review
2. **reinvention is common**: agents often create new utilities instead of finding extant ones
3. **conventions drift**: without explicit review, new code diverges from extant patterns
4. **elimination is safer**: better to flag and get confirmation than to silently ship extras

### open questions

1. ~~should minimalism reviews also apply to execution guards, or only blueprint?~~ **resolved: both**
2. ~~should consistency reviews require explicit codebase exploration before promising?~~ **resolved: yes, ask author to search for related codepaths if not already done in prior research stone**

### validated with wisher

- ✓ both light and heavy blueprint guards should include these reviews
- ✓ elimination-as-default posture for minimalism
- ✓ align-with-extant posture for consistency

## what is awkward?

### what feels off

- **slugs settled**: `has-pruned-X` and `has-consistent-X` read well as "i promise that it has-pruned-yagni"
- **overlap with deletables**: the extant `has-questioned-deletables` review has some overlap with YAGNI

### where design fights mental model

- developers often think "more is better" - minimalism actively fights this
- developers often think "new is better" - consistency actively fights this
- backwards compat feels "safe" - the review challenges this assumption

### tradeoffs

- **thoroughness vs speed**: 4 more reviews = slower iteration
- **strictness vs autonomy**: forcing open questions reduces agent autonomy
- **consistency vs improvement**: aligning with bad conventions perpetuates them

the tradeoffs feel acceptable because:
- blueprint is a design phase where thoroughness matters
- over-engineering and inconsistency costs compound over time
- open questions are cheap; wrong assumptions are expensive
- refactor tasks can explicitly relax consistency requirements
