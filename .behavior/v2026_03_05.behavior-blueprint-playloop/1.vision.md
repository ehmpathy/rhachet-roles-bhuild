# vision: enframe.playloops

## the outcome world

### before: tests pass, users suffer

today, a behavior goes through the route:
1. wish declared
2. vision painted
3. criteria defined
4. blueprint created with test coverage
5. execution completed, tests pass
6. ... user tries it and finds critical friction

the gap: tests verify *code correctness*, not *user experience correctness*. a test can assert `expect(result).toBeDefined()` while the actual experience is unusable - wrong dependencies, missing edge cases, poor ergonomics.

### after: playtest before you ship

with enframe.playloops:
1. wish declared
2. vision painted
3. criteria defined
4. **playloop designed** - "here's exactly how to experience this as the user would"
5. codepaths designed - leverages the playloop for experience tests
6. execution completed, **playloop validates the experience**

the aha moment: when a builder runs an experience test and sees the *exact same experience* the user will have - same inputs, same outputs, same friction points. experience tests leverage the playloop framework. if it feels wrong in the test, it'll feel wrong for the user.

### day-in-the-life

**before playloop:**
- builder finishes feature
- runs tests, all green
- ships to user
- user reports "this doesn't work with X" or "Y is unusable"
- builder debugs, realizes they never tested from user perspective
- churn: fix, re-ship, re-test

**after playloop:**
- builder finishes feature
- runs experience tests (which leverage the playloop)
- immediately feels the friction the user would feel
- fixes it before ship
- ships confident that the experience matches the wish

## user experience

### usecases

1. **designing acceptance tests**: builder uses playloop to structure how they'll verify the user experience
2. **rapid iteration**: builder loops through playloop quickly to diagnose and eliminate friction
3. **experience validation**: playloop proves the deliverable matches the wish from user's pov

### contract inputs & outputs

**stones to create:**

1. `3.1.research.playloop._.stone` - research what framework enables rapid playtest feedback (optional, as needed)
2. `3.2.enframe.playloops.stone` + guard - declare/propose the playloop framework

**what we get from each stone:**

- `3.1.research.playloop._.stone`: understanding of what framework/tools enable rapid playtest feedback for this behavior (if not already known)
- `3.2.enframe.playloops.stone`: a declared playloop framework that enables the builder to rapidly reproduce and validate the user experience via acceptance tests

**playloop guard self-reviews:**
```yaml
reviews:
  self:
    - slug: has-realworld-reproductions
      say: |
        does this playloop reproduce the user experience?
        - does it start from the user's entry point?
        - does it use realistic inputs?
        - does it validate what the user actually cares about?
```

### timelines

1. **enframe phase**: design playloop before blueprint
2. **execution phase**: playloop implemented alongside feature
3. **validation phase**: experience tests run through playloop
4. **continuous**: playloop lives as acceptance test, catches regressions

## mental model

### how users describe this to a friend

> "before we ship a feature, we have to write down exactly how the user will experience it - like a sequence of steps. then we build a test that runs through that sequence. if the test feels wrong, the feature feels wrong. it's like playtesting a game before you ship."

### analogies

- **game playtesting**: before shipping a level, you play through it yourself to feel what players will feel
- **dress rehearsal**: actors run through the full show before opening night to catch issues
- **test drive**: before buying a car, you drive it to feel how it handles

### terms

| we say | users say |
|--------|-----------|
| playloop | "experience test" |
| acceptance/contract test | "does the API work?" |
| acceptance/experience test | "does the user journey work?" |
| rapid iteration | "quick feedback" |
| codepaths | "the actual code and tests" |

### acceptance test taxonomy

**acceptance tests split into two categories:**

| type | focus | verifies |
|------|-------|----------|
| acceptance/contract | API correctness | inputs/outputs match contract |
| acceptance/experience | user journey | playloop - full user experience reproduction |

experience tests leverage the playloop framework. they reproduce the user journey from entry point to outcome.

the enframe.playloops stone ensures the playloop is designed before implementation, so experience tests have a clear pattern to follow.

## evaluation

### how well does it solve the goals?

| goal | how playloop addresses it |
|------|---------------------------|
| playtestable deliverables | playloop forces explicit design of user experience before code |
| fast feedback | playloop enables rapid iteration through experience tests |
| reduce churn | catches experience issues before ship |
| acceptance test quality | playloop becomes the framework for experience tests |

### pros

- forces user-perspective thinking before implementation
- fast iteration catches friction early
- acceptance tests become meaningful (not just "does it not crash")
- clear separation: playloop = experience, codepaths = implementation

### cons

- adds a stone to the route (more time upfront)
- requires discipline to design playloop properly
- may feel redundant if codepaths already has acceptance tests

### edgecases & pit of success

| edgecase | how contracts keep users safe |
|----------|------------------------------|
| playloop doesn't match user experience | guard self-review asks "does this reproduce the user's entry point?" |
| playloop uses unrealistic inputs | guard self-review asks "does it use realistic inputs?" |
| playloop validates wrong things | guard self-review asks "does it validate what the user actually cares about?" |

## what is awkward?

### feels off

- **naming**: "playloop" is clear but might be confused with "gamedev". alternatives: "testdrive", "experienceloop", "userloop"?
- ~~**relationship to codepaths**: the boundary between playloop and codepaths acceptance tests could be fuzzy.~~ **clarified**: playloop defines the pattern for blackbox/experience tests. codepaths implements tests that leverage the playloop.

### fights mental model

- ~~builders might think "i already write acceptance tests in codepaths, why another stone?"~~ **clarified**: playloop defines HOW to structure experience tests. codepaths implements tests that leverage the playloop. the distinction is design-first (playloop) vs implementation (codepaths).

### uncomfortable tradeoffs

- **more upfront work**: adding a stone means more time before execution. the payoff is less churn, but it's still more work.
- **discipline required**: a bad playloop is worse than no playloop (false confidence). guards must catch lazy playloops.
