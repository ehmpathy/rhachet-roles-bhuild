# blueprint: enframe.playloops

## summary

add two new stones to the behavior route:
1. `3.1.research.playloop._.v1.stone` - optional research for playloop framework discovery
2. `3.2.enframe.playloops.v1.stone` + guard - declare/propose the playloop framework

these stones slot before the extant `3.3.blueprint.v1.stone` which remains untouched.


## codepaths

### 3.1.research.playloop._.v1.stone

```
[+] research what framework enables rapid playtest feedback for this behavior
    ├── reference wish, vision, criteria
    ├── guide discovery of playtest frameworks
    │   ├── what tools/patterns enable rapid playtest feedback?
    │   ├── what extant frameworks could be leveraged?
    │   └── what needs to be built vs reused?
    └── emit to $BEHAVIOR_DIR_REL/3.1.research.playloop._.v1.i1.md
```

### 3.2.enframe.playloops.v1.stone

```
[+] declare the playloop framework for this behavior
    ├── reference wish, vision, criteria, research (if done)
    ├── guide declaration of playloop
    │   ├── what framework enables rapid playtest feedback?
    │   ├── what setup is required?
    │   ├── what is the experience sequence?
    │   └── what assertions validate the experience?
    ├── scope: happy paths + guard paths
    │   ├── happy paths: expected user journey
    │   └── guard paths: paths that keep users on track, self heal, or provide idempotency
    └── emit to $BEHAVIOR_DIR_REL/3.2.enframe.playloops.v1.i1.md
```

### 3.2.enframe.playloops.v1.guard

```
[+] guard for enframe.playloops stone
    ├── reviews.self
    │   └── has-realworld-reproductions
    │       ├── does this playloop reproduce the user experience?
    │       ├── does it start from the user's entry point?
    │       ├── does it use realistic inputs?
    │       └── does it validate what the user actually cares about?
    └── judges
        └── approved? (human approval required)
```

## contracts

the guard uses `has-realworld-reproductions` slug per the "i promise that it $slug" pattern.

## integration notes

### behavior init

the `initBehavior` operation will copy these new template files when it scaffolds a behavior route.

### stone order

route stones are ordered by numeric prefix:
- 3.1.* (research) comes before 3.2.* (distill/enframe) comes before 3.3.* (blueprint)

stones with the same prefix are parallel (e.g., `3.2.distill.domain` and `3.2.enframe.playloops` are both at the 3.2 level). this follows the same pattern as research stones (multiple `3.1.research.*` stones coexist).

the numeric prefix ensures correct order without changes to bhrain route logic.
