# vision: fix reviews against shells

## the outcome world

### before/after contrast

| before | after |
|--------|-------|
| behavior touches only `.sh` files | same |
| peer reviews configured for `**/*.ts` | peer reviews configured for `**/*.{ts,sh}` |
| `--diffs since-main` ∩ `--paths-with '**/*.ts'` = ∅ | `--diffs since-main` ∩ `--paths-with '**/*.{ts,sh}'` = `.sh` files |
| "combined scope resolves to zero files" error | reviews run against shell files |
| stone blocked, human intervention required | stone passes, reviews complete |

### day-in-the-life

mechanic writes a new skill (`my-skill.sh`). they complete execution, mark as passed. the guard's peer reviews now include `.sh` in their path patterns, so the reviews run against the shell file changes and provide feedback.

### aha moment

"oh, we can review shell files the same way we review typescript — the rules that apply to code patterns, error handling, and naming conventions work for bash too."

## user experience

### usecases

1. **shell-only behavior**: mechanic implements a behavior that only touches `.sh` files (e.g., new skill, skill fix)
2. **mixed behavior**: mechanic implements behavior that touches both `.ts` and `.sh` files
3. **typescript-only behavior**: unchanged — reviews still work as before

### contract inputs & outputs

no new contracts. extant peer review commands gain shell file coverage via brace expansion:

```bash
# before: only typescript
$rhx review --paths-with '**/*.ts' --diffs since-main --join intersect

# after: typescript + shell (brace expansion)
$rhx review --paths-with '**/*.{ts,sh}' --diffs since-main --join intersect
```

### timelines

instant. once guards are updated, all behaviors that touch shell files get reviewed.

## mental model

### how users describe it

"reviews now cover shell files, not just typescript"

### analogies

like adding `.css` to a linter that previously only checked `.js` — same rules engine, broader file coverage.

### terms

| user term | system term |
|-----------|-------------|
| shell file | `**/*.sh` glob pattern |
| reviews work | `--paths-with` includes `.sh` |
| zero files error | intersection of diffs and paths is empty |

## evaluation

### pros

- **complete coverage**: behaviors touching shell files get reviewed
- **minimal change**: expand `**/*.ts` to `**/*.{ts,sh}` in extant review commands
- **consistent**: same review mechanisms work for both languages
- **no regressions**: typescript-only behaviors unchanged

### cons

- **rule applicability**: some rules written for typescript may not apply well to bash (e.g., type-safety rules)
  - mitigation: rules should be language-appropriate; bash-specific rules could be added later
- **false positives**: bash has different idioms than typescript
  - mitigation: reviewers (brains) can distinguish context

### edgecases

| edge case | handling |
|-----------|----------|
| behavior touches only `.md` files | still zero files — but `.md` changes are documentation, not code |
| behavior touches `.ts`, `.sh`, and `.md` | `.ts` and `.sh` reviewed; `.md` excluded (as intended) |
| no changed files at all | reviews short-circuit gracefully (extant behavior) |

## open questions & assumptions

### assumptions

1. **rules are language-agnostic enough**: rules like "forbid failhide" apply to bash error handling too
2. **review brains handle bash**: the brains reading code can understand bash patterns
3. **no bash-specific rules needed yet**: extant rules suffice for now

### questions for wisher

2. should `.sh` be added to ALL peer reviews, or only a subset?
3. are there other file types that should be included (`.zsh`, `.bash`)?

### external research

none — no external dependencies. this is internal configuration change.

### internal research

**verified extant behavior:**

1. **guard files with peer reviews** — `src/domain.operations/behavior/init/templates/*.guard*`
   - `5.1.execution.from_vision.guard` lines 156-215: peer reviews use `--paths-with '**/*.ts'`
   - `5.3.verification.guard` lines 229-286: peer reviews use `--paths-with '**/*.ts'`

2. **paths-with flag semantics** — from review skill usage:
   - brace expansion works (e.g., `**/*.{ts,sh}`)
   - multiple `--paths-with` flags are OR'd together
   - `--join intersect` intersects paths with diffs

3. **the error** — "combined scope resolves to zero files" occurs when:
   - `--diffs since-main` returns `.sh` files only
   - `--paths-with '**/*.ts'` filters for `.ts` only
   - intersection is empty

**vocab to reuse:**
- `--paths-with` — extant flag pattern
- peer review slugs — extant naming convention

## what is awkward?

1. **rule applicability**: some typescript rules (e.g., `rule.require.given-when-then.md` for tests) won't find matches in `.sh` files. this is fine — the review simply won't find violations.

2. **inconsistent patterns**: some peer reviews use `--paths-with '**/*.test.ts'` specifically. should these become `--paths-with '**/*.test.ts' --paths-with '**/*.test.sh'`? probably not — we don't have `.test.sh` files.

3. **future extensibility**: if we add more file types later (e.g., `.zsh`, `.bash`), we'll need to expand the brace pattern again.

---

## summary

change `--paths-with '**/*.ts'` to `--paths-with '**/*.{ts,sh}'` in peer review commands in execution and verification guards. this fixes the "zero files" error for shell-only behaviors while maintaining extant typescript coverage.
