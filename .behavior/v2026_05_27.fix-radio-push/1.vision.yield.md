# vision: rename --repo to --into/--from for radio skills

## the outcome world

**before**: mechanic runs `rhx radio.task.push --repo ehmpathy/foo --title "..."` but rhx intercepts `--repo` as its own flag (which repo's skills to use), so the skill never receives the target repo argument.

**after**: mechanic runs `rhx radio.task.push --into ehmpathy/foo --title "..."` and it works. the flag names are semantically clear: push INTO a repo, pull FROM a repo.

**aha moment**: when the mechanic realizes `--into` and `--from` not only avoid the collision but actually read better in context.

## user experience

### usecases

| usecase | command |
|---------|---------|
| push task to current repo | `rhx radio.task.push --via gh.issues --title "..." --description "..."` |
| push task to specific repo | `rhx radio.task.push --via gh.issues --into owner/repo --title "..."` |
| pull tasks from current repo | `rhx radio.task.pull --via gh.issues --list` |
| pull tasks from specific repo | `rhx radio.task.pull --via gh.issues --from owner/repo --list` |

### contract inputs & outputs

**radio.task.push**:
- `--via` (required): channel — `gh.issues` or `os.fileops`
- `--into` (optional): target repo as `owner/name`, defaults to current git repo
- `--title` (required for new): task title
- `--description`: task description
- `--exid`: external id for updates
- `--status`: QUEUED, CLAIMED, DELIVERED
- `--idem`: findsert or upsert

**radio.task.pull**:
- `--via` (required): channel — `gh.issues` or `os.fileops`
- `--from` (optional): source repo as `owner/name`, defaults to current git repo
- `--list`: list all tasks
- `--exid` or `--title`: fetch specific task
- `--status`: filter by status
- `--limit`: max tasks to return

## mental model

**how users describe it**: "push tasks INTO a repo, pull tasks FROM a repo — the flags match the direction."

**analogy**: like `cp --into dir/` or `git clone --from url` — the preposition matches the data flow direction.

## evaluation

**solves the goal?** yes — `--into` and `--from` avoid collision with rhx's `--repo` flag.

**pros**:
- semantic clarity: push INTO, pull FROM
- rhx passthrough: `--repo` now passes through as ignored arg
- documented: skill headers now list all options

**cons**:
- breaking change for anyone using `--repo` (but skill is new, unlikely users)

**edge cases**:
- no `--into`/`--from` provided: falls back to git context (current repo)
- invalid format: fails fast with clear error message

## open questions & assumptions

**assumptions made**:
- no extant users depend on `--repo` flag (skill is recent)
- `--into`/`--from` are not reserved by rhx

**questions answered**:
- rhx passthrough args require `repo` in schema — added as ignored passthrough

**must validate with wisher**: none — task description was clear
