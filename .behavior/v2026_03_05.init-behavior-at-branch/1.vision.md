# vision: init.behavior --name @branch

## the outcome world

### before

```sh
$ git checkout -b casey/add-user-auth
$ rhx init.behavior --name add-user-auth
```

developer has to:
1. look at their branch name
2. mentally strip the `casey/` prefix
3. type it out manually
4. hope they didn't typo

### after

```sh
$ git checkout -b casey/add-user-auth
$ rhx init.behavior --name @branch
# creates: .behavior/v2026_03_05.add-user-auth/
```

one token, zero thought.

### the "aha" moment

> "wait, i don't have to type the behavior name? it just knows from my branch?"

the value clicks when you realize:
- branch names already encode intent (that's why you named it that way)
- behavior names should match branch names (that's the convention)
- why type it twice?

## user experience

### usecases

| goal | command | result |
|------|---------|--------|
| start new behavior for feature branch | `rhx init.behavior --name @branch` | behavior created, named after branch |
| start behavior on `vlad/fix-auth-bug` | `rhx init.behavior --name @branch` | `.behavior/v2026_03_05.fix-auth-bug/` |
| start behavior on `main` | `rhx init.behavior --name @branch` | **error**: cannot init behavior on main |

### contract

**input:**
- `--name @branch` тАФ magic token that expands at runtime

**expansion:**
```
git branch --show-current
  тЖТ casey/add-user-auth
  тЖТ strip author prefix
  тЖТ add-user-auth
```

**output:**
- behavior directory created with expanded name
- or error if on protected branch

### timeline

```
t0: developer creates feature branch (casey/cool-feature)
t1: developer runs: rhx init.behavior --name @branch
t2: system expands @branch тЖТ cool-feature
t3: system creates .behavior/v{date}.cool-feature/
t4: developer proceeds with behavior-driven development
```

## mental model

### how users describe it

> "it names the behavior after my branch"
> "@ means 'current', like @latest or @head"
> "@branch is a shortcut so i don't have to type the name twice"

### analogies

- `@branch` is like `HEAD` in git тАФ it means "where i am now"
- `@branch` is like `$PWD` in shell тАФ context-aware expansion
- `@branch` is like `@latest` in npm тАФ expands to current state

### user terms vs our terms

| user says | we mean |
|-----------|---------|
| "the @ thing" | magic token |
| "branch shortcut" | @branch token |
| "auto-name" | token expansion |

## evaluation

### how well does it solve the goal?

**goal: reduce friction when starting behaviors**
- eliminates manual name typing тЬУ
- removes chance of typo тЬУ
- enforces branch/behavior name alignment тЬУ

### pros

- zero cognitive load тАФ don't think, just type `@branch`
- consistency тАФ behavior name always matches branch
- speed тАФ one token vs many characters
- safety тАФ rejects protected branches automatically

### cons

- new syntax to learn (`@branch`)
- doesn't help if branch name is bad (garbage in, garbage out)

### edgecases & pit of success

| edgecase | handling |
|----------|----------|
| on `main` or `master` | **error**: "cannot init behavior on protected branch" |
| branch has no prefix (e.g., `fix-bug`) | use full name: `fix-bug` |
| branch has multiple slashes (`org/team/feature`) | strip all but last: `feature` |
| detached HEAD | **error**: "cannot expand @branch in detached HEAD state" |
| `--name @branch` with explicit name after | explicit name wins (or error?) |

## open questions & assumptions

### assumptions

1. **strip all but last**: for any branch with `/`, use only the last segment
2. **@branch is unambiguous**: assumes no other `@` tokens in the wild

### self-answered questions

1. **no-prefix branches**: if branch is just `feature`, use as-is
   - rationale: no `/` means no prefix to strip тАФ use whole name

2. **case sensitivity**: preserve case (`Casey/Feature` тЖТ `Feature`)
   - rationale: don't mangle user's input тАФ preserve what they typed

3. **conflict with explicit name**: non-issue
   - `--name` takes one argument: either `@branch` or an explicit name, not both

4. **multi-segment prefixes**: strip all but last (`casey/team/feature` тЖТ `feature`)
   - rationale: wisher confirmed тАФ always use last segment

## what is awkward?

### potential friction points

1. **learning curve**: users need to know `@branch` exists
   - mitigation: error message suggests it when on feature branch

2. **prefix stripping heuristic**: might strip wrong thing
   - mitigation: preview what name will be used before creating

3. **branch name quality**: if branch is `wip` or `test`, behavior name is bad
   - mitigation: that's a user problem, not ours

### design tensions

- **magic vs explicit**: `@branch` is magic, which some devs distrust
  - but the alternative (manual entry) is worse
